## 单例基类

### 什么是单例模式

单例模式是设计模式中最常见的一种：**保证一个类在整个程序生命周期中只有一个实例，并提供全局访问点**。

在游戏开发中，很多管理器天然就应该是"唯一的"——你不可能有两个音效管理器同时播放背景音乐，也不可能有两套事件中心各自分发消息。如果没有单例，每个需要用管理器的脚本都要自己 `new` 一个，结果就是各管各的，数据不同步，逻辑全乱套。

单例模式的好处：

- **全局唯一**：整个程序只有一个实例，避免了数据不同步的问题
- **访问方便**：任何地方通过 `XXX.GetInstance()` 就能拿到实例，不需要拖拽赋值
- **统一初始化**：实例的创建时机和方式由类自己控制，外部不用关心

### 泛型基础

框架中有多个管理器（PoolMgr、EventCenter、MusicMgr……），如果每个都写一遍单例的 `instance` + `GetInstance()`，代码就重复了。泛型可以解决这个问题——**把"单例的逻辑"抽成一个基类，具体的类型作为泛型参数传进去**。

```csharp
// 没有泛型时，每个管理器都要写一遍：
public class PoolMgr
{
    private static PoolMgr instance;
    public static PoolMgr GetInstance() { ... }
}
public class EventCenter
{
    private static EventCenter instance;
    public static EventCenter GetInstance() { ... }
}
// 每个类都在重复相同的模式

// 用泛型后，抽成一个基类：
public class BaseManager<T> { ... }
// PoolMgr 直接继承：BaseManager<PoolMgr>
// EventCenter 直接继承：BaseManager<EventCenter>
```

泛型最核心的思想：**用类型参数 `T` 来代表"还不知道是什么类型"的占位符**，等到实际使用时才指定具体类型。

### 泛型约束 — 为什么要 `where T : new()`

泛型参数 `T` 默认可以是任何类型，但如果你要在代码里 `new T()`，就必须告诉编译器"T 一定有无参构造函数"。这就是 `where T : new()` 的作用——**约束 T 必须有无参构造函数**。

```csharp
// 没有 new() 约束 → 编译报错
public class BaseManager<T>
{
    public static T GetInstance()
    {
        instance = new T();  // 编译器不知道 T 能不能 new，直接报错
    }
}

// 加上 new() 约束 → 编译通过
public class BaseManager<T> where T : new()
{
    public static T GetInstance()
    {
        instance = new T();  // 编译器知道 T 一定有无参构造函数，放心通过
    }
}
```

常见的泛型约束：

| 约束                        | 含义                            |
| --------------------------- | ------------------------------- |
| `where T : new()`         | T 必须有无参构造函数            |
| `where T : class`         | T 必须是引用类型                |
| `where T : struct`        | T 必须是值类型                  |
| `where T : MonoBehaviour` | T 必须是 MonoBehaviour 或其子类 |
| `where T : 某基类`        | T 必须继承自指定基类            |

### BaseManager — 最基础的单例基类

`BaseManager<T>` 是整个框架的基石，所有非 MonoBehaviour 的管理器都继承它。它适用于**纯 C# 类**，不依赖 Unity 的生命周期。

```csharp
public class BaseManager<T> where T : new()
{
    private static T instance;

    public static T GetInstance()
    {
        if (instance == null)
            instance = new T();
        return instance;
    }
}
```

关键点：

- `static T instance` — **静态字段属于类本身**，不属于任何一个实例，所以整个程序只有一份。不管你在多少地方调用 `GetInstance()`，拿到的都是同一个 `instance`。
- `where T : new()` — 因为需要 `new T()` 来创建实例，所以必须约束 T 有无参构造函数。
- **懒加载（Lazy Initialization）**：第一次调用 `GetInstance()` 时才创建实例，而不是程序一启动就创建，避免浪费资源。

使用示例：

```csharp
// 定义自己的管理器，继承 BaseManager 即可
public class MyManager : BaseManager<MyManager>
{
    public void DoSomething() { }
}

// 任何地方使用
MyManager.GetInstance().DoSomething();
```

### SingletonMono — 需要手动挂载的 MonoBehaviour 单例

有些管理器必须依附到 GameObject 上才能工作（比如需要 Update、协程），那就得继承 MonoBehaviour。但 **MonoBehaviour 不能用 `new` 创建**——Unity 中继承 MonoBehaviour 的脚本只能通过挂载到对象上或 `AddComponent` 来实例化。

所以 `SingletonMono<T>` 的约束变成了 `where T : MonoBehaviour`，并且**不能再写 `new T()`**，需要在 Awake 中把自身赋值给 `instance`。

```csharp
using UnityEngine;

//继承了 MonoBehaviour 的 单例模式对象 需要我们自己保证它的位移性
public class SingletonMono<T> : MonoBehaviour where T : MonoBehaviour
{
    private static T instance;

    public static T GetInstance()
    {
        //继承了Mono的脚本 不能够直接new
        //只能通过拖动到对象上 或者 通过 加脚本的api AddComponent去加脚本
        //U3D内部帮助我们实例化它
        return instance;
    }

    protected virtual void Awake()
    {
        instance = this as T;
    }
}
```

关键点：

- **不能 `new T()`**：MonoBehaviour 的实例化由 Unity 引擎控制，所以 `GetInstance()` 只负责返回，不负责创建——创建需要你手动把脚本挂到场景中的 GameObject 上。
- **Awake 中赋值**：当 Unity 创建这个脚本实例时，Awake 自动被调用，在这里把 `this` 赋给 `instance`，完成单例的注册。
- **需要自己保证唯一性**：如果你在多个 GameObject 上挂了同一个脚本，后面的 Awake 会覆盖前面的 `instance`，这一点需要手动控制。

### SingletonAutoMono — 自动创建的 MonoBehaviour 单例

`SingletonMono` 的缺点是每次都要手动拖脚本到对象上，很麻烦。`SingletonAutoMono` 进一步优化：**第一次调用 `GetInstance()` 时自动创建 GameObject 并挂载脚本**，不需要手动操作。

```csharp
using UnityEngine;

//C#中 泛型知识点
//设计模式 单例模式的知识点
//继承这种自动创建的 单例模式基类 不需要我们手动去拖 或者 api去加了
//想用他 直接 GetInstance就行了
public class SingletonAutoMono<T> : MonoBehaviour where T : MonoBehaviour
{
    private static T instance;

    public static T GetInstance()
    {
        if (instance == null)
        {
            GameObject obj = new GameObject();
            //设置对象的名字为脚本名
            obj.name = typeof(T).ToString();
            //让这个单例模式对象 过场景 不移除
            //因为 单例模式对象 往往 是存在整个程序生命周期中的
            DontDestroyOnLoad(obj);
            instance = obj.AddComponent<T>();
        }
        return instance;
    }
}
```

关键点：

- **自动创建**：首次调用时 `new GameObject()` + `AddComponent<T>()`，用户什么都不用管。
- **DontDestroyOnLoad**：管理器通常伴随整个游戏生命周期，所以切换场景时不应被销毁。
- **`typeof(T).ToString()`**：用类名来命名 GameObject，在 Hierarchy 窗口中方便查找调试。

### 三种单例的对比

|            | BaseManager                        | SingletonMono               | SingletonAutoMono           |
| ---------- | ---------------------------------- | --------------------------- | --------------------------- |
| 基类       | 纯 C# 类                           | MonoBehaviour               | MonoBehaviour               |
| 创建方式   | `new T()`                        | 手动挂载/AddComponent       | 首次调用自动创建            |
| 约束       | `where T : new()`                | `where T : MonoBehaviour` | `where T : MonoBehaviour` |
| 适用场景   | 纯逻辑管理器（事件中心、场景管理） | 需要手动控制挂载时机的组件  | 需要 Update/协程的管理器    |
| 过场景保留 | 静态字段天然保留                   | 需手动 DontDestroyOnLoad    | 自动 DontDestroyOnLoad      |

在整个框架中，大部分管理器（PoolMgr、EventCenter、MonoMgr、ScenesMgr、ResMgr、InputMgr、MusicMgr、UIManager、AStarMgr）都继承自 `BaseManager<自身类名>`。它们不需要继承 MonoBehaviour，所以统一走 `BaseManager`。而当某个模块需要 Update 帧循环时，则通过 **公共Mono** 模块（下一章）来间接获取帧更新能力，而不是自己变成 MonoBehaviour。

## 公共Mono

### 解决什么问题

上一章我们学了 `BaseManager`——所有纯 C# 的管理器都可以通过继承它来获得单例能力。但这里有一个问题：**纯 C# 类没有 Unity 的生命周期函数**。

`Update()` 每帧调用？纯 C# 类做不到。
`StartCoroutine()` 启动协程？纯 C# 类也做不到。

这些能力只有继承自 `MonoBehaviour` 的脚本才有。那像 `InputMgr`（需要每帧检测按键）、`MusicMgr`（需要每帧检查音效是否播完）这样的管理器怎么办？

思路很简单：**找一个 MonoBehaviour 替我们跑 Update，然后让其他管理器来"订阅"这个 Update**。公共 Mono 模块就是做这件事的。

### 设计思路：两层结构

公共 Mono 模块由两个类组成：

```
外部管理器（纯C#）                    公共Mono模块
┌──────────────┐          ┌─────────────────────────┐
│  InputMgr    │──订阅──→ │  MonoMgr (单例管理器)    │
│  MusicMgr    │          │    ↓ 持有               │
│  其他任何类  │          │  MonoController (Mono)  │
└──────────────┘          │    ├─ Update() 帧循环    │
                          │    ├─ StartCoroutine()   │
                          │    └─ DontDestroyOnLoad  │
                          └─────────────────────────┘
```

- **MonoController**：挂载在隐藏 GameObject 上的 MonoBehaviour，真正拥有 Update 和协程能力。它通过 C# 的 `event` 关键字暴露一个 `updateEvent`，在 Update 中触发，外部方法订阅后就能"蹭"到每帧调用。
- **MonoMgr**：继承 `BaseManager<MonoMgr>` 的单例管理器，在构造函数中创建 MonoController 对象，对外提供 `AddUpdateListener`、`RemoveUpdateListener`、`StartCoroutine` 三个接口。外部只需要跟 MonoMgr 打交道，不需要知道 MonoController 的存在。

这种分层的好处是**职责分离**：MonoController 只管 Unity 生命周期，MonoMgr 管访问控制和对外接口。

### C# event 关键字

MonoController 中用了 `private event UnityAction updateEvent`，这里补充一下 `event` 关键字的作用。

`UnityAction` 本身是一个委托类型，外部可以直接 `+=` 和 `-=`。但如果把委托声明为 `public`，外部就可以直接调用 `updateEvent()` 甚至 `updateEvent = null` 清空所有订阅——这很危险。

加上 `event` 关键字后：**外部只能 `+=`（订阅）和 `-=`（取消订阅），不能直接调用，也不能清空**。调用权留在 MonoController 自己的 `Update()` 里。这是一种很好的封装手段。

### MonoController — 帧循环的提供者

```csharp
using UnityEngine;
using UnityEngine.Events;

public class MonoController : MonoBehaviour
{

    private event UnityAction updateEvent;

    // Use this for initialization
    void Start()
    {
        DontDestroyOnLoad(this.gameObject);
    }

    // Update is called once per frame
    void Update()
    {
        if (updateEvent != null)
            updateEvent();
    }

    /// <summary>
    /// 给外部提供的 添加帧更新事件的函数
    /// </summary>
    /// <param name="fun"></param>
    public void AddUpdateListener(UnityAction fun)
    {
        updateEvent += fun;
    }

    /// <summary>
    /// 提供给外部 用于移除帧更新事件函数
    /// </summary>
    /// <param name="fun"></param>
    public void RemoveUpdateListener(UnityAction fun)
    {
        updateEvent -= fun;
    }
}
```

关键点：

- **`DontDestroyOnLoad`**：MonoController 依附的 GameObject 在切换场景时不能被销毁，否则所有依赖它的管理器都会失去 Update 能力。
- **Update 中的空检查**：`if (updateEvent != null)` 是必须的——如果没有任何订阅者就去 Invoke，会抛 NullReferenceException。
- **协程天然可用**：因为 MonoController 本身就是 MonoBehaviour，它的 `StartCoroutine` 可以直接用，不需要额外处理。

### MonoMgr — 对外的统一接口

```csharp
using System.Collections;
using System.ComponentModel;
using UnityEngine;
using UnityEngine.Events;

/// <summary>
/// 1.可以提供给外部添加帧更新事件的方法
/// 2.可以提供给外部添加 协程的方法
/// </summary>
public class MonoMgr : BaseManager<MonoMgr>
{
    private MonoController controller;

    public MonoMgr()
    {
        //保证了MonoController对象的唯一性
        GameObject obj = new GameObject("MonoController");
        controller = obj.AddComponent<MonoController>();
    }

    /// <summary>
    /// 给外部提供的 添加帧更新事件的函数
    /// </summary>
    /// <param name="fun"></param>
    public void AddUpdateListener(UnityAction fun)
    {
        controller.AddUpdateListener(fun);
    }

    /// <summary>
    /// 提供给外部 用于移除帧更新事件函数
    /// </summary>
    /// <param name="fun"></param>
    public void RemoveUpdateListener(UnityAction fun)
    {
        controller.RemoveUpdateListener(fun);
    }

    public Coroutine StartCoroutine(IEnumerator routine)
    {
        return controller.StartCoroutine(routine);
    }

    public Coroutine StartCoroutine(string methodName, [DefaultValue("null")] object value)
    {
        return controller.StartCoroutine(methodName, value);
    }

    public Coroutine StartCoroutine(string methodName)
    {
        return controller.StartCoroutine(methodName);
    }
}
```

关键点：

- **构造函数中创建 MonoController**：`BaseManager<T>` 的 `GetInstance()` 首次调用时触发 `new T()`，即触发此构造函数。GameObject 和 MonoController 随着首次访问自动创建，不需要手动初始化。
- **三个 StartCoroutine 重载**：分别对应 `StartCoroutine(IEnumerator)`、`StartCoroutine(string, object)`、`StartCoroutine(string)`，覆盖了 MonoBehaviour 的协程启动方式。
- **所有方法都是转发（委托）给 controller**：MonoMgr 本身不做逻辑，只是把调用传递给 MonoController。这种模式叫做 **Facade（外观模式）**——对外提供一个简洁的接口，内部隐藏复杂性。

### 使用示例

```csharp
// 某个纯 C# 的管理器需要每帧执行逻辑
public class MyManager : BaseManager<MyManager>
{
    public MyManager()
    {
        // 订阅帧更新
        MonoMgr.GetInstance().AddUpdateListener(OnUpdate);
    }

    private void OnUpdate()
    {
        // 这里的代码每帧都会执行
    }

    public void DoAsyncWork()
    {
        // 启动协程
        MonoMgr.GetInstance().StartCoroutine(AsyncTask());
    }

    private IEnumerator AsyncTask()
    {
        yield return new WaitForSeconds(2f);
        Debug.Log("两秒后执行");
    }

    // 注意：不再需要时应该取消订阅，避免内存泄漏和空调用
    public void Dispose()
    {
        MonoMgr.GetInstance().RemoveUpdateListener(OnUpdate);
    }
}
```

### 补充：为什么不在 BaseManager 里直接加 Update

你可能会想：既然这么多管理器需要 Update，为什么不在 `BaseManager` 里直接搞一个 Update？

因为 `BaseManager` 是纯 C# 类，它不继承 MonoBehaviour，就没有 Update。如果让 `BaseManager` 继承 MonoBehaviour，那所有子类都变成了 MonoBehaviour，就不能 `new T()` 了——这就回到了上一章讨论的**泛型约束的二选一**：要 `new()` 就不能是 MonoBehaviour，要 MonoBehaviour 就不能 `new()`。

唐老师这套框架的选择是：**管理器保持纯 C#（走 BaseManager），需要 Unity 生命周期时统一通过公共 Mono 模块来获取**。职责清晰，各不越界。

## 资源加载

### 解决什么问题

Unity 中加载资源最基本的方式是 `Resources.Load`，但如果每个需要资源的脚本都自己去写 `Resources.Load`，会有几个问题：

- 路径散落在各处，如果资源目录调整了，要改 N 个文件
- 同步加载大资源会卡住主线程，必须要有异步加载的能力
- GameObject 类型的资源加载后只是 Prefab 引用，还需要 `Instantiate` 实例化才能用——每个调用方都自己写一遍很啰嗦

所以需要一个统一的资源加载模块，把同步/异步、GameObject 实例化这些细节都封装好，外部只需要告诉它"我要什么资源 + 加载完了你要干嘛"。

### 泛型方法 + `where T : Object`

前面的 `BaseManager<T>` 约束了 `new()`，`SingletonMono<T>` 约束了 `MonoBehaviour`，这次 `ResMgr` 的 `T` 约束的是 `UnityEngine.Object`——这涵盖了所有 Unity 资源类型：`GameObject`、`AudioClip`、`TextAsset`、`Sprite`、`Material` 等等。

```csharp
// T extends UnityEngine.Object
// 这意味着 T 可以是 GameObject、AudioClip、Sprite 等 Unity 资源类型
public T Load<T>(string name) where T : Object
{
    T res = Resources.Load<T>(name);
    ...
}
```

约束 `Object` 而不是更宽泛的类型，是因为 `Resources.Load<T>` 本身就要求 T 是 `Object` 的子类，编译器需要这个约束来通过检查。

### 为什么 GameObject 要特殊处理

`Resources.Load` 加载后返回的是一个**资源引用**（Prefab 的蓝图），不是场景中可以用的实体。对于 `GameObject`，必须再调用 `Instantiate` 才能得到一个真正的、可以在场景中显示和移动的对象。而对于 `AudioClip`、`TextAsset` 这类非可视化资源，直接返回引用就够了。

```csharp
// 同步版中的判断：
if (res is GameObject)
    return GameObject.Instantiate(res);  // 实例化后返回克隆体
else
    return res;  // AudioClip/TextAsset 直接返回
```

这里 `Instantiate` 产生的是**深拷贝**：克隆体拥有原 Prefab 的所有组件、子对象、材质等，但它是独立的实体，修改克隆体不会影响原 Prefab。这也是为什么缓存池那一章需要配合 ResMgr——池子里的对象都是用 ResMgr 加载并实例化出来的。

### 同步加载 — `Load<T>`

同步加载适合**小资源**或**确定资源一定很小**的场景。比如加载一个配置文件 TextAsset，或者一个 UI 面板的 Prefab（在异步加载机制准备好之前）。

```csharp
//同步加载资源
public T Load<T>(string name) where T : Object
{
    T res = Resources.Load<T>(name);
    //如果对象是一个GameObject类型的 我把他实例化后 再返回出去 外部 直接使用即可
    if (res is GameObject)
        return GameObject.Instantiate(res);
    else//TextAsset AudioClip
        return res;
}
```

流程很简单：加载 → 判断类型 → 如果 GameObject 就实例化 → 返回。整个过程是阻塞的，所以**不要用同步方法加载大图集或大场景**，否则游戏会卡一下。

### 异步加载 — `LoadAsync<T>`

异步加载是实际项目中的主力。核心思路是：**利用协程的 `yield return` 把加载操作分散到多帧执行**，不阻塞主线程。

```csharp
//异步加载资源
public void LoadAsync<T>(string name, UnityAction<T> callback) where T : Object
{
    //开启异步加载的协程
    MonoMgr.GetInstance().StartCoroutine(ReallyLoadAsync(name, callback));
}
```

`LoadAsync` 本身只负责启动协程，真正的加载逻辑在 `ReallyLoadAsync` 中。这里用到了**上一章公共 Mono 模块提供的协程能力**——`MonoMgr.GetInstance().StartCoroutine(...)`。

注意 `LoadAsync` 没有返回值，而是接收一个 `UnityAction<T> callback`：**加载完成后通过回调把资源传出去**。这是异步编程最常见的模式——调用方不阻塞等待，而是"加载好了你通知我"。

### 协程内部的真正加载逻辑

```csharp
//真正的协同程序函数  用于 开启异步加载对应的资源
private IEnumerator ReallyLoadAsync<T>(string name, UnityAction<T> callback) where T : Object
{
    ResourceRequest r = Resources.LoadAsync<T>(name);
    yield return r;

    if (r.asset is GameObject)
        callback(GameObject.Instantiate(r.asset) as T);
    else
        callback(r.asset as T);
}
```

逐行拆解：

1. **`Resources.LoadAsync<T>(name)`**：Unity 在后台线程加载资源，返回一个 `ResourceRequest` 对象。这个对象包含 `progress`（加载进度）和 `asset`（加载完成的资源）。
2. **`yield return r`**：协程在这里暂停，**每一帧 Unity 检查资源是否加载完毕**。加载完毕了就继续往下执行，没加载完就让出这一帧继续等待。这就是异步不卡顿的关键。
3. **`r.asset`**：加载完成的资源引用。同样做 GameObject 判断：是 GameObject 就 Instantiate，不是就直接回调传出。
4. **`as T` 转换**：因为 `r.asset` 返回的是 `Object` 类型，需要 `as T` 转成泛型类型再传给回调。

### lambda 表达式在回调中的角色

外部调用 `LoadAsync` 时，回调通常用 lambda 写：

```csharp
ResMgr.GetInstance().LoadAsync<GameObject>("UI/LoginPanel", (obj) =>
{
    // obj 就是加载完成并 Instantiate 好的 GameObject
    obj.transform.SetParent(canvas);
    Debug.Log("面板加载完成");
});
```

`(obj) => { ... }` 就是一个 lambda 表达式——等价于写一个匿名函数作为 `UnityAction<GameObject>` 类型的参数。这种写法比单独定义一个方法更紧凑，而且能捕获外部变量（闭包），比如上面代码中的 `canvas`。

### 与公共 Mono 的关系

`ResMgr` 本身继承 `BaseManager<ResMgr>`，是纯 C# 类，没有协程能力。它的异步加载依赖于 **上一章的公共 Mono 模块**：`MonoMgr.GetInstance().StartCoroutine(...)`。

```
ResMgr.LoadAsync() 
    → MonoMgr.GetInstance().StartCoroutine(ReallyLoadAsync(...))
        → MonoController.StartCoroutine(...) 
            → Unity 引擎执行协程
```

这就是框架模块之间开始产生依赖关系的体现——前面打好的基础（单例 → 公共 Mono → 资源加载）现在开始发挥作用了。

### 使用示例

**同步加载配置文件：**

```csharp
TextAsset config = ResMgr.GetInstance().Load<TextAsset>("Config/GameSettings");
string json = config.text;
```

**异步加载敌人预制体：**

```csharp
ResMgr.GetInstance().LoadAsync<GameObject>("Enemy/Orc", (enemyPrefab) =>
{
    // 加载完成后实例化到场景中
    enemyPrefab.transform.position = spawnPoint;
    Debug.Log("敌人预制体加载并实例化完成");
});
```

**异步加载音效资源：**

```csharp
ResMgr.GetInstance().LoadAsync<AudioClip>("Music/Sound/Explosion", (clip) =>
{
    // AudioClip 不需要 Instantiate，直接使用
    audioSource.PlayOneShot(clip);
});
```

### 注意事项

- **Resources 目录限制**：`Resources.Load` 只能加载 `Resources` 文件夹下的资源。所有需要用 ResMgr 加载的资源必须放在 Resources 目录或其子目录中，Unity 打包时会把 Resources 下所有资源都打进包，所以不要在这里放暂时不用的东西。
- **异步回调中访问 GameObject**：加载是异步的，回调执行时场景状态可能已经变了（比如加载了一个面板但玩家已经退出到主菜单）。在实际项目中，回调里通常需要加一个"是否仍然需要这个资源"的判断。
- **路径区分大小写吗**：Unity 的资源路径在大多数平台上**不区分大小写**，但为了跨平台兼容，建议统一用你项目中的实际目录结构拼路径，比如 `"Music/BK/"` 这种约定的前缀。

## 缓存池

### 解决什么问题

游戏中有些对象会被反复创建和销毁——子弹发射出去飞一会就没了，怪物死了又刷一批新的，特效播完就消失。如果每次都 `Instantiate` 创建、`Destroy` 销毁，会带来两个问题：

- **性能开销**：`Instantiate` 和 `Destroy` 都很重，频繁调用会导致帧率波动
- **GC 压力**：`Destroy` 后 GameObject 变成垃圾等待 GC 回收，GC 触发时会造成明显卡顿

缓存池（Object Pool）的思路很简单：**用完了不销毁，失活存起来，下次需要时直接激活复用**。就像图书馆的书——还回来放回书架，下一个人继续借，不会看一本印一本。

### 三层比喻：衣柜 → 抽屉 → 物品

这个模块用了一个很形象的比喻来组织数据结构：

```
PoolMgr (衣柜)
  └─ poolDic: Dictionary<string, PoolData>
       ├─ "Bullet"  → PoolData (抽屉)
       │    └─ poolList: List<GameObject>
       │         ├─ Bullet (失活)   ← 用完了塞回来
       │         ├─ Bullet (失活)
       │         └─ Bullet (失活)
       ├─ "Explosion" → PoolData (抽屉)
       │    └─ poolList: List<GameObject>
       │         └─ Explosion (失活)
       └─ ...
```

- **衣柜 (PoolMgr)**：总管所有对象池，通过 `Dictionary<string, PoolData>` 按名字找抽屉。名字就是资源的 key，比如 `"Bullet"`、`"Explosion"`。
- **抽屉 (PoolData)**：管理同一类对象。内部有一个 `List<GameObject>` 存所有"暂时不用"的对象，还有一个 `fatherObj` 作为这些对象的父节点（在 Hierarchy 中把它们收在一团，看着清爽）。
- **物品 (GameObject)**：存在抽屉里时 `SetActive(false)`，取出来时 `SetActive(true)`。进出之间不需要 `Instantiate` 也不需要 `Destroy`。

### PoolData — 抽屉的入和出

```csharp
/// <summary>
/// 抽屉数据  池子中的一列容器
/// </summary>
public class PoolData
{
    //抽屉中 对象挂载的父节点
    public GameObject fatherObj;
    //对象的容器
    public List<GameObject> poolList;

    public PoolData(GameObject obj, GameObject poolObj)
    {
        //给我们的抽屉 创建一个父对象 并且把他作为我们pool(衣柜)对象的子物体
        fatherObj = new GameObject(obj.name);
        fatherObj.transform.parent = poolObj.transform;
        poolList = new List<GameObject>() { };
        PushObj(obj);
    }

    /// <summary>
    /// 往抽屉里面 压东西
    /// </summary>
    /// <param name="obj"></param>
    public void PushObj(GameObject obj)
    {
        //失活 让其隐藏
        obj.SetActive(false);
        //存起来
        poolList.Add(obj);
        //设置父对象
        obj.transform.parent = fatherObj.transform;
    }

    /// <summary>
    /// 从抽屉里面 取东西
    /// </summary>
    /// <returns></returns>
    public GameObject GetObj()
    {
        GameObject obj = null;
        //取出第一个
        obj = poolList[0];
        poolList.RemoveAt(0);
        //激活 让其显示
        obj.SetActive(true);
        //断开了父子关系
        obj.transform.parent = null;

        return obj;
    }
}
```

关键点：

- **构造函数**：创建一个新的空抽屉时，同时创建一个 `fatherObj` 作为这个抽屉里所有对象的父节点，挂到 `poolObj`（即衣柜的根节点）下。这样在 Hierarchy 中层级结构清晰：`Pool → Bullet → (三个失活的子弹)`。
- **PushObj（压入）**：三步走——`SetActive(false)` 藏起来 → 加入 `poolList` → 把 GameObject 挂到 `fatherObj` 下当子物体。**不销毁对象**，只是让它不可见且归档。
- **GetObj（取出）**：三步走——从 `poolList[0]` 取出第一个 → `SetActive(true)` 唤醒 → `transform.parent = null` **断开父子关系**，让对象独立出去。取的是第一个（FIFO 先入先出），简单高效。
- **为什么取出来要断开父对象？** 取出来的子弹要在场景中飞行，爆炸特效要出现在碰撞点——它们需要有独立的世界坐标。如果还挂在 fatherObj 下，移动时会受到父节点位置影响。

### PoolMgr — 衣柜的调度

```csharp
public class PoolMgr : BaseManager<PoolMgr>
{
    //缓存池容器 （衣柜）
    public Dictionary<string, PoolData> poolDic = new Dictionary<string, PoolData>();

    private GameObject poolObj;

    /// <summary>
    /// 往外拿东西
    /// </summary>
    /// <param name="name"></param>
    /// <returns></returns>
    public void GetObj(string name, UnityAction<GameObject> callBack)
    {
        //有抽屉 并且抽屉里有东西
        if (poolDic.ContainsKey(name) && poolDic[name].poolList.Count > 0)
        {
            callBack(poolDic[name].GetObj());
        }
        else
        {
            //通过异步加载资源 创建对象给外部用
            ResMgr.GetInstance().LoadAsync<GameObject>(name, (o) =>
            {
                o.name = name;
                callBack(o);
            });
        }
    }

    /// <summary>
    /// 换暂时不用的东西给我
    /// </summary>
    public void PushObj(string name, GameObject obj)
    {
        if (poolObj == null)
            poolObj = new GameObject("Pool");

        //里面有抽屉
        if (poolDic.ContainsKey(name))
        {
            poolDic[name].PushObj(obj);
        }
        //里面没有抽屉
        else
        {
            poolDic.Add(name, new PoolData(obj, poolObj));
        }
    }

    /// <summary>
    /// 清空缓存池的方法 
    /// 主要用在 场景切换时
    /// </summary>
    public void Clear()
    {
        poolDic.Clear();
        poolObj = null;
    }
}
```

逐段拆解：

**GetObj — 拿东西**：

1. 先看字典里有没有这个名字对应的抽屉，并且抽屉里还有没有库存（`poolList.Count > 0`）
2. **有存货** → 直接从抽屉 `GetObj()` 取出，通过回调返回
3. **没存货** → 调用 `ResMgr.GetInstance().LoadAsync<GameObject>(name, ...)` 异步加载资源。**这里依赖了上一章的 ResMgr**——缓存池自己不处理资源加载，而是委托给 ResMgr。加载完成后把对象名字设为资源名（`o.name = name`），方便后续 Push 回来时能找到对应的抽屉

注意 `GetObj` 没有返回值，用的是回调模式。因为没存货时要走异步加载，不能同步 return，所以统一用 `UnityAction<GameObject>` 回调把对象传出去。

**PushObj — 还东西**：

1. 如果衣柜的根节点 `poolObj` 还不存在，先 `new GameObject("Pool")` 创建。这个空对象就是 Hierarchy 中所有抽屉父节点的根。
2. 如果已有这个名字的抽屉 → 直接塞进去
3. 如果还没有 → 先 `new PoolData(obj, poolObj)` 创建一个新抽屉，再加到字典里

**第一个对象是怎么进入池子的？** 第一个对象并不是通过 PushObj 进来的——它是通过 GetObj 时发现没存货，走 ResMgr 异步加载创建出来的，然后直接通过回调给外部用。**用完之后**才 PushObj 回来，从此以后它就在池子里循环了。所以池子是"先用后存，越用越富"。

**Clear — 场景切换清空**：
场景切换时，旧场景的对象全部销毁了（由 Unity 自动处理），但 `poolDic` 里的引用还指向那些已经销毁的 GameObject。所以必须 `Clear()`，把字典清空、根节点引用置 null，下一个场景重新来过。

### 依赖关系链条

这一章可以看到框架模块之间的依赖关系开始清晰起来：

```
PoolMgr.GetObj("Bullet") 
    ├─ 有存货 → PoolData.GetObj() → 直接返回复用对象
    └─ 没存货 → ResMgr.GetInstance().LoadAsync<GameObject>(...)
                   └─ MonoMgr.GetInstance().StartCoroutine(...)
                          └─ Resources.LoadAsync + Instantiate
```

缓存池依赖资源加载，资源加载依赖公共 Mono。三层协作，每一层只做自己份内的事。

### 使用示例

**发射子弹（从池子取）：**

```csharp
// 发射一颗子弹
PoolMgr.GetInstance().GetObj("Bullet", (bullet) =>
{
    bullet.transform.position = firePoint.position;
    bullet.transform.rotation = firePoint.rotation;
    // 给子弹设置速度等初始化逻辑...
});
```

**子弹命中（归还池子）：**

```csharp
// 子弹碰撞时调用——不 Destroy，而是 Push 回池子
void OnBulletHit(GameObject bullet)
{
    PoolMgr.GetInstance().PushObj("Bullet", bullet);
}
```

**场景切换时清空：**

```csharp
// 切换场景前
PoolMgr.GetInstance().Clear();
```

### 注意事项

- **取出来的对象一定要 Push 回来**：如果取出来用了就忘，池子里的存货会越来越少，最终等同于没池子。子弹出屏、特效播完、怪物死亡——这些时机都要记得归还。
- **SetActive 的隐藏规则**：对象失活期间，它上面的所有脚本（Update、碰撞检测等）都不会执行。所以 Push 回池子后不需要担心它还在运行逻辑。
- **归还后的状态重置**：PushObj 只做了失活 + 改父节点。如果对象上有自定义数据（比如子弹的速度残留、特效的播放进度），需要在 Push 之前手动重置。这个框架没有自动重置机制，调用方自己负责。
- **Clear 时机**：`SceneManager.LoadScene` 会销毁旧场景的所有 GameObject，池子里的引用会全部变成 `null`。必须在场景切换前或切换后立刻 `Clear()`，否则字典里残留大量失效引用，下次 GetObj 时会取出一个已销毁的对象。
- **异步加载的时间差**：第一次 `GetObj("Bullet")` 时如果池子是空的，会走异步加载。这意味着**获取到的对象不是立即可用的**——它通过回调返回。调用方必须把使用逻辑写在回调里，不能同步地 `obj = GetObj(...)` 然后直接用。

## 事件中心

### 解决什么问题

假设你在做这样一个游戏：

- 怪物死亡时 → 成就系统要计数 + UI 要刷新分数 + 音效要播放惨叫 + 任务系统要检查是否完成
- 如果让"怪物"去一一通知这些系统，怪物脚本里就要持有四个系统的引用：`成就系统.DoCount()`、`UIManager.UpdateScore()`、`MusicMgr.PlaySound()`……

这还只是一个事件。随着游戏变大，模块之间互相引用的关系会变成一张蜘蛛网——改一个模块可能牵连十个模块，完全没法维护。

事件中心用 **发布-订阅模式（观察者模式）** 解决这个问题：

```
触发者（怪物）                      监听者们
┌──────────┐                    ┌──────────┐
│ 怪物死亡  │──触发"怪物死亡"──→  │ 成就系统  │
│  (发布者) │      事件中心       │ UI系统   │ ← 各自独立，互不知道
└──────────┘     (中介)         │ 音效系统  │
                              │ 任务系统  │
                              └──────────┘
```

怪物只做一件事：`EventCenter.GetInstance().EventTrigger("怪物死亡", data)`。谁在听？它完全不知道，也不需要知道。成就系统、UI、音效各自订阅 `"怪物死亡"` 事件，事件中心负责把消息分发给所有订阅者。

**发布者和订阅者完全解耦**——新增一个"击杀特效系统"只需订阅事件，不影响任何已有代码。

### 从"直接调用"到"事件驱动"

先看没有事件中心时怎么写：

```csharp
// 怪物死亡脚本 —— 强耦合写法
void OnMonsterDie()
{
    AchievementSystem.GetInstance().AddKillCount(1);  // 成就
    UIManager.GetInstance().UpdateScore(100);          // UI
    MusicMgr.GetInstance().PlaySound("OrcDie");        // 音效
    QuestSystem.GetInstance().CheckQuest("杀兽人");    // 任务
    // 每加一个新系统，这里就要多一行……
}
```

有了事件中心后：

```csharp
// 怪物死亡脚本 —— 解耦写法
void OnMonsterDie()
{
    EventCenter.GetInstance().EventTrigger("怪物死亡", this);
    // 就这一行，至于谁在听，怪物完全不关心
}

// 各系统各自订阅（通常在 Awake 或 Start 中）
EventCenter.GetInstance().AddEventListener<Monster>("怪物死亡", OnMonsterDieCallback);
```

同样的功能，耦合度天壤之别。

### IEventInfo — 用一个空接口统一两种委托

事件中心的核心挑战是：有的事件需要传参数（`UnityAction<T>`），有的事件不需要（`UnityAction`），但它们要存在同一个 `Dictionary<string, ???>` 里。字典的 value 类型必须一致。

解决方案：用一个**空接口 `IEventInfo`** 作为标记，让两种包装类都实现它，然后字典存 `IEventInfo`，用的时候再 `as` 转回具体类型。

```csharp
public interface IEventInfo
{
    // 空接口，只做类型标记用
}
```

这种模式叫做**标记接口（Marker Interface）**——接口本身没有任何方法，纯粹为了把不同类型归到同一个集合中。

### EventInfo vs EventInfo\<T\> — 包装委托的两种容器

```csharp
// 无参版 —— 包装 UnityAction
public class EventInfo : IEventInfo
{
    public UnityAction actions;

    public EventInfo(UnityAction action)
    {
        actions += action;
    }
}

// 有参版 —— 包装 UnityAction<T>
public class EventInfo<T> : IEventInfo
{
    public UnityAction<T> actions;

    public EventInfo(UnityAction<T> action)
    {
        actions += action;
    }
}
```

这两个类非常简洁，作用就是把委托"装"起来：

- **构造函数直接 `+=`**：创建容器时就把第一个监听者加进去，后续的监听者通过 `AddEventListener` 中 `actions += action` 追加
- **`actions` 是 public 字段**：因为委托支持 `+=` 多播，一个 `UnityAction` 可以链式持有多个方法。`actions += f1` 后 `actions += f2`，调用 `actions.Invoke()` 时 f1 和 f2 都会被依次调用

为什么需要两个类而不是一个？因为 `UnityAction` 和 `UnityAction<T>` 是**两个不同的类型**，没法统一成一个字段。泛型特化让有参版携带不同类型的参数。

### EventCenter — 事件分发的核心

```csharp
public class EventCenter : BaseManager<EventCenter>
{
    //key —— 事件的名字（比如：怪物死亡，玩家死亡，通关 等等）
    //value —— 对应的是 监听这个事件 对应的委托函数们
    private Dictionary<string, IEventInfo> eventDic = new Dictionary<string, IEventInfo>();
```

`eventDic` 是核心数据结构：key 用 `string` 作为事件名（如 `"怪物死亡"`、`"玩家升级"`），value 是 `IEventInfo`，通过 `as` 转换后访问内部委托。

#### AddEventListener — 订阅事件

```csharp
// 有参版
public void AddEventListener<T>(string name, UnityAction<T> action)
{
    if (eventDic.ContainsKey(name))
    {
        (eventDic[name] as EventInfo<T>).actions += action;  // 已有事件 → 追加
    }
    else
    {
        eventDic.Add(name, new EventInfo<T>(action));         // 新事件 → 创建容器
    }
}

// 无参版
public void AddEventListener(string name, UnityAction action)
{
    if (eventDic.ContainsKey(name))
    {
        (eventDic[name] as EventInfo).actions += action;
    }
    else
    {
        eventDic.Add(name, new EventInfo(action));
    }
}
```

逻辑很简单：

- 字典里已有该事件 → 直接 `+=` 追加到委托链末尾
- 字典里还没有 → `new EventInfo<T>(action)` 创建容器并放入字典

**关键细节 `(eventDic[name] as EventInfo<T>)`**：字典的 value 类型是 `IEventInfo` 接口，但我们要访问 `.actions` 字段就必须转回具体类型。`as` 是 C# 的安全类型转换——转成功了返回对象，失败了返回 null 而不抛异常。

这里的设计有一个约定：**同一个事件名对应的 EventInfo 类型必须在首次 AddEventListener 时就确定**。第一次用有参版添加 `"怪物死亡"`，后续必须也用有参版；混用无参版会导致 `as EventInfo` 转换失败，后续操作静默无效（因为 `as` 返回 null，但代码没有判空）。实际项目中通常不会在同一个事件名上混用两种版本。

#### RemoveEventListener — 取消订阅

```csharp
// 有参版
public void RemoveEventListener<T>(string name, UnityAction<T> action)
{
    if (eventDic.ContainsKey(name))
        (eventDic[name] as EventInfo<T>).actions -= action;
}

// 无参版
public void RemoveEventListener(string name, UnityAction action)
{
    if (eventDic.ContainsKey(name))
        (eventDic[name] as EventInfo).actions -= action;
}
```

取消订阅的逻辑和添加对称——`-=` 从委托链中移除指定方法。

#### EventTrigger — 触发事件

```csharp
// 有参版
public void EventTrigger<T>(string name, T info)
{
    if (eventDic.ContainsKey(name))
    {
        if ((eventDic[name] as EventInfo<T>).actions != null)
            (eventDic[name] as EventInfo<T>).actions.Invoke(info);
    }
}

// 无参版
public void EventTrigger(string name)
{
    if (eventDic.ContainsKey(name))
    {
        if ((eventDic[name] as EventInfo).actions != null)
            (eventDic[name] as EventInfo).actions.Invoke();
    }
}
```

触发时做了两层检查：

1. **`ContainsKey`**：这个事件名有没有人订阅过
2. **`actions != null`**：委托链是否为空（虽然 Add 了就不会空，但 Remove 全部后可能变成 null，或者接口转换失败导致 null）

`actions.Invoke(info)` 会依次调用委托链上的所有方法。调用顺序是添加顺序，但**不要依赖调用顺序**——这是观察者模式的惯例。

#### Clear — 场景切换清理

```csharp
public void Clear()
{
    eventDic.Clear();
}
```

场景切换时，旧场景的对象会被销毁，但它们订阅的委托可能还留在事件中心里，指向已销毁对象的回调方法。`Clear()` 一次性清空所有订阅，避免野指针。调用时机通常在场景切换前或切换后立刻执行。

### 使用示例

**完整流程：怪物死亡 → 多方响应**

```csharp

public class AchievementSystem : MonoBehaviour
{
    void Awake()
    {
        EventCenter.GetInstance().AddEventListener<Monster>("怪物死亡", OnMonsterDie);
    }

    void OnDestroy()
    {
        // 千万别忘了取消订阅！
        EventCenter.GetInstance().RemoveEventListener<Monster>("怪物死亡", OnMonsterDie);
    }

    void OnMonsterDie(Monster monster)
    {
        Debug.Log($"成就进度更新：击杀 {monster.monsterName}");
    }
}

// ===== 怪物脚本 =====
public class Monster : MonoBehaviour
{
    public string monsterName;

    void Die()
    {
        // 就这一行 —— 触发事件

        Destroy(gameObject);
    }
}
```

**无参事件示例：**

```csharp
{
    Debug.Log("游戏结束了！");
});
EventCenter.GetInstance().EventTrigger("游戏结束");
```

### 为什么一定要 RemoveEventListener

这是事件中心最容易踩的坑。委托持有对方法的引用，而方法又属于某个对象：

```
eventDic["怪物死亡"].actions → 持有 → MonsterDieCallback → 持有 → AchievementSystem 实例
```

如果你在 `AchievementSystem` 的 `OnDestroy` 中没有 `RemoveEventListener`：

1. `AchievementSystem` 的 GameObject 被销毁了
2. 但 eventDic 中的委托链还持有它的方法引用
3. GC 看到"还有引用指向这个对象"，不会回收它 → **内存泄漏**
4. 下次触发 `"怪物死亡"` 事件时，会调用一个已销毁对象上的方法 → **报错或异常行为**

规则很简单：**`AddEventListener` 和 `RemoveEventListener` 必须成对出现**，就像 `+=` 和 `-=`。

### 注意事项

- **事件名字符串管理**：用裸字符串容易拼错（`"怪物死亡"` vs `"怪物死亡!"`）。建议用常量类或枚举统一管理事件名，比如 `public const string EVENT_MONSTER_DIE = "怪物死亡";`
- **不要依赖回调执行顺序**：`Invoke` 按添加顺序调用，但这是实现细节而非约定。如果需要顺序控制，自己分事件或用其他机制。
- **触发时不要修改订阅列表**：在事件回调中 Add/Remove 同一个事件的监听者是危险操作（在遍历时修改集合）。如果确实需要，考虑延迟处理。
- **性能考虑**：事件中心本质是字典查找 + 委托调用，性能开销很低。但如果在 Update 中高频触发（每帧数千次），可以考虑用其他方案（如直接引用调用）。正常使用场景（几十到几百个订阅，偶尔触发）完全没问题。

## 场景切换

### 解决什么问题

Unity 的场景加载 API 本身不复杂——`SceneManager.LoadScene("场景名")` 一行就搞定了。但直接裸调有几个不足：

- **同步加载会卡顿**：大场景的同步加载会冻结主线程好几秒，玩家看到画面定住，体验很差
- **缺少进度反馈**：直接 `LoadScene` 没有进度信息，没法做 loading 条
- **加载完成后要做事**：比如场景加载完后初始化角色位置、开启 BGM、隐藏 loading 界面等，需要有回调机制

场景切换模块就是对 Unity 原生 API 的一层薄封装，加上协程异步 + 事件中心进度广播。

### 同步加载 — `LoadScene`

```csharp
/// <summary>
/// 切换场景 同步
/// </summary>
/// <param name="name"></param>
public void LoadScene(string name, UnityAction fun)
{
    //场景同步加载
    SceneManager.LoadScene(name);
    //加载完成过后 才会去执行fun
    fun();
}
```

逻辑极其简单：调用 `SceneManager.LoadScene` → 等引擎加载完毕 → 执行回调 `fun()`。全程阻塞，`fun` 的执行一定在场景完全加载之后。

同步加载只在**场景非常小**（比如一个空场景、一个简单 UI 场景）时用。大多数情况下应该用异步加载。

### 异步加载 — `LoadSceneAsyn`

异步加载是这个模块的核心。它借助两个前面学过的模块——**公共 Mono**（提供协程）和**事件中心**（广播进度）：

```csharp
/// <summary>
/// 提供给外部的 异步加载的接口方法
/// </summary>
/// <param name="name"></param>
/// <param name="fun"></param>
public void LoadSceneAsyn(string name, UnityAction fun)
{
    MonoMgr.GetInstance().StartCoroutine(ReallyLoadSceneAsyn(name, fun));
}
```

外部调用 `LoadSceneAsyn` 时，它不直接做加载工作，而是启动一个协程，真正的逻辑在 `ReallyLoadSceneAsyn` 中。

### 协程内部的加载循环

```csharp
/// <summary>
/// 协程异步加载场景
/// </summary>
/// <param name="name"></param>
/// <param name="fun"></param>
/// <returns></returns>
private IEnumerator ReallyLoadSceneAsyn(string name, UnityAction fun)
{
    AsyncOperation ao = SceneManager.LoadSceneAsync(name);
    //可以得到场景加载的一个进度
    while (!ao.isDone)
    {
        //事件中心 向外分发 进度情况  外面想用就用
        EventCenter.GetInstance().EventTrigger("进度条更新", ao.progress);
        //这里面去更新进度条
        yield return ao.progress;
    }
    //加载完成过后 才会去执行fun
    fun();
}
```

逐行拆解：

1. **`SceneManager.LoadSceneAsync(name)`**：Unity 在后台开始异步加载场景，立即返回一个 `AsyncOperation` 对象。这个对象有两个关键属性：

   - `progress`：加载进度，0.0 ~ 1.0（注意：Unity 的进度值 0.0 ~ 0.9 是实际加载过程，0.9 是"等待激活"阶段，到 1.0 表示场景完全激活）
   - `isDone`：是否加载完成
2. **`while (!ao.isDone)` 循环**：只要还没加载完，就每帧执行循环体。循环体内的 `yield return` 让协程在每帧末尾挂起，不会阻塞主线程。循环退出是由 `ao.isDone` 变为 `true` 控制的。
3. **`EventCenter.GetInstance().EventTrigger("进度条更新", ao.progress)`**：通过事件中心广播当前进度。任何需要做 loading 条的 UI 面板，只要订阅 `"进度条更新"` 事件就能拿到实时的进度值（一个 0~1 的 float）。
4. **`yield return ao.progress`**：这一行的作用是**等一帧再继续循环**。注意这里 yield 的是 `ao.progress`（一个 float 值），不是 `ao`（AsyncOperation 对象）。Unity 协程调度器只认特定类型做特殊处理——`AsyncOperation` 会让协程等到操作完成才继续，但 `float` 不在白名单里，Unity 不认识它，直接当 `null` 处理，等价于等一帧。真正让循环在加载完成后退出的，是外层的 `while (!ao.isDone)`。

   **扩展：`yield return` 的不同类型**

   Unity 协程只对以下类型做特殊处理：

   | `yield return` 的值                             | 行为                                     |
   | ------------------------------------------------- | ---------------------------------------- |
   | `null`                                          | 等一帧                                   |
   | `WaitForSeconds` / `WaitForEndOfFrame` 等     | 各自有特殊行为                           |
   | `AsyncOperation`                                | 等异步操作完成                           |
   | `CustomYieldInstruction`                        | 自定义等待逻辑                           |
   | **其他一切**（float、int、普通 object……） | **不认识 → 当 null 处理，等一帧** |

   所以 `yield return ao.progress`、`yield return null`、`yield return 0` 在效果上完全一样，都是暂停一帧。唯一的小区别是 float 是值类型，yield 时会装箱（boxing），在堆上分配临时对象，有微小 GC 开销，`yield return null` 没有这个开销。如果追求严谨，这里写成 `yield return null` 更干净。
5. **`fun()`**：加载完成后执行回调，比如初始化场景、播放背景音乐、隐藏 loading 面板。

### 依赖的三层协作

```
ScenesMgr.LoadSceneAsyn("BattleScene", callback)
    ├─ MonoMgr.GetInstance().StartCoroutine(...)     ← 借协程
    │     └─ ReallyLoadSceneAsyn 协程
    │           ├─ SceneManager.LoadSceneAsync        ← Unity 原生 API
    │           └─ EventCenter.EventTrigger("进度条更新", progress)  ← 广播进度
    └─ callback()  ← 加载完成后回调
```

场景切换模块本身只有 20 行逻辑代码，但它串联了框架中两个关键基础设施：**公共 Mono** 和 **事件中心**。这也是框架搭建的意义所在——基础模块打好后，上层模块写起来非常轻量。

### 使用示例

**完整流程：点击按钮 → 异步切场景 + loading 条**

```csharp
// ===== UI 按钮上挂的脚本 =====

{
    // 先显示 loading 面板
    UIManager.GetInstance().ShowPanel<LoadingPanel>("LoadingPanel");

    // 异步加载战斗场景

    {
        // 场景加载完成后的收尾工作
        MusicMgr.GetInstance().PlayBkMusic("Battle");
        UIManager.GetInstance().HidePanel("LoadingPanel");
        Debug.Log("战斗场景加载完毕");
    });
}

// ===== LoadingPanel — 进度条 UI =====
public class LoadingPanel : BasePanel
{
    public Slider progressBar;

    void Awake()
    {
        // 订阅进度更新事件
        EventCenter.GetInstance().AddEventListener<float>("进度条更新", OnProgressUpdate);
    }

    void OnDestroy()

        EventCenter.GetInstance().RemoveEventListener<float>("进度条更新", OnProgressUpdate);
```

### 注意事项

- **同步加载只适合小场景**：`LoadScene` 会在加载期间冻结整个游戏。对于打包后体积较大的场景，哪怕 2~3 秒的冻结也很影响体验。正式项目基本都用异步。
- **`AsyncOperation.progress` 到不了 1.0**：Unity 的异步加载默认会在 `progress` 达到 0.9 时等待场景激活（`allowSceneActivation = true`），激活后才跳到 1.0。如果发现进度条卡在 90%，不是 bug，是场景在等激活。可以在 `while` 循环中手动判断 `ao.progress >= 0.9f` 后设置 `ao.allowSceneActivation = true` 来控制激活时机。
- **场景名必须在 Build Settings 中注册**：`SceneManager.LoadScene` 只能加载已添加到 Build Settings → Scenes In Build 列表中的场景，否则会报错。用场景索引（int）替代场景名（string）可以避免拼写错误。
- **回调中的安全操作**：`fun()` 在场景加载完成后执行，此时新场景的 GameObject 已经存在，但它们的 `Awake` 和 `Start` 可能还没跑完。如果回调中需要访问新场景的对象，最好延迟一帧再操作。

## 输入控制

### 解决什么问题

`Input.GetKeyDown(KeyCode.W)` 写起来很简单，但如果在每个需要响应按键的脚本里都散落着 `Input.GetKeyDown`，会带来几个问题：

- **代码分散**：玩家移动脚本里有 WASD 检测，UI 脚本里有 ESC 检测，战斗脚本里有技能快捷键检测……每个地方都在各自轮询输入，难以统一管理
- **无法控制开关**：对话中要禁用移动输入、过场动画中要禁用所有操作——散落的 Input 检测很难统一关闭
- **按键和逻辑耦合**：移动脚本直接依赖 `KeyCode.W`，想改键位（比如 W → 上箭头）就得改逻辑代码

输入控制模块的思路是：**把输入检测集中到一处，每帧统一检测，通过事件中心分发结果**。谁需要响应按键，就去订阅事件，完全不碰 `Input` 类。

### 核心设计

```
InputMgr (每帧检测)
  ├─ MonoMgr.Update → MyUpdate()
  │     ├─ StartOrEndCheck 控制开关
  │     └─ 遍历 WASD → CheckKeyCode(KeyCode)
  │
  └─ EventCenter.EventTrigger("某键按下", KeyCode.W)
        └─ → 玩家移动脚本监听 → 向前移动
        └─ → 其他脚本也可以同时监听同一个键
```

- **构造函数订阅 Update**：`MonoMgr.GetInstance().AddUpdateListener(MyUpdate)`，每帧自动检测
- **开关控制**：`StartOrEndCheck(bool)` 控制是否检测，对话/过场时关闭即可
- **事件分发**：检测到按键后通过事件中心广播 `"某键按下"` / `"某键抬起"`，携带 `KeyCode` 参数让监听方知道是哪个键

### 代码实现

```csharp
using UnityEngine;

/// <summary>
/// 1.Input类
/// 2.事件中心模块
/// 3.公共Mono模块的使用
/// </summary>
public class InputMgr : BaseManager<InputMgr>
{

    private bool isStart = false;
    /// <summary>
    /// 构造函数中 添加Updata监听
    /// </summary>
    public InputMgr()
    {
        MonoMgr.GetInstance().AddUpdateListener(MyUpdate);
    }

    /// <summary>
    /// 是否开启或关闭 我的输入检测
    /// </summary>
    public void StartOrEndCheck(bool isOpen)
    {
        isStart = isOpen;
    }

    /// <summary>
    /// 用来检测按键抬起按下 分发事件的
    /// </summary>
    /// <param name="key"></param>
    private void CheckKeyCode(KeyCode key)
    {
        //事件中心模块 分发按下抬起事件
        if (Input.GetKeyDown(key))
            EventCenter.GetInstance().EventTrigger("某键按下", key);
        //事件中心模块 分发按下抬起事件
        if (Input.GetKeyUp(key))
            EventCenter.GetInstance().EventTrigger("某键抬起", key);
    }

    private void MyUpdate()
    {
        //没有开启输入检测 就不去检测 直接return
        if (!isStart)
            return;

        CheckKeyCode(KeyCode.W);
        CheckKeyCode(KeyCode.S);
        CheckKeyCode(KeyCode.A);
        CheckKeyCode(KeyCode.D);
    }
}
```

逐段拆解：

**构造函数中的 Update 订阅**：`MonoMgr.GetInstance().AddUpdateListener(MyUpdate)` 把 `MyUpdate` 注册到公共 Mono 模块的帧循环中。因为 `InputMgr` 继承 `BaseManager<InputMgr>` 是纯 C# 类，自己没有 Update，所以借用 MonoMgr 的 Update 来每帧执行检测。首次调用 `GetInstance()` 时构造函数自动执行，订阅即刻生效。

**`StartOrEndCheck(bool)` 开关**：`isStart` 是一个简单的 bool 标志。设为 `false` 时 `MyUpdate` 直接 return，所有按键检测停止。典型使用场景：对话中禁用移动、打开菜单时禁用操作、过场动画中屏蔽输入。

**`CheckKeyCode(KeyCode)` 检测单个按键**：对传入的按键做两种检测——`GetKeyDown`（按下那一帧）和 `GetKeyUp`（抬起那一帧）。检测到后通过事件中心广播，事件名固定为 `"某键按下"` 和 `"某键抬起"`，事件参数是 `KeyCode` 枚举值，监听方可以根据 key 的值判断具体是哪个键。

**`MyUpdate()` 每帧遍历**：目前只检测了 WASD 四个键。按需扩展——需要更多键就在 `MyUpdate` 里加一行 `CheckKeyCode(KeyCode.XXX)` 即可。

### `GetKeyDown` vs `GetKey` vs `GetKeyUp`

这三个方法的区别是初学者最容易搞混的：

| 方法                 | 触发时机                   | 调用 `CheckKeyCode(KeyCode.W)` 时的行为 |
| -------------------- | -------------------------- | ----------------------------------------- |
| `Input.GetKeyDown` | 按键**按下**的那一帧 | 只在按下的瞬间触发一次                    |
| `Input.GetKey`     | 按键**按住**期间     | 每一帧都触发（持续触发）                  |
| `Input.GetKeyUp`   | 按键**松开**的那一帧 | 只在松开的瞬间触发一次                    |

框架只用 `GetKeyDown` 和 `GetKeyUp`，没有用 `GetKey`。因为持续按住的需求可以通过"监听按下开始 + 抬起结束"自己实现，而且 `GetKey` 每帧都触发事件会带来不必要的性能开销和逻辑复杂度。

### 为什么用事件中心而不是直接调用移动逻辑

你可能会有疑问：既然 InputMgr 知道 W 被按下了，为什么不直接调用 `Player.MoveForward()`？

因为 **InputMgr 只负责"检测输入"**，不应该知道"谁响应输入"。如果它直接调用移动逻辑，那就和移动系统耦合了——以后换一个移动脚本、加一个摄像机控制、或者让另一个系统也响应 W 键（比如 UI 导航），都得改 InputMgr 的代码。

通过事件中心分发后：

```csharp
// InputMgr 只管触发 —— 不知道谁在听
EventCenter.GetInstance().EventTrigger("某键按下", KeyCode.W);

// 谁需要 W 键，谁自己去订阅
EventCenter.GetInstance().AddEventListener<KeyCode>("某键按下", OnKeyDown);

void OnKeyDown(KeyCode key)
{
    if (key == KeyCode.W) MoveForward();   // 移动系统
    if (key == KeyCode.W) ScrollUp();      // UI导航系统
    if (key == KeyCode.W) SwitchWeapon();  // 武器切换系统（按住W滚轮？随你）
}
```

新增一个响应 W 键的系统，不需要改 InputMgr 一行代码。这就是事件驱动架构的灵活性。

### 使用示例

**玩家移动脚本：**

```csharp
public class PlayerMovement : MonoBehaviour
{
    void Awake()
    {
        // 开启输入检测
        InputMgr.GetInstance().StartOrEndCheck(true);
        // 订阅按键事件
        EventCenter.GetInstance().AddEventListener<KeyCode>("某键按下", OnKeyDown);
        EventCenter.GetInstance().AddEventListener<KeyCode>("某键抬起", OnKeyUp);
    }

    void OnDestroy()
    {
        EventCenter.GetInstance().RemoveEventListener<KeyCode>("某键按下", OnKeyDown);
        EventCenter.GetInstance().RemoveEventListener<KeyCode>("某键抬起", OnKeyUp);
    }

    void OnKeyDown(KeyCode key)
    {
        switch (key)
        {
            case KeyCode.W: startMoving(Vector3.forward); break;
            case KeyCode.S: startMoving(Vector3.back); break;
            case KeyCode.A: startMoving(Vector3.left); break;
            case KeyCode.D: startMoving(Vector3.right); break;
        }
    }

    void OnKeyUp(KeyCode key)
    {
        // 松开按键时停止移动
        if (key == KeyCode.W || key == KeyCode.S || key == KeyCode.A || key == KeyCode.D)
            stopMoving();
    }

    void startMoving(Vector3 dir) { /* ... */ }
    void stopMoving() { /* ... */ }
}
```

**对话中禁用移动输入：**

```csharp
// 进入对话
InputMgr.GetInstance().StartOrEndCheck(false);

// 对话结束
InputMgr.GetInstance().StartOrEndCheck(true);
```

### 扩展：怎么加更多按键

框架目前只检测了 WASD，实际项目中通常需要更多键。扩展很简单，在 `MyUpdate` 中加：

```csharp
private void MyUpdate()
{
    if (!isStart) return;

    // 移动键
    CheckKeyCode(KeyCode.W);
    CheckKeyCode(KeyCode.S);
    CheckKeyCode(KeyCode.A);
    CheckKeyCode(KeyCode.D);

    // 功能键 —— 按需添加
    CheckKeyCode(KeyCode.Escape);
    CheckKeyCode(KeyCode.Space);
    CheckKeyCode(KeyCode.E);
    CheckKeyCode(KeyCode.Tab);
    CheckKeyCode(KeyCode.LeftShift);
}
```

不需要改任何其他代码，监听方也只需要在回调中 `switch` 新加的 key 就行。

### 注意事项

- **按键事件名统一**：所有按键都通过同一个事件名 `"某键按下"` / `"某键抬起"` 分发，通过参数 `KeyCode` 区分具体按键。这意味着所有监听者会收到所有按键事件，需要在回调中自己过滤关心的键——这样做的好处是只需订阅一次，坏处是每次按键所有监听者都被唤醒。对于 WASD + 几个功能键的规模来说，性能完全没问题。
- **按键响应频率**：`GetKeyDown` 和 `GetKeyUp` 只在状态变化的那一帧触发，不是持续触发。如果玩家按下 W 不放，`GetKeyDown` 只在按下的第一帧触发一次，后续帧不会再触发——直到松开再按下。如果需要"按住 W 持续前进"的效果，在按下那一帧开始移动，抬起那一帧停止移动（见上面的 PlayerMovement 示例）。
- **不要忘了开启检测**：`isStart` 默认是 `false`，必须在游戏开始时调用 `StartOrEndCheck(true)` 才能让输入生效。这个设计是有意为之——输入检测应该在游戏正确初始化后再开启，避免加载阶段误响应按键。
- **`KeyCode` 是 Unity 的枚举**：它包含了键盘所有键、鼠标按钮、手柄按钮等。框架目前只检测键盘按键，需要鼠标或手柄检测可以直接加 `CheckKeyCode(KeyCode.Mouse0)` 等。

## 音效管理

### 解决什么问题

游戏中的音频分两种：

- **背景音乐（BGM）**：同一时间只播一首，循环播放，贯穿整个场景。比如主菜单的主题曲、战斗场景的激昂配乐
- **音效（SFX）**：短促、可能同时播放多个，播完就停。比如按钮点击声、子弹发射声、爆炸声、怪物叫声

Unity 原生 API `AudioSource.Play()` 能用，但如果每个脚本都自己挂 AudioSource 组件、自己加载 AudioClip、自己管理生命周期，很快就会乱套。音效管理模块把两种音频统一管理起来，外部只需告诉它"播什么"，不用关心底层细节。

### 核心设计

```
MusicMgr
├─ 背景音乐
│   ├─ bkMusic: AudioSource (全局唯一)
│   ├─ bkValue: float (音量，独立于音效)
│   └─ PlayBkMusic / PauseBKMusic / StopBKMusic / ChangeBKValue
│
├─ 音效
│   ├─ soundObj: GameObject (音效组件挂载的父节点)
│   ├─ soundList: List<AudioSource> (所有正在播放的音效)
│   ├─ soundValue: float (音效音量，独立于背景音乐)
│   ├─ PlaySound / StopSound / ChangeSoundValue
│   └─ Update 中自动清理播放完毕的音效
│
└─ 依赖
    ├─ MonoMgr → Update 自动清理
    └─ ResMgr → 异步加载 AudioClip
```

背景音乐和音效完全分开管理：两个独立 AudioSource（或列表）、两套独立音量。互不干扰。

### 背景音乐

背景音乐只有一个 `AudioSource`（`bkMusic`），因为同一时间只会有一首背景音乐在播。切换时直接换 `clip` 就行。

```csharp
//唯一的背景音乐组件
private AudioSource bkMusic = null;
//音乐大小
private float bkValue = 1;

public MusicMgr()
{
    MonoMgr.GetInstance().AddUpdateListener(Update);
}

/// <summary>
/// 播放背景音乐
/// </summary>
/// <param name="name"></param>
public void PlayBkMusic(string name)
{
    if (bkMusic == null)
    {
        GameObject obj = new GameObject();
        obj.name = "BkMusic";
        bkMusic = obj.AddComponent<AudioSource>();
    }
    //异步加载背景音乐 加载完成后 播放
    ResMgr.GetInstance().LoadAsync<AudioClip>("Music/BK/" + name, (clip) =>
    {
        bkMusic.clip = clip;
        bkMusic.loop = true;
        bkMusic.volume = bkValue;
        bkMusic.Play();
    });
}
```

关键点：

- **懒创建 AudioSource**：第一次播放时才 `new GameObject + AddComponent<AudioSource>`，而不是构造函数里就创建。如果用不到背景音乐（比如纯 UI 场景），就不会浪费资源
- **异步加载 AudioClip**：通过 `ResMgr.GetInstance().LoadAsync<AudioClip>(...)` 异步加载音频资源，不卡主线程。资源路径约定在 `"Music/BK/"` 下
- **`bkMusic.loop = true`**：背景音乐循环播放，不会播一遍就停
- **`bkMusic.volume = bkValue`**：音量用存储的 `bkValue` 设置。这样改过音量后切歌，新歌也自动用新音量

**暂停、停止、改音量**：

```csharp
public void PauseBKMusic()
{
    if (bkMusic == null) return;
    bkMusic.Pause();   // 暂停，Resume 可以从暂停位置继续
}

public void StopBKMusic()
{
    if (bkMusic == null) return;
    bkMusic.Stop();    // 停止，下次 Play 从头开始
}

public void ChangeBKValue(float v)
{
    bkValue = v;       // 先存起来
    if (bkMusic == null) return;
    bkMusic.volume = bkValue;  // 如果已在播放，立刻生效
}
```

`ChangeBKValue` 的设计值得注意：**音量值单独存储在 `bkValue` 字段中**，而不是直接改 `bkMusic.volume`。这样即使当前没有在播放音乐（`bkMusic == null`），设置的音量值也能保留，下次 `PlayBkMusic` 时自动应用——因为 `PlayBkMusic` 中写了 `bkMusic.volume = bkValue`。

### 音效

音效和背景音乐不同——同一时间可能有多个音效同时播放（子弹声 + 爆炸声 + UI 点击声……），所以需要 `List<AudioSource>` 来管理多个 AudioSource。

**播放音效**：

```csharp
//音效依附对象
private GameObject soundObj = null;
//音效列表
private List<AudioSource> soundList = new List<AudioSource>();
//音效大小
private float soundValue = 1;

public void PlaySound(string name, bool isLoop, UnityAction<AudioSource> callBack = null)
{
    if (soundObj == null)
    {
        soundObj = new GameObject();
        soundObj.name = "Sound";
    }
    //当音效资源异步加载结束后 再添加一个音效
    ResMgr.GetInstance().LoadAsync<AudioClip>("Music/Sound/" + name, (clip) =>
    {
        AudioSource source = soundObj.AddComponent<AudioSource>();
        source.clip = clip;
        source.loop = isLoop;
        source.volume = soundValue;
        source.Play();
        soundList.Add(source);
        if (callBack != null)
            callBack(source);
    });
}
```

关键点：

- **`soundObj` 是所有音效的父节点**：音效的 AudioSource 通过 `soundObj.AddComponent<AudioSource>()` 动态添加。注意这里不是 `new GameObject` 再 AddComponent，而是直接在已有的 `soundObj` 上 AddComponent——多个 AudioSource 可以挂在同一个 GameObject 上
- **异步加载 + 动态创建**：每次播音效都是异步加载 AudioClip → 加载完成后 AddComponent → 设置参数 → Play → 加入列表
- **`isLoop` 参数**：允许某些音效循环播放（比如引擎轰鸣声），大部分音效为 false
- **回调 `callBack`**：可选的，播放完成后外部可以拿到 AudioSource 引用（比如需要后续手动停止某个特定音效）

**Update 自动清理播放完毕的音效**：

```csharp
private void Update()
{
    for (int i = soundList.Count - 1; i >= 0; --i)
    {
        if (!soundList[i].isPlaying)
        {
            GameObject.Destroy(soundList[i]);
            soundList.RemoveAt(i);
        }
    }
}
```

这是音效管理中最关键的逻辑——**倒序遍历 + 自动清理**。

为什么要倒序？看一个具体的例子。假设 soundList 有三个音效[A,B,C]，索引 0 和 1 都播完了：

```
正序遍历（错误）：
i=0: isPlaying=false → RemoveAt(0) → 列表变成 [B, C]（原1,2前移）
i=1: 检查的是原来的2号(C)，跳过了原来的1号（即新的0号B）← BUG！

倒序遍历（正确）：
i=2: isPlaying=true → 保留
i=1: isPlaying=false → RemoveAt(1) → 列表变成 [A, C]，后面的索引不受影响
i=0: isPlaying=false → RemoveAt(0) → 列表变成 [C]
```

倒序从末尾开始删除，前面的元素索引不会变化，不会漏删。这是遍历列表并删除元素的标准做法。

**改音效音量和停止音效**：

```csharp
public void ChangeSoundValue(float value)
{
    soundValue = value;
    for (int i = 0; i < soundList.Count; ++i)
        soundList[i].volume = value;
}

public void StopSound(AudioSource source)
{
    if (soundList.Contains(source))
    {
        soundList.Remove(source);
        source.Stop();
        GameObject.Destroy(source);
    }
}
```

`ChangeSoundValue` 和背景音乐的音量逻辑一样——先存值，然后遍历所有正在播放的音效立刻应用。

`StopSound` 手动停止一个特定音效（比如角色闭嘴时停止语音）。注意要先从列表移除、再 Stop 再 Destroy——如果先 Destroy 了，列表中残留的引用就变成 `null`。

### 构造函数中的 Update 订阅

```csharp
public MusicMgr()
{
    MonoMgr.GetInstance().AddUpdateListener(Update);
}
```

MusicMgr 订阅了 MonoMgr 的 Update，用于每帧检查音效是否播放完毕。注意方法名就叫 `Update`，但它不是 MonoBehaviour 的 `Update()`——它是通过公共 Mono 模块的帧循环被调用的普通方法。

### 依赖关系

```
MusicMgr.PlayBkMusic / PlaySound
    └─ ResMgr.GetInstance().LoadAsync<AudioClip>(...)
           └─ MonoMgr.GetInstance().StartCoroutine(...)

MusicMgr.Update (音效清理)
    └─ MonoMgr.GetInstance().AddUpdateListener(...)
```

音效管理同时依赖了**资源加载**（异步加载音频文件）和**公共 Mono**（帧循环清理音效 + 异步加载用的协程）。

### 使用示例

```csharp
// 进入主菜单 —— 播放背景音乐
MusicMgr.GetInstance().PlayBkMusic("MainMenu");

// 调整背景音乐音量为 50%
MusicMgr.GetInstance().ChangeBKValue(0.5f);

// 点击按钮 —— 播放音效
MusicMgr.GetInstance().PlaySound("Click", false);

// 播放循环音效（引擎声），并保存引用以便后续停止
AudioSource engineSound = null;
MusicMgr.GetInstance().PlaySound("Engine", true, (source) =>
{
    engineSound = source;
});

// 停止引擎声
MusicMgr.GetInstance().StopSound(engineSound);

// 暂停 / 继续背景音乐
MusicMgr.GetInstance().PauseBKMusic();   // 切到后台时暂停
MusicMgr.GetInstance().PlayBkMusic(...); // 恢复时重新播放
```

### 注意事项

- **AudioSource 数量上限**：每个 `soundObj` 上可以挂多个 AudioSource 组件，但同一帧同时播放的音效数量太多时 AddComponent 有开销。如果项目中有极端需求（比如弹幕游戏的数百个子弹同时发声），应该配合缓存池预先创建 AudioSource
- **Update 中的 `GameObject.Destroy`**：对 AudioSource 组件调用 Destroy 会销毁该组件，不会销毁整个 `soundObj`。实际上这里传的是 AudioSource 组件引用，Unity 会只销毁这个组件
- **路径约定**：背景音乐路径 `"Music/BK/" + name`，音效路径 `"Music/Sound/" + name`，拼接后的完整路径如 `"Music/BK/MainMenu"`。对应的资源文件必须放在 `Resources/Music/BK/` 和 `Resources/Music/Sound/` 目录下（因为 ResMgr 用的是 `Resources.LoadAsync`）
- **`ChangeBKValue` 在音乐加载前就调用也有效**：因为音量值存在 `bkValue` 字段中，`PlayBkMusic` 加载完成后会用 `bkValue` 设置音量。所以可以在一开始就设置音量，后续播放的音乐自动生效

## UI管理

### 解决什么问题

UI 系统是游戏开发中最繁琐的部分之一。通常你会面对两个痛点：

1. **手动拖控件**：每个按钮、文本框、图片都要在 Inspector 里拖到脚本的 public 字段上，十几二十个控件拖到眼花。更糟的是，改了 Hierarchy 中的名字还得重新拖一遍
2. **面板管理分散**：每个面板各自 `Instantiate` + `SetActive`，没有统一的显隐管理。同一个面板可能被重复加载两份，切场景时旧的 UI 残留在新场景中

UI 管理模块用两个类解决这两个问题：**BasePanel** 自动查找和绑定子控件，**UIManager** 统一管理所有面板的加载、显示、隐藏和层级。

### 整体架构

```
UIManager (单例)
├─ Canvas (DontDestroyOnLoad)
│   ├─ Bot    (底层 — 背景、装饰)
│   ├─ Mid    (中层 — 默认层级，大部分面板)
│   ├─ Top    (顶层 — 弹窗、提示)
│   └─ System (系统层 — 网络提示、错误弹窗，永远在最上面)
├─ EventSystem (DontDestroyOnLoad)
└─ panelDic: Dictionary<string, BasePanel>
    └─ 所有已加载的面板实例
```

每个面板预制体上挂一个继承 `BasePanel` 的脚本。Awake 时自动扫描所有子控件存到字典里，之后通过名字就能拿到任意控件的引用。

### BasePanel — 自动查找子控件

#### 里氏转换（Liskov Substitution）

BasePanel 的核心依赖一个重要的面向对象原则——里氏转换：**子类对象可以赋值给父类变量**。Unity 的所有 UI 控件（Button、Image、Text……）最终都继承自 `UIBehaviour`，所以它们都可以被当作 `UIBehaviour` 存到同一个容器里。

```
UIBehaviour (基类)
  ├─ Graphic
  │   ├─ MaskableGraphic
  │   │   ├─ Image
  │   │   ├─ Text
  │   │   └─ RawImage
  ├─ Selectable
  │   ├─ Button
  │   ├─ Toggle
  │   ├─ Slider
  │   ├─ Scrollbar
  │   ├─ Dropdown
  │   └─ InputField
  └─ ScrollRect
```

这个继承关系让 BasePanel 可以用**一个 `Dictionary<string, List<UIBehaviour>>`** 装下所有类型的 UI 控件。

#### FindChildrenControl — 扫描子控件

```csharp
public class BasePanel : MonoBehaviour
{
    //通过里式转换原则 来存储所有的控件
    private Dictionary<string, List<UIBehaviour>> controlDic = new Dictionary<string, List<UIBehaviour>>();

    protected virtual void Awake()
    {
        FindChildrenControl<Button>();
        FindChildrenControl<Image>();
        FindChildrenControl<Text>();
        FindChildrenControl<Toggle>();
        FindChildrenControl<Slider>();
        FindChildrenControl<ScrollRect>();
        FindChildrenControl<InputField>();
    }
```

Awake 中一口气扫描了七种常用 UI 控件。每种调用一次泛型方法 `FindChildrenControl<T>`。

```csharp
private void FindChildrenControl<T>() where T : UIBehaviour
{
    T[] controls = this.GetComponentsInChildren<T>();
    for (int i = 0; i < controls.Length; ++i)
    {
        string objName = controls[i].gameObject.name;
        if (controlDic.ContainsKey(objName))
            controlDic[objName].Add(controls[i]);
        else
            controlDic.Add(objName, new List<UIBehaviour>() { controls[i] });
        //如果是按钮控件
        if (controls[i] is Button)
        {
            (controls[i] as Button).onClick.AddListener(() =>
            {
                OnClick(objName);
            });
        }
        //如果是单选框或者多选框
        else if (controls[i] is Toggle)
        {
            (controls[i] as Toggle).onValueChanged.AddListener((value) =>
            {
                OnValueChanged(objName, value);
            });
        }
    }
}
```

逐段拆解：

1. **`GetComponentsInChildren<T>()`**：递归查找当前 GameObject 及其所有子对象上挂载的 T 类型组件。比如 `FindChildrenControl<Button>()` 会把面板下的所有 Button 组件全部找出来，不管嵌套多深
2. **按名字存入字典**：`objName` 是控件所在 GameObject 的名字（在 Hierarchy 中的名字）。同一个名字可能有多种类型的控件（比如 `"Confirm"` 上同时有 Button 和 Image），所以字典的 value 是 `List<UIBehaviour>`
3. **自动绑定 Button**：检测到 Button 后，用 lambda 表达式给 `onClick` 添加监听——点击时调用 `OnClick(objName)`，子类重写 `OnClick` 就能响应所有按钮
4. **自动绑定 Toggle**：同样用 lambda 给 `onValueChanged` 添加监听，子类重写 `OnValueChanged` 响应所有 Toggle

#### GetControl — 按名字取控件

```csharp
protected T GetControl<T>(string controlName) where T : UIBehaviour
{
    if (controlDic.ContainsKey(controlName))
    {
        for (int i = 0; i < controlDic[controlName].Count; ++i)
        {
            if (controlDic[controlName][i] is T)
                return controlDic[controlName][i] as T;
        }
    }
    return null;
}
```

同一个名字下可能存了多种控件（Button 和 Image 都叫 `"PlayBtn"`），`GetControl<T>` 遍历列表找到类型匹配的那个返回。用的是 `is` 类型判断 + `as` 安全转换。

#### 虚方法 — 子类重写入口

```csharp
public virtual void ShowMe()  { }
public virtual void HideMe()  { }

protected virtual void OnClick(string btnName) { }
protected virtual void OnValueChanged(string toggleName, bool value) { }
```

这四个方法是子类的扩展点。`virtual` 表示"有默认实现（空），子类可重写"。子类只需重写自己关心的部分：

```csharp
public class LoginPanel : BasePanel
{
    protected override void OnClick(string btnName)
    {
        switch (btnName)
        {
            case "LoginBtn":
                Debug.Log("点击了登录按钮");
                break;
            case "RegisterBtn":
                Debug.Log("点击了注册按钮");
                break;
        }
    }

    protected override void OnValueChanged(string toggleName, bool value)
    {
        // "RememberPwd" Toggle 的值变化
        if (toggleName == "RememberPwd")
            rememberPassword = value;
    }
}
```

不需要手动拖控件、不需要 `public Button loginBtn`——只要 Hierarchy 中按钮的名字是 `"LoginBtn"`，`OnClick` 就能收到。

### UIManager — 面板的生命周期管理

#### 构造函数 — 创建 Canvas 和 EventSystem

```csharp
public enum E_UI_Layer
{
    Bot,    // 底层
    Mid,    // 中层（默认）
    Top,    // 顶层
    System, // 系统层
}

public class UIManager : BaseManager<UIManager>
{
    public Dictionary<string, BasePanel> panelDic = new Dictionary<string, BasePanel>();

    private Transform bot;
    private Transform mid;
    private Transform top;
    private Transform system;

    public RectTransform canvas;

    public UIManager()
    {
        //创建Canvas 让其过场景的时候 不被移除
        GameObject obj = ResMgr.GetInstance().Load<GameObject>("UI/Canvas");
        canvas = obj.transform as RectTransform;
        GameObject.DontDestroyOnLoad(obj);

        //找到各层
        bot = canvas.Find("Bot");
        mid = canvas.Find("Mid");
        top = canvas.Find("Top");
        system = canvas.Find("System");

        //创建EventSystem 让其过场景的时候 不被移除
        obj = ResMgr.GetInstance().Load<GameObject>("UI/EventSystem");
        GameObject.DontDestroyOnLoad(obj);
    }
```

构造函数中做了初始化工作：

- **加载 Canvas 预制体**：Canvas 必须是预先做好的预制体，放在 `Resources/UI/Canvas`。注意用的是 `ResMgr.Load<GameObject>` 同步加载——Canvas 是 UI 根节点，必须在 UI 系统初始化时就位，不适合异步
- **四层结构**：Canvas 下面必须有 `Bot`、`Mid`、`Top`、`System` 四个子对象，各自是一个空节点，作为不同层级面板的父节点。渲染顺序从下往上：Bot → Mid → Top → System，System 永远在最前面
- **加载 EventSystem**：UGUI 的事件系统（按钮点击、拖拽等）依赖 EventSystem 组件。如果没有它，所有 UI 交互都会失效。同样同步加载并 DontDestroyOnLoad

层级设计的典型用法：

| 层级   | 用途                             |
| ------ | -------------------------------- |
| Bot    | 背景图、装饰元素                 |
| Mid    | 主界面（背包、技能栏、任务面板） |
| Top    | 弹窗、确认框、提示               |
| System | 网络断开提示、FPS 显示、GM 面板  |

#### ShowPanel — 显示面板

```csharp
public void ShowPanel<T>(string panelName, E_UI_Layer layer = E_UI_Layer.Mid, UnityAction<T> callBack = null) where T : BasePanel
{
    if (panelDic.ContainsKey(panelName))
    {
        panelDic[panelName].ShowMe();
        if (callBack != null)
            callBack(panelDic[panelName] as T);
        //避免面板重复加载 如果存在该面板 即直接显示 调用回调函数后  直接return 不再处理后面的异步加载逻辑
        return;
    }

    ResMgr.GetInstance().LoadAsync<GameObject>("UI/" + panelName, (obj) =>
    {
        Transform father = bot;
        switch (layer)
        {
            case E_UI_Layer.Mid: father = mid; break;
            case E_UI_Layer.Top: father = top; break;
            case E_UI_Layer.System: father = system; break;
        }
        obj.transform.SetParent(father);

        obj.transform.localPosition = Vector3.zero;
        obj.transform.localScale = Vector3.one;

        (obj.transform as RectTransform).offsetMax = Vector2.zero;
        (obj.transform as RectTransform).offsetMin = Vector2.zero;

        T panel = obj.GetComponent<T>();
        if (callBack != null) callBack(panel);
        panel.ShowMe();

        //把面板存起来
        panelDic.Add(panelName, panel);
    });
}
```

逻辑分两条路径：

**路径 A — 面板已存在**：`panelDic` 中已有 → 直接 `ShowMe()` → 执行回调 → return。不会再加载一次。这意味着 HidePanel 后如果再 ShowPanel，会重新走异步加载（因为 Hide 时从字典移除了）。但如果 **没调过 HidePanel**，重复调 ShowPanel 不会重复加载。

**路径 B — 首次加载**：异步加载 `"UI/" + panelName` 预制体 → 设置父节点为对应层级 → 重置位置和缩放 → 设置 RectTransform 铺满父节点 → 拿到 BasePanel 脚本 → 执行回调 → `ShowMe()` → 存入字典。

**RectTransform 四行设置的含义**：

```csharp
obj.transform.localPosition = Vector3.zero;    // 位置归零（锚点对齐后居中）
obj.transform.localScale = Vector3.one;         // 缩放归 1（预制体可能被改过缩放）
(obj.transform as RectTransform).offsetMax = Vector2.zero;  // 右上角贴紧父节点
(obj.transform as RectTransform).offsetMin = Vector2.zero;  // 左下角贴紧父节点
```

`offsetMax` 和 `offsetMin` 都设为零意味着面板的四个边都贴紧父节点，即**铺满整个父节点**。这是全屏面板的标准设置。

#### HidePanel — 隐藏面板

```csharp
public void HidePanel(string panelName)
{
    if (panelDic.ContainsKey(panelName))
    {
        panelDic[panelName].HideMe();
        GameObject.Destroy(panelDic[panelName].gameObject);
        panelDic.Remove(panelName);
    }
}
```

隐藏面板做了三步：调用 `HideMe()` 给子类处理机会 → Destroy 整个 GameObject → 从字典移除。Destroy 后会触发该面板下所有组件的 `OnDestroy`，子类可以在这里做清理（比如取消事件订阅）。

#### GetPanel — 获取已显示的面板

```csharp
public T GetPanel<T>(string name) where T : BasePanel
{
    if (panelDic.ContainsKey(name))
        return panelDic[name] as T;
    return null;
}
```

有时需要在外部获取某个正在显示的面板引用（比如修改面板上的文本），这个方法提供按名字查找。

#### AddCustomEventListener — 自定义事件

```csharp
public static void AddCustomEventListener(UIBehaviour control, EventTriggerType type, UnityAction<BaseEventData> callBack)
{
    EventTrigger trigger = control.GetComponent<EventTrigger>();
    if (trigger == null)
        trigger = control.gameObject.AddComponent<EventTrigger>();

    EventTrigger.Entry entry = new EventTrigger.Entry();
    entry.eventID = type;
    entry.callback.AddListener(callBack);

    trigger.triggers.Add(entry);
}
```

BasePanel 自动绑定了 Button 的 onClick 和 Toggle 的 onValueChanged。但如果需要其他事件（比如**鼠标进入/离开、拖拽、长按**），就要用 `EventTrigger`。这个静态方法封装了 EventTrigger 的创建和绑定流程，一行代码就能给任意控件加自定义事件：

```csharp
// 给按钮加鼠标悬停事件
UIManager.AddCustomEventListener(btn, EventTriggerType.PointerEnter, (data) =>
{
    Debug.Log("鼠标移入按钮");
});
```

### 依赖关系

```
UIManager 构造函数
    └─ ResMgr.GetInstance().Load<GameObject>("UI/Canvas")  ← 同步加载
    └─ ResMgr.GetInstance().Load<GameObject>("UI/EventSystem")

UIManager.ShowPanel
    └─ ResMgr.GetInstance().LoadAsync<GameObject>("UI/" + panelName)  ← 异步加载
```

UI 管理只依赖资源加载模块，不依赖公共 Mono 和事件中心——因为它自己就是最上层的模块，下面没有其他模块需要它来支撑。

### 完整使用示例

**定义一个登录面板：**

```csharp
public class LoginPanel : BasePanel
{
    protected override void Awake()
    {
        base.Awake();  // 必须先调基类 Awake，扫描子控件

        // 初始化控件
        GetControl<Text>("TitleText").text = "欢迎回来";
        GetControl<Image>("Logo").sprite = logoImage;

        // 自定义事件 —— 输入框获得焦点
        UIManager.AddCustomEventListener(
            GetControl<InputField>("UsernameInput"),
            EventTriggerType.Select,
            (data) => Debug.Log("用户名输入框被选中")
        );
    }

    protected override void OnClick(string btnName)
    {
        switch (btnName)
        {
            case "LoginBtn":
                string username = GetControl<InputField>("UsernameInput").text;
                string password = GetControl<InputField>("PasswordInput").text;
                DoLogin(username, password);
                break;
            case "CloseBtn":
                UIManager.GetInstance().HidePanel("LoginPanel");
                break;
        }
    }

    public override void ShowMe()
    {
        // 每次显示时清空输入框
        GetControl<InputField>("UsernameInput").text = "";
        GetControl<InputField>("PasswordInput").text = "";
    }

    void DoLogin(string user, string pwd) { /* ... */ }
}
```

**显示和隐藏：**

```csharp
// 显示登录面板（首次异步加载）
UIManager.GetInstance().ShowPanel<LoginPanel>("LoginPanel", E_UI_Layer.Mid, (panel) =>
{
    Debug.Log("登录面板加载完成");
});

// 隐藏登录面板（会 Destroy，下次再 Show 会重新加载）
UIManager.GetInstance().HidePanel("LoginPanel");

// 获取正在显示的面板引用
LoginPanel login = UIManager.GetInstance().GetPanel<LoginPanel>("LoginPanel");
if (login != null)
    login.DoSomething();
```

### 注意事项

- **子类 Awake 必须调 `base.Awake()`**：BasePanel 的控件扫描在 Awake 中执行。如果子类重写 Awake 忘了调基类，`controlDic` 就是空的，`GetControl` 永远返回 null
- **控件命名不能重复**：BasePanel 按 GameObject 名字存控件。如果两个不同按钮都叫 `"Button"`，字典里会存到同一个 key 下，GetControl 可能返回错误的那一个。每个控件应该有一个独一无二的名字
- **Canvas 预制体必须事先做好**：Canvas 预制体需要包含四层结构（Bot/Mid/Top/System 四个子节点），放在 `Resources/UI/Canvas`。EventSystem 预制体放 `Resources/UI/EventSystem`。这两个预制体的制作是使用此框架的前提
- **HidePanel 会 Destroy**：隐藏即销毁。如果面板打开频繁（比如背包），重复加载浪费性能。可以考虑改为只 SetActive(false) 而不 Remove + Destroy，但框架当前按销毁处理——简单直接
- **ShowPanel 异步加载**：面板第一次显示时是异步加载的，不是立刻出现。加载完成前的短暂延迟中，如果玩家连点两次按钮，第二次调用会因 `panelDic.ContainsKey` 为 false（第一次的回调还没执行到 `panelDic.Add`）而触发第二次异步加载——造成两个相同面板。实际项目中需要加一个"正在加载中"的标记防止重复触发
