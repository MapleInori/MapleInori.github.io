## 1. 冷更新 vs 热更新

在聊代码怎么写之前，先把这两个概念分清楚。

### 冷更新

开发者将测试好的代码发布到应用商店的审核平台，平台方会进行稳定性和性能测试。测试通过后，用户在 AppStore 看到更新信息，点击更新——

关键步骤是：**必须先关闭应用，再进行更新**。

也就是说，用户要经历"看到更新 → 关掉应用 → 等待下载 → 重新打开"这个完整流程。

### 热更新

**广义**：无需关闭应用，不停机状态下修复漏洞、更新资源等，重点在于更新逻辑代码。

**狭义（特指 iOS 场景）**：无需将代码重新打包提交至 AppStore，即可更新客户端的执行代码。也就是不用下载 App 就能自动更新程序。

> **现状**：苹果禁止了 C# 的部分反射操作、禁止 JIT（即时编译，即程序运行时创建并运行新代码）。因此 iOS 不允许真正的"逻辑热更新"，只允许使用 AssetBundle 进行**资源热更新**。
>
> 2017 年苹果更是明确：上线后的项目一旦发现使用逻辑热更新，一样会做下架处理。

### 为什么还是需要热更新

- AppStore 审核周期难以控制（短则几天，长则一两周）。
- 手机应用更新频繁，每次都走审核不现实。
- 大型应用动辄几 GB，全量更新成本太大。
- **终极目标**：不重新下载、不停机状态下完全变换一个应用的内容。

即便在 iOS 限制下，Android 和 PC 平台依然可以做逻辑热更新。下面看看各平台方案。

### 各平台怎么做

| 平台               | 方案                                                                                                                                              |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| Android / PC（C#） | 将执行代码预编译为 AssemblyDLL → 将代码作为 TextAsset 打包进 AssetBundle → 运行时调用 AssemblyDLL 代码 → 更新对应的 AssetBundle 即可实现热更新 |
| iOS（Lua）         | 苹果禁止 JIT → 不能走 C# 逻辑热更新 → 方案：Unity + Lua 插件（xLua / toLua / sLua 等）                                                          |

### 常见 Unity 热更新插件

| 插件      | 特点                                                          |
| --------- | ------------------------------------------------------------- |
| sLua      | 最快的 Lua 插件                                               |
| toLua     | 由 uLua 发展而来，第三代 Lua 热更新方案                       |
| xLua      | 特性最先进的 Lua 插件，腾讯出品                               |
| ILRuntime | 纯 C# 实现的热更新插件（不走 Lua，直接在 C# 层面解释执行 IL） |

本文用的是 **xLua**。

## 2. xLua 是什么

xLua 是腾讯开源的一个 Unity 下的 Lua 编程解决方案。它在 Unity 中提供一个 **Lua 运行环境**（`LuaEnv`），让 C# 可以执行 Lua 代码，也让 Lua 可以调用 C# 的类和方法。

可以把它理解成一座桥：

```text
C# 代码  ←──── LuaEnv（解析器 + 加载器） ────→  Lua 代码
```

有了这座桥之后：

- C# 可以调用 Lua：把 Lua 当成"配置脚本"或"逻辑脚本"来执行。
- Lua 可以调用 C#：在 Lua 里使用 Unity 的 API（比如 `Debug.Log`、`GameObject` 等）。

## 3. 第一步：获得 Lua 解析器并执行代码

xLua 的核心类是 `LuaEnv`。它就是一个 Lua 解析器——创建出来以后，你可以把一段 Lua 代码以**字符串**的形式交给它执行。

```csharp
using System.Collections;
using System.Collections.Generic;
using UnityEngine;
using XLua; // 使用 xLua 的命名空间

public class First : MonoBehaviour
{
    void Start()
    {
        // ── 步骤 1：创建 Lua 解析器 ──
        // Lua 是解释型语言，不能直接运行。
        // 在 Unity 里要先 new 一个 LuaEnv，它就像一个"Lua 虚拟机"。
        LuaEnv env = new LuaEnv();

        // ── 步骤 2：执行 Lua 代码 ──
        // DoString 的意思是把字符串当成 Lua 代码来执行。
        // 这里传入了 Lua 的 print 函数，它会在 Unity 控制台输出 "Hello world!"。
        env.DoString("print('Hello world!')");

        // ── 步骤 3：释放解析器 ──
        // LuaEnv 用完后要释放，否则会一直占用内存。
        env.Dispose();
    }
}
```

**执行流程拆解：**

```text
new LuaEnv()          创建 Lua 虚拟机，准备好执行环境
        ↓
env.DoString("...")   把字符串 "print('Hello world!')" 交给 Lua 虚拟机
        ↓
                      虚拟机解释执行：调用 Lua 的 print 函数
        ↓
                      控制台输出 "Hello world!"
        ↓
env.Dispose()         销毁虚拟机，释放资源
```

> **关键理解**：`DoString` 里的字符串是 **Lua 代码**，不是 C# 代码。虽然它写在 C# 文件里、被双引号包着，但它最终会被 xLua 的虚拟机当成 Lua 语法来解释执行。`print` 是 Lua 的内置函数，不是 C# 的 `Debug.Log`。

## 4. Lua 调用 C# 代码

xLua 里最让人兴奋的一点是：**Lua 可以直接调用 C# 的代码**。做法是在 Lua 字符串里使用 `CS.命名空间.类名.方法名(参数)` 的格式。

```csharp
using System.Collections;
using System.Collections.Generic;
using UnityEngine;
using XLua;

public class DoString : MonoBehaviour
{
    void Start()
    {
        // 两个示例分别看效果，可以只运行其中一个
        // LuaCallCSharpCode();
        LuaReturnData();
    }

    // ═══════════════════════════════════════
    // 示例 A：Lua 调用 C# 代码
    // ═══════════════════════════════════════
    public void LuaCallCSharpCode()
    {
        LuaEnv env = new LuaEnv();

        // Lua 中通过 "CS." 前缀访问 C# 的类和方法。
        // 这行 Lua 代码等价于在 C# 里写：Debug.Log("from lua");
        //
        // 拆开看：
        //   CS                      → xLua 提供的命名空间入口
        //   UnityEngine            → C# 的命名空间
        //   Debug                  → C# 的类名
        //   Log("from lua")        → 调用静态方法 Log，传入字符串参数
        env.DoString("CS.UnityEngine.Debug.Log('from lua')");

        env.Dispose();
    }

    // ═══════════════════════════════════════
    // 示例 B：Lua 返回值给 C#
    // ═══════════════════════════════════════
    public void LuaReturnData()
    {
        LuaEnv env = new LuaEnv();

        // DoString 的返回值是 object[] 数组。
        // Lua 可以一次返回多个值，每个返回值会成为数组的一个元素。
        //
        // Lua 代码 "return 100, true" 的意思是：
        //   返回两个值 —— 数字 100 和 布尔值 true
        object[] data = env.DoString("return 100, true");

        // data[0] 就是第一个返回值（100）
        Debug.Log("Lua的第一个返回值：" + data[0].ToString());

        // data[1] 就是第二个返回值（true）
        Debug.Log("Lua的第二个返回值：" + data[1].ToString());

        env.Dispose();
    }
}
```

**示例 A 的执行流程：**

```text
env.DoString("CS.UnityEngine.Debug.Log('from lua')")
        ↓
xLua 解析字符串中的 "CS.UnityEngine.Debug.Log"
        ↓
映射到 C# 的 UnityEngine.Debug.Log 方法
        ↓
调用 Log，传入字符串 "from lua"
        ↓
Unity 控制台输出：from lua
```

**示例 B 的执行流程（关键——Lua 多返回值 → C# object[]）：**

```text
env.DoString("return 100, true")
        ↓
Lua 虚拟机执行 return 语句，返回两个值
        ↓
xLua 把这两个值打包成一个 object[] 数组
        ↓
object[0] = (int)100
object[1] = (bool)true
        ↓
C# 代码从数组中取出每个值使用
```

> **对比记忆**：
>
> - C# 调用 Lua：用 `DoString` 执行 Lua 代码字符串，如果 Lua 代码有 `return`，返回值在 `object[]` 里。
> - Lua 调用 C#：在 Lua 字符串里写 `CS.命名空间.类名.方法()`，就像在 Lua 里直接操作 C# 一样。

## 5. xLua 的加载器——让 require 找到你的 Lua 文件

前面都是直接把 Lua 代码写成字符串传给 `DoString`。但在真实项目里，Lua 代码会写在独立的 `.lua` 文件里（可能有几十上百个），不可能全部用字符串硬编码。这时候就要用到 **Lua 的 `require` 机制**和 **xLua 的加载器**。

### 5.1 什么是加载器

回顾一下 [热更新开发2](../HotUpdate2) 里讲的：纯 Lua 环境下，`require("模块名")` 会根据 `package.path` 中定义的路径模板去磁盘上找文件。

在 Unity + xLua 的环境里，情况不一样了：

- 游戏打包后，文件系统结构和编辑器里完全不同。
- 资源可能在 `Resources` 里、可能在 `StreamingAssets` 里、可能在 AB 包里。
- xLua 不能简单地用文件路径去查找。

所以 xLua 设计了 **加载器（Loader）** 机制：当 Lua 代码里执行 `require("xxx")` 时，xLua 会依次询问每个已注册的加载器"你能找到这个模块吗？"。谁先返回了文件内容，就用谁的。

```text
Lua: require("test")
        ↓
xLua: 问加载器 1 → "你能找到 test 吗？"
        ↓ 返回 null（找不到）
xLua: 问加载器 2 → "你能找到 test 吗？"
        ↓ 返回 null（找不到）
xLua: 问内置加载器 → "你能找到 test 吗？"
        ↓ 找到了！返回文件内容的字节数组
xLua: 把字节数组当成 Lua 代码执行
```

### 5.2 内置加载器——为什么放在 StreamingAssets 下就能加载

xLua 自带了一个内置加载器，它会去 `StreamingAssets` 目录下查找 `.lua` 文件。

```csharp
using System.Collections;
using System.Collections.Generic;
using System.IO;
using UnityEngine;
using XLua;

public class Loader : MonoBehaviour
{
    void Start()
    {
        MyLoader(); // 先看自定义加载器的效果
        // SystemLoader(); // 也可以换这个看内置加载器
    }

    // ═══════════════════════════════════════
    // 方式一：使用 xLua 内置加载器
    // ═══════════════════════════════════════
    public void SystemLoader()
    {
        LuaEnv env = new LuaEnv();

        // require('test') 会触发 xLua 的内置加载器去查找 test.lua
        // 内置加载器默认会在 StreamingAssets 目录下搜索。
        //
        // 也就是说，如果你把 test.lua 放在
        //   Assets/StreamingAssets/test.lua
        // 那么 require('test') 就能找到它。
        env.DoString("require('test')");
        env.Dispose();
    }
}
```

> **为什么是 `StreamingAssets`？**
>
> `StreamingAssets` 是 Unity 的一个特殊目录。放在里面的文件会被原样复制到最终的游戏包里，不会被 Unity 序列化或修改。对于 Lua 脚本来说这是合适的——它就是一个纯文本文件，不需要 Unity 处理。
>
> 但后面我们会改用 `Resources` 或自定义路径，因为 `StreamingAssets` 在某些平台（比如 Android）上只能通过 `WWW` 或 `UnityWebRequest` 读取，不能直接用 `File.ReadAllBytes`。

### 5.3 自定义加载器——把 Lua 文件放在任何你想要的目录

内置加载器只认固定的目录。实际项目里，Lua 文件的存放位置往往是自定义的（比如项目根目录下的 `DataPath/Lua/`）。这时候就要写自己的加载器，并通过 `env.AddLoader()` 注册进去。

```csharp
public void MyLoader()
{
    LuaEnv env = new LuaEnv();

    // AddLoader：把自定义加载器注册到 xLua 的解析环境中。
    // 自定义加载器会先于系统内置加载器执行。
    // 当自定义加载器成功返回了文件内容（非 null）后，
    // 后续的加载器就不会再被调用。
    env.AddLoader(ProjectLoader);

    // 这时候再 require('test1')，xLua 就会先调用我们写的 ProjectLoader
    env.DoString("require('test1')");

    env.Dispose();
}

// ═══════════════════════════════════════
// 自定义加载器 —— 核心方法
// ═══════════════════════════════════════
//
// 这个方法的签名是固定的：
//   参数：ref string filepath    → Lua 的 require("xxx") 中传入的模块名
//   返回：byte[]                 → Lua 文件的完整内容（字节数组）
//          返回 null             → 表示"我找不到这个文件，让下一个加载器试试"
//
// 每当 Lua 代码执行 require() 时，xLua 就会调用这个方法。
// 注意：不仅是 C# 里写 env.DoString("require('xxx')") 会触发，
//      如果 Lua 脚本内部也写了 require("其他模块")，同样会触发！
//
public byte[] ProjectLoader(ref string filepath)
{
    // 打印一下看看 filepath 到底是什么。
    // 比如执行 require('test1')，这里就会输出 "test1"。
    Debug.Log("加载器收到的模块名：" + filepath);

    // ── 构造 Lua 文件的实际路径 ──
    //
    // Application.dataPath 在编辑器里指向：
    //   项目根目录/Assets
    //
    // 我们的目的：从 dataPath 回到项目根目录，再拼上自定义子目录。
    //
    //   dataPath            = "D:/MyProject/Assets"     （长度 19）
    //
    // "Assets" 是 6 个字母，"/Assets" 是 7 个字符。
    //
    //   减 7 → Substring(0, 12) = "D:/MyProject"        ← 去掉 "/Assets"
    //         再拼 "/DataPath/Lua/test1.lua"
    //         结果："D:/MyProject/DataPath/Lua/test1.lua"  （斜杠自己补上）
    //
    //   减 6 → Substring(0, 13) = "D:/MyProject/"       ← 只去掉了 "Assets"，斜杠保留
    //         可以直接拼 "DataPath/Lua/test1.lua"（不用加斜杠）
    //         但习惯上统一用减 7 + 手动拼斜杠，语义更清楚："我回到了根目录，再从头写子路径"。
    //
    string path = Application.dataPath;
    path = path.Substring(0, path.Length - 7) + "/DataPath/Lua/" + filepath + ".lua";

    // ── 判断文件是否存在 ──
    if (File.Exists(path))
    {
        // 找到了！把整个 .lua 文件读成字节数组返回。
        // xLua 拿到这个字节数组后，会把它当成 Lua 代码来执行。
        //
        // 这里有一个非常重要的扩展思路：
        //   既然加载器返回的是"字节数组"，那这个字节数组从哪来并不重要。
        //   可以是 File.ReadAllBytes（从磁盘读），
        //   也可以是从 Resources 加载 TextAsset 拿到的 bytes，
        //   甚至可以从 AssetBundle 里提取 TextAsset 的 bytes，
        //   还可以从网络下载后解密得到。
        //   加载器只负责"给 Lua 虚拟机提供代码内容"，内容来源可以任意定制。
        return File.ReadAllBytes(path);
    }
    else
    {
        // 找不到就返回 null。
        // xLua 会继续询问下一个加载器（如果有的话）。
        return null;
    }
}
```

**自定义加载器的执行流程图：**

```text
C#: env.DoString("require('test1')")
        ↓
xLua 查找已注册的加载器列表
        ↓
─────────────────────────────────────
加载器 1（自定义，ProjectLoader）：
  收到 filepath = "test1"
  ↓
  拼接路径 → "项目根/DataPath/Lua/test1.lua"
  ↓
  文件存在？
  ├─ 是 → ReadAllBytes → 返回字节数组 → ✅ xLua 执行这段 Lua 代码
  └─ 否 → return null → 继续下一个加载器
─────────────────────────────────────
加载器 2（xLua 内置）：
  在 StreamingAssets 下查找 test1.lua
  ↓
  ...（同上逻辑）
─────────────────────────────────────
```

> **和纯 Lua 的 `package.path` 对比理解：**
>
> |                  | 纯 Lua                      | xLua（Unity 内）                                 |
> | ---------------- | --------------------------- | ------------------------------------------------ |
> | 配置文件查找规则 | `package.path` 字符串模板 | 自定义加载器（C# 委托）                          |
> | 找到后           | 直接读取磁盘文件            | 加载器返回 byte[]                                |
> | 找不到时         | 报错                        | 返回 null，尝试下一个加载器                      |
> | 优势             | 简单直接                    | 可以从任意来源加载：磁盘、Resources、AB 包、网络 |
>
> 核心思想是一样的：**当 Lua 执行 `require("xxx")` 时，有一套机制去"找到"对应的代码内容**。只是 xLua 把这套机制做成了可编程的 C# 回调——这给了我们极大的自由度。

> **关于"能不能把 Lua 代码放入 AB 包"的思考：**
>
> 加载器的签名是 `byte[] Loader(ref string filepath)`，它只要求返回字节内容。所以理论上完全可以这样：
>
> ```text
> 1. 把 .lua 文件以 .txt 或 .bytes 格式打进 AB 包
> 2. 加载器中加载 AB 包，用 LoadAsset<TextAsset> 取出内容
> 3. 返回 textAsset.bytes
> ```
>
> 这样 Lua 脚本就可以和 AB 包里的资源一起被热更新！这是后续文章会讲的内容。

### 5.4 Application.dataPath 到底是什么

很多初学者在这里会迷惑，专门解释一下。

在 Unity **编辑器**中：

```csharp
// 假设项目路径是 D:\MyProject\
Application.dataPath  // → "D:/MyProject/Assets"
```

也就是说 `dataPath` 指向的是项目里的 `Assets` 目录，**不是**项目根目录。

所以如果要访问和 `Assets` 同级的文件夹（比如 `D:/MyProject/DataPath/`），就需要从 `dataPath` 往上退一层，去掉末尾的 `"/Assets"` 这 7 个字符：

```csharp
string projectRoot = Application.dataPath.Substring(0, Application.dataPath.Length - 7);
// 结果： "D:/MyProject"
```

在**打包后的游戏**中，`Application.dataPath` 的指向会变化，不同平台也不同。所以这种"减 7 取项目根目录"的写法**只在编辑器下有效**。正式项目里通常会根据平台动态选择路径，或者用 `Application.streamingAssetsPath`、`Application.persistentDataPath` 等 Unity 提供的路径。

## 6. 单例 LuaEnv——避免重复创建虚拟机

### 6.1 为什么需要单例

前面每个示例都写成了：

```csharp
LuaEnv env = new LuaEnv();  // 创建一个新虚拟机
// ... 执行一些代码
env.Dispose();              // 销毁虚拟机
```

这种方式在**单个脚本**里没问题。但在真实项目中：

- 场景里可能有十几个脚本都需要执行 Lua 代码。
- 如果每个脚本都 `new LuaEnv()`，就会创建十几个 Lua 虚拟机。
- 每个虚拟机都有独立的内存空间和 GC，**它们之间无法共享数据**。

比如脚本 A 在 Lua 里定义了一个全局变量 `playerLevel = 10`，脚本 B 在另一个 `LuaEnv` 里是读不到的——因为它们根本不在同一个 Lua 虚拟机里。

解决方式就是：**整个应用只维护一个 `LuaEnv` 实例**，也就是写成单例。

### 6.2 实现一个 xLuaEnv 单例

注意这个单例**不继承 `MonoBehaviour`**——它就是一个普通的 C# 类。因为它只负责持有 `LuaEnv` 并提供统一的访问入口，不需要挂在场景物体上。

```csharp
using System.Collections;
using System.Collections.Generic;
using System.IO;
using UnityEngine;
using XLua;

/// <summary>
/// xLua 运行环境的单例封装。
/// 全局只有一个 LuaEnv 实例，确保所有 Lua 代码在同一个虚拟机中执行，
/// 这样全局变量、缓存模块等数据才能在不同脚本之间共享。
/// </summary>
public class xLuaEnv
{
    // ═══════════════════════════════════════
    // 第一部分：单例模式
    // ═══════════════════════════════════════

    // 静态字段：保存全局唯一的 xLuaEnv 实例。
    // static 意味着这个字段属于"类本身"，不属于任何一个对象。
    private static xLuaEnv _Instance = null;

    // 公开的静态属性：外部代码通过 xLuaEnv.Instance 来访问单例。
    // 这是一个属性（property），不是普通字段。
    // 它的 get 访问器会在第一次被访问时创建实例。
    public static xLuaEnv Instance
    {
        get
        {
            // 如果还没有创建过实例，就 new 一个。
            // 因为构造函数是 private 的，外部代码不能直接 new xLuaEnv()，
            // 只能通过 Instance 属性来获取。
            if (_Instance == null)
            {
                _Instance = new xLuaEnv();
            }

            // 返回已有的唯一实例。
            return _Instance;
        }
    }

    // ═══════════════════════════════════════
    // 第二部分：创建并持有 LuaEnv
    // ═══════════════════════════════════════

    // 真正的 Lua 虚拟机对象，只有这一个。
    private LuaEnv _Env;

    // 构造函数是 private 的——外部无法直接 new。
    // 创建单例的同时，也创建了 Lua 运行环境，并注册好加载器。
    private xLuaEnv()
    {
        _Env = new LuaEnv();

        // 在构造时就注册好自定义加载器。
        // 之后任何地方 require("xxx") 都会使用这个加载逻辑，
        // 不需要每次调用时再单独 AddLoader。
        _Env.AddLoader(_ProjectLoader);
    }

    // ═══════════════════════════════════════
    // 第三部分：自定义加载器
    // ═══════════════════════════════════════

    // 和前面 Loader 里的 ProjectLoader 逻辑完全一样。
    // 写在单例里，确保整个项目使用统一的 Lua 文件查找规则。
    private byte[] _ProjectLoader(ref string filepath)
    {
        // 从 Assets 目录往上退一层到项目根目录，再拼 DataPath/Lua/
        string path = Application.dataPath;
        path = path.Substring(0, path.Length - 7) + "/DataPath/Lua/" + filepath + ".lua";

        if (File.Exists(path))
        {
            return File.ReadAllBytes(path);
        }
        else
        {
            return null;
        }
    }

    // ═══════════════════════════════════════
    // 第四部分：释放 LuaEnv
    // ═══════════════════════════════════════

    /// <summary>
    /// 释放 Lua 虚拟机并重置单例。
    /// 调用后下次访问 Instance 会重新创建 LuaEnv。
    /// </summary>
    public void Free()
    {
        // Dispose：销毁 Lua 虚拟机，释放所有 Lua 相关的内存和资源。
        _Env.Dispose();

        // 把静态字段置为 null，这样下次访问 Instance 时
        // 会再次触发构造函数，创建一个全新的 LuaEnv。
        _Instance = null;
    }

    // ═══════════════════════════════════════
    // 第五部分：对 Lua 操作的封装
    // ═══════════════════════════════════════

    /// <summary>
    /// 执行一段 Lua 代码字符串，并返回 Lua 的返回值。
    /// 这是对 _Env.DoString 的简单封装，外部不需要直接接触 _Env。
    /// </summary>
    public object[] DoString(string code)
    {
        return _Env.DoString(code);
    }

    // ── 后续可以继续封装更多操作 ──
    // 比如：
    //   public T Get<T>(string path)      → 获取 Lua 全局变量
    //   public void Set<T>(string path, T value) → 设置 Lua 全局变量
    //   public void Require(string name)  → 加载指定的 Lua 模块
}
```

**单例模式的执行流程：**

```text
第一次调用 xLuaEnv.Instance
        ↓
_Instance 是 null 吗？
  ├─ 是 → new xLuaEnv()
  │        ↓
  │      构造函数：new LuaEnv() + AddLoader + 其他初始化
  │        ↓
  │      把新对象赋值给 _Instance
  │        ↓
  └─ 否 → 直接用已有的 _Instance
        ↓
返回 _Instance  ←── 调用方拿到单例
```

> **为什么构造函数是 `private`？**
>
> 如果不加限制，任何代码都可以 `new xLuaEnv()`，那就又回到"多个实例"的问题了。`private` 构造函数的含义是：**只有这个类自己内部的代码（具体来说是 `Instance` 属性的 get 访问器）才能创建实例**。外部代码只能通过 `xLuaEnv.Instance` 来获取那个唯一的实例。

### 6.3 测试单例

有了单例后，调用 Lua 就简单多了——不需要在每个脚本里 `new LuaEnv()`，也不需要手动注册加载器。直接一行代码搞定：

```csharp
using System.Collections;
using System.Collections.Generic;
using UnityEngine;

public class TestSingleton : MonoBehaviour
{
    void Start()
    {
        // 直接通过单例执行 Lua 的 require。
        // 这句话的完整流程：
        //   1. xLuaEnv.Instance → 获取全局唯一的 xLuaEnv 单例
        //       （如果是首次调用，会自动创建 LuaEnv 并注册加载器）
        //   2. .DoString("require('test1')") → 在 Lua 虚拟机里执行 require
        //   3. require 触发 _ProjectLoader → 在 DataPath/Lua 下找 test1.lua
        //   4. 找到文件 → 读取内容 → Lua 虚拟机执行
        //
        // 注意：DoString("require('test1')") 相当于直接在 Lua 环境里执行
        // require 语句。如果 test1.lua 里有全局函数定义或 print 语句，
        // 它们会在 require 时立即执行。
        xLuaEnv.Instance.DoString("require('test1')");
    }

    void OnDestroy()
    {
        // 当前脚本所在的物体被销毁时，释放 Lua 虚拟机。
        // 实际项目中，Free 的时机通常由更上层的"游戏生命周期管理器"来控制，
        // 比如切场景时、游戏退出时。
        xLuaEnv.Instance.Free();
    }
}
```

**执行流程对比——有无单例的区别：**

```text
【没有单例的时候——每个脚本各自为政】
脚本 A:  new LuaEnv() → LuaVM-1  ─┐
脚本 B:  new LuaEnv() → LuaVM-2   ├─ 三个独立的虚拟机，无法共享数据
脚本 C:  new LuaEnv() → LuaVM-3  ─┘

【有单例之后——所有脚本共享同一个虚拟机】
脚本 A:  xLuaEnv.Instance ─┐
脚本 B:  xLuaEnv.Instance  ├─ 全部指向同一个 LuaEnv
脚本 C:  xLuaEnv.Instance ─┘
           ↓
        唯一 LuaVM
```

## 7. C# 调用 Lua vs Lua 调用 C#——方向要分清

这是最容易被搞混的地方。因为"调用方向"决定了代码的主体写在哪里。

### C# 调用 Lua

**意思**：C# 脚本作为"主人"，在自己的代码里使用 xLua 去执行 Lua 脚本。

**写法**：在 `.cs` 文件里写 `env.DoString(...)` 或其他 xLua API。

**典型场景**：

- 把游戏配置写在 Lua 表里，C# 启动时加载。
- 把经常改的业务逻辑（战斗公式、UI 布局计算等）写成 Lua，C# 在合适的时机调用。
- C# 作为框架，Lua 作为"可热更的插件"。

```csharp
// C# 代码
xLuaEnv.Instance.DoString("require('battle_formula')");  // 加载战斗公式模块
```

### Lua 调用 C#

**意思**：Lua 脚本作为"主人"，通过 `CS.` 前缀去调用 Unity 或 C# 提供的能力。

**写法**：在 `.lua` 文件里写 `CS.UnityEngine.Debug.Log(...)` 等。

**典型场景**：

- Lua 脚本里需要访问 GameObject、Transform 等 Unity 对象。
- Lua 需要调用项目自定义的 C# 工具类。
- Lua 逻辑需要操作 UI、播放音效等。

```lua
-- Lua 代码（写在 .lua 文件里）
CS.UnityEngine.Debug.Log("这条日志是从 Lua 发出的")
```

### 一个具体的对比

假设有一个需求：角色升级时，根据等级计算所需经验值。

|              | C# 调用 Lua                                                                                                | Lua 调用 C#                                                                         |
| ------------ | ---------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| 谁主导       | C# 代码                                                                                                    | Lua 代码                                                                            |
| C# 里的样子  | `env.DoString("require('levelup')");` `<br>` `int exp = env.Global.Get<int>("CalculateExp", level);` | `local exp = CS.GameUtils.CalculateExp(level)`                                    |
| Lua 里的样子 | `function CalculateExp(level)` `<br>` `  return level * 100` `<br>` `end`                        | `local exp = CS.GameUtils.CalculateExp(level)` `<br>` `-- 调用 C# 的工具方法` |
| 可热更       | ✅ Lua 脚本可以独立更新                                                                                    | ❌ 核心计算在 C#，要跟着包更新                                                      |
| 适合         | 经常改动的逻辑                                                                                             | 稳定、底层的功能                                                                    |

实际项目里通常是**双向配合**：

- C# 负责底层（渲染、物理、网络、IO）。
- Lua 负责上层业务逻辑。
- C# 通过 xLua 加载并驱动 Lua 脚本。
- Lua 需要底层能力时，通过 `CS.` 前缀回调 C#。

```text
        ┌─────────────────────────┐
        │     C#（底层框架）       │
        │  渲染 / 物理 / 网络      │
        └──────────┬──────────────┘
                   │
            xLuaEnv.DoString("require(...)")
            （C# 调用 Lua，把 Lua 脚本加载进来）
                   │
                   ↓
        ┌─────────────────────────┐
        │   Lua（业务逻辑层）      │
        │  战斗公式 / UI 逻辑      │
        └──────────┬──────────────┘
                   │
            CS.UnityEngine.Debug.Log(...)
            CS.GameManager.DoSomething(...)
            （Lua 调用 C#，使用底层能力）
                   │
                   ↓
        ┌─────────────────────────┐
        │     C#（底层框架）       │
        │  响应 Lua 的调用请求      │
        └─────────────────────────┘
```

所以"方向"不是二选一，而是**谁在哪个层面做什么**。理解了这一点，再看任何 xLua 项目，思路就会清晰很多。

## 8. Lua 调用 C# 实战：11 个场景

前面第 4 节只演示了最简单的 `CS.UnityEngine.Debug.Log`。实际项目中 Lua 需要调用 C# 的各种成员——静态字段、对象属性、重载方法、委托、事件、枚举、泛型、out/ref 参数等等。

下面每个场景都由 **两份文件** 组成：

| 文件     | 位置             | 作用                                                               |
| -------- | ---------------- | ------------------------------------------------------------------ |
| C# 脚本  | `Scripts/C2L/` | 定义要被 Lua 调用的类型（类、枚举、委托等），并触发对应的 Lua 脚本 |
| Lua 脚本 | `Lua/C2L/`     | 真正演示"Lua 怎么调用 C#"的代码                                    |

**阅读方式**：每个场景先看 C# 定义了哪些类型，再看 Lua 怎么用 `CS.` 去调用它们。

> 所有 C# 脚本里的 `MonoBehaviour` 部分几乎一样，都只是 `Start` 里 `require` 对应的 Lua 文件、`OnDestroy` 里 `Free`。下面只展开最有代表性的几个，其余只贴类型定义部分，避免重复。

### 8.1 基础：类的继承与虚方法（LuaCallBase）

Lua 里实例化 C# 对象就像 `CS.类名()`，调用方法用冒号 `:`。

**C# 端——定义父类和子类：**

```csharp
public class Father
{
    public string Name = "father";

    public void Talk()
    {
        Debug.Log("这是父类中的方法");
    }

    public virtual void Overide()
    {
        Debug.Log("这是父类中的虚方法");
    }
}

public class Child : Father
{
    public override void Overide()
    {
        Debug.Log("这是子类中的重写方法");
    }
}
```

**Lua 端——实例化并调用：**

```lua
-- 调用 Father
local father = CS.Father()       -- 等价于 C# 的 new Father()
print(father.Name)               -- 访问字段 → "father"
father:Overide()                 -- 冒号调用 → "这是父类中的虚方法"

-- 调用 Child
local child = CS.Child()
print(child.Name)                -- Child 没有定义 Name，继承了父类的 → "father"
child:Talk()                     -- 继承自父类的方法 → "这是父类中的方法"
child:Overide()                  -- 子类重写的方法 → "这是子类中的重写方法"
```

> **关键理解——冒号 `:` vs 点号 `.`**：
>
> - `father:Overide()` 相当于 `father.Overide(father)`，Lua 自动把 `father` 作为 `self` 传入。
> - C# 对象的实例方法需要一个隐式的 `this` 引用，冒号刚好满足这个需求。
> - 如果写成 `father.Overide()`（点号），很可能会报错，因为 C# 方法收不到 `this`。

### 8.2 静态类：命名空间、字段、属性与默认参数（LuaCallStatic）

**C# 端——定义带命名空间的静态类：**

```csharp
namespace HX
{
    public static class TestStatic
    {
        public static int ID = 99;

        public static string Name { get; set; }

        public static string Output()
        {
            return "static";
        }

        // 带默认参数的静态方法
        public static void Default(string str = "abc")
        {
            Debug.Log(str);
        }
    }
}
```

**Lua 端——访问静态成员：**

```lua
-- 规则：CS.命名空间.类名.成员

-- 读取静态字段
print(CS.HX.TestStatic.ID)        -- 99

-- 给静态属性赋值
CS.HX.TestStatic.Name = "admin"
print(CS.HX.TestStatic.Name)      -- "admin"

-- 调用静态方法
print(CS.HX.TestStatic.Output())  -- "static"

-- 使用默认参数——不传参时走 C# 的默认值
CS.HX.TestStatic.Default()        -- 输出 "abc"（默认值）

-- 传入 Lua 的值覆盖默认值
CS.HX.TestStatic.Default("def")   -- 输出 "def"
```

> **对比记忆**：
>
> - Lua 里直接给 C# 属性赋值：`CS.XXX.Name = "admin"`，和 C# 的 `TestStatic.Name = "admin"` 写起来几乎一样。
> - C# 方法的默认参数在 Lua 里也有效——不传就用默认值，传了就覆盖。

### 8.3 对象实例化：构造函数重载、属性、冒号（LuaCallObject）

**C# 端——定义带重载构造函数的类：**

```csharp
public class Npc
{
    public string Name;
    public int HP { get; set; }   // 自动属性，有 get 和 set

    public Npc() { }              // 无参构造

    public Npc(string name)       // 有参构造
    {
        Name = name;
    }

    public string Output()
    {
        return this.Name;
    }
}
```

**Lua 端——实例化并访问属性/方法：**

```lua
-- 无参构造实例化（和 C# 的 new Npc() 一样）
local obj = CS.Npc()
obj.HP = 100
print(obj.HP)                     -- 100

-- 有参构造实例化
local obj1 = CS.Npc("admin")
print(obj1.Name)                  -- "admin"

-- 冒号调用实例方法（方法内部需要 this）
print(obj1:Output())              -- "admin"
-- 冒号调用 = 自动传入 self。
-- obj1:Output() 等价于 obj1.Output(obj1)
```

> **Lua 里还能直接 `new` Unity 的 GameObject：**
>
> ```lua
> local go = CS.UnityEngine.GameObject("LuaCreateGO")
> -- 等价于 C# 的 new GameObject("LuaCreateGO")
> ```
>
> 这说明 `CS.` 不仅能调我们写的类，Unity 内置的类也一样能调。

### 8.4 枚举：访问值与类型转换（LuaCallEnum）

**C# 端——定义一个普通枚举：**

```csharp
public enum TestEnum
{
    LoL = 0,
    Dota2
}
```

**Lua 端——访问和转换枚举：**

```lua
-- C# TestEnum.LoL 的对应写法
-- 规则：CS.命名空间.枚举名.枚举值

-- 注意：从 C# 获取到的枚举，在 Lua 里类型是 "userdata"（非 Lua 原生类型）
print(type(CS.TestEnum.LoL))      -- "userdata"
print(CS.TestEnum.Dota2)          -- 输出 "Lua:Dota2:1"（Lua前缀+枚举名:整数值）

-- 通过整数值转回枚举
print(CS.TestEnum.__CastFrom(0))        -- 得到 CS.TestEnum.LoL

-- 通过字符串名转回枚举
print(CS.TestEnum.__CastFrom("Dota2"))  -- 得到 CS.TestEnum.Dota2
```

> **关键点**：Lua 没有枚举这个类型。C# 的枚举传到 Lua 后会被 xLua 包装成 `userdata`。如果拿到一个整数值想转回枚举，用 `__CastFrom`。

### 8.5 委托：把 Lua 函数赋给 C# 委托（LuaCallDelegate）

这是最实用的场景之一——C# 定义委托类型，Lua 把自己的函数挂上去，C# 在合适的时机回调 Lua。

**C# 端——定义委托类型和持有者：**

```csharp
public delegate void DelegateLua();     // 定义一个无参无返回的委托类型

public class TestDelegate
{
    public static DelegateLua Static;   // 静态委托字段
    public DelegateLua Dynamic;         // 实例委托字段

    public static void StaticFunc()
    {
        Debug.Log("C#静态成员函数");
    }
}
```

**Lua 端——给 C# 委托赋值、加减、调用：**

```lua
-- ═══ 静态委托：把 C# 方法赋给委托 ═══
CS.TestDelegate.Static = CS.TestDelegate.StaticFunc
CS.TestDelegate.Static()                 -- 调用 → "C#静态成员函数"

-- Lua 中如果添加了函数到静态委托变量，
-- 在不再使用后，记得释放：赋为 nil
CS.TestDelegate.Static = nil

-- ═══ 静态委托：把 Lua 函数赋给委托 ═══
local func = function()
    print("这是Lua的函数")
end

-- 覆盖赋值
CS.TestDelegate.Static = func

-- 加法赋值（+=）——注意：必须先有值才能加
CS.TestDelegate.Static = CS.TestDelegate.Static + func

-- 减法赋值（-=）——移除之前添加的函数
CS.TestDelegate.Static = CS.TestDelegate.Static - func

CS.TestDelegate.Static()                 -- 调用委托
CS.TestDelegate.Static = nil             -- 用完清空

-- ═══ 安全调用：判断委托是否为空 ═══
-- if CS.TestDelegate.Static ~= nil then
--     CS.TestDelegate.Static()
-- end

-- ═══ 实例委托 ═══
local obj = CS.TestDelegate()            -- new TestDelegate()
obj.Dynamic = func                       -- 给实例委托字段赋值
obj.Dynamic()                            -- 调用实例委托
obj.Dynamic = nil                        -- 用完清空
```

> **委托的加减操作注意**：
>
> - xLua 不支持直接写 `+=`、`-=`，要写成 `= Static + func` 和 `= Static - func`。
> - 做减法之前，委托上必须已经有回调函数，否则可能报错。
> - Lua 函数挂在 C# 委托上之后，如果不再使用，务必设为 `nil`，否则 Lua 函数对象不会被 GC。

### 8.6 事件：添加/移除 C# 事件回调（LuaCallEvent）

C# 的 `event` 在外部只能做 `+=` 和 `-=`，不能直接赋值。xLua 里用字符串 `"+"` 和 `"-"` 来表达。

**C# 端——定义事件：**

```csharp
public delegate void EventLua();

public class TestEvent
{
    public static event EventLua Static;         // 静态事件

    public static void StaticFunc()
    {
        Debug.Log("这是静态函数");
    }

    public static void CallStatic()
    {
        if (Static != null) { Static(); }        // 触发静态事件
    }

    public event EventLua Dynamic;               // 实例事件

    public void CallDynamic()
    {
        if (Dynamic != null) { Dynamic(); }      // 触发实例事件
    }
}
```

**Lua 端——用字符串 "+"/"-" 添加/移除事件：**

```lua
-- ═══ 静态事件 ═══
-- Lua 添加事件回调：用 ("+", 回调函数)
CS.TestEvent.Static("+", CS.TestEvent.StaticFunc)
CS.TestEvent.CallStatic()                       -- 触发 → "这是静态函数"
CS.TestEvent.Static("-", CS.TestEvent.StaticFunc) -- 移除回调

-- ═══ 实例事件 ═══
local func = function()
    print("来自于Lua的回调函数")
end

local obj = CS.TestEvent()                      -- new TestEvent()
obj:Dynamic("+", func)                           -- 添加 Lua 函数到实例事件
obj:CallDynamic()                                -- 触发 → "来自于Lua的回调函数"
obj:Dynamic("-", func)                           -- 移除 Lua 函数
```

> **委托 vs 事件在 xLua 里的写法对比**：
>
> |      | 委托（delegate）      | 事件（event）        |
> | ---- | --------------------- | -------------------- |
> | 赋值 | `Delegate = func`   | 不允许直接赋值       |
> | 添加 | `= Delegate + func` | `Event("+", func)` |
> | 移除 | `= Delegate - func` | `Event("-", func)` |
> | 调用 | `Delegate()`        | 不能从外部直接触发   |
>
> 事件用字符串 `"+"` / `"-"` 是因为 C# 的事件只暴露 `add` / `remove` 访问器，xLua 用字符串来指明操作。

### 8.7 重载：同名不同参的方法调用（LuaCallOverload）

C# 里重载是在编译时根据参数类型和数量决定调用哪个版本。Lua 是动态类型，但 xLua 能根据传入的实参自动匹配。

**C# 端——三个同名重载：**

```csharp
public class TestOverload
{
    public static void Test(int id)
    {
        Debug.Log("数字类型: " + id);
    }

    public static void Test(string name)
    {
        Debug.Log("字符串类型: " + name);
    }

    public static void Test(int id, string name)
    {
        Debug.Log("两个数值: " + id + ", " + name);
    }
}
```

**Lua 端——传什么类型就匹配哪个重载：**

```lua
-- 传入整数 → 匹配 Test(int)
CS.TestOverload.Test(99)                    -- "数字类型: 99"

-- 传入字符串 → 匹配 Test(string)
CS.TestOverload.Test("admin")               -- "字符串类型: admin"

-- 传入两个参数 → 匹配 Test(int, string)
CS.TestOverload.Test(100, "root")           -- "两个数值: 100, root"
```

> xLua 自动根据 Lua 实参的**个数和类型**去匹配 C# 的重载版本。绝大多数情况下它猜得对。如果出现歧义（比如两个重载的差别太微妙），可能需要用 xLua 提供的强制类型转换来辅助。

### 8.8 out / ref 参数：多返回值处理（LuaCallOutRef）

C# 的 `out` 和 `ref` 本质是让方法"多返回一些东西"。Lua 本身支持多返回值，xLua 就把它们统一映射为 **Lua 的多返回值**。

**C# 端——对比普通返回 / out / ref：**

```csharp
public class TestOutRef
{
    // 普通返回
    public static string Func1()
    {
        return "Func1";
    }

    // out 参数——第二个参数标记为 out
    public static string Func2(string str1, out string str2)
    {
        str2 = "Func2 out";
        return "Func2";
    }

    // ref 参数——第二个参数标记为 ref
    public static string Func3(string str1, ref string str2)
    {
        str2 = "Func3 Ref";
        return "Func3";
    }

    // ref 作为第一个参数
    public static string Func4(ref string str1, string str2)
    {
        str1 = "Func4 Ref";
        return "Func4";
    }
}
```

**Lua 端——out/ref 的结果变成额外的返回值：**

```lua
-- 普通返回：和 C# 一样
local r1 = CS.TestOutRef.Func1()
print(r1)  -- "Func1"

-- ═══ out 参数 ═══
-- C# 的 out 变量，会作为 Lua 的第二个返回值
local out2                            -- 声明一个变量占位（实际不需要初始化）
local r2, out1 = CS.TestOutRef.Func2("admin", out2)
print(r2, out1, out2)                 -- "Func2"  "Func2 out"  nil
--  ↑ C# 的 return    ↑ C# 中 out 出来的值   ↑ 原 Lua 变量，未被修改

-- ═══ ref 参数 ═══
-- C# 的 ref 变量，同样作为第二个返回值
local ref2
local r3, ref1 = CS.TestOutRef.Func3("root", ref2)
print(r3, ref1, ref2)                 -- "Func3"  "Func3 Ref"  nil

-- ═══ ref 作为第一个参数 ═══
-- 即使 out/ref 在参数列表的第一个位置，
-- 返回值仍然遵从 "先放 C# return，再放 out/ref 结果" 的格式
local ref4
local r4, ref3 = CS.TestOutRef.Func4(ref4, "test")
print(r4, ref3, ref4)                 -- "Func4"  "Func4 Ref"  nil
```

> **out / ref 在 xLua 中的映射规则**：
>
> ```text
> C# 方法返回值           → Lua 第 1 个返回值（和普通方法一样）
> 方法的 out/ref 参数结果 → 按参数声明顺序，依次成为 Lua 第 2、3、4... 个返回值
> ```
>
> **传入的 Lua 变量本身不会被修改**——和 C# 不同，xLua 不会回写你传入的那个 `out2` 变量，而是把结果**额外返回**给你，参数里的那个out/ref变量仅作占位。

### 8.9 扩展方法：需要加 [LuaCallCSharp] 标记（LuaCallExtend）

C# 的扩展方法本质上是一个静态方法，xLua 默认不会把它当成目标类型的"成员方法"暴露给 Lua。需要手动标记。

**C# 端——定义类 + 扩展方法：**

```csharp
public class TestExtend
{
    public void Output()
    {
        Debug.Log("类本身带的方法");
    }
}

// 扩展方法所在类必须加 [LuaCallCSharp]，否则 Lua 看不到 Show()
[LuaCallCSharp]
public static class MyExtend
{
    public static void Show(this TestExtend obj)
    {
        Debug.Log("类扩展实现的方法");
    }
}
```

**Lua 端——扩展方法和本类方法一样用冒号调用：**

```lua
local obj = CS.TestExtend()

obj:Output()    -- 类本身的方法
obj:Show()      -- 扩展方法（加了 [LuaCallCSharp] 后 Lua 才能看到）
```

> 如果不加 `[LuaCallCSharp]`，`obj:Show()` 会报类似"找不到 Show 方法"的错误。这是因为 xLua 为减少生成的代码量，默认不会把所有扩展方法都映射到 Lua 侧。

### 8.10 泛型方法与 typeof（LuaCallGenericType）

C# 的泛型方法需要指定类型参数（`<T>`），Lua 不支持这种语法。xLua 的应对思路是：**包装一个非泛型版本来调用泛型方法**，或者使用 `typeof`。

**C# 端——定义泛型方法 + 包装方法：**

```csharp
public class TestGenericType
{
    // 泛型方法——Lua 不能直接调用 TestGenericType.Output<T>(data)
    public void Output<T>(T data)
    {
        Debug.Log("泛型方法：" + data.ToString());
    }

    // 包装方法 1：指定 T = float
    public void Output(float data)
    {
        Output<float>(data);
    }

    // 包装方法 2：指定 T = string
    public void Output(string data)
    {
        Output<string>(data);
    }
}
```

**Lua 端——调用包装方法和使用 typeof：**

```lua
local obj = CS.TestGenericType()

-- 调用非泛型包装（和普通方法一样）
obj:Output(99)              -- 传入数字 → 匹配 Output(float) 包装
obj:Output("admin")         -- 传入字符串 → 匹配 Output(string) 包装

-- ═══ typeof 关键字 ═══
-- xLua 实现了 typeof 关键字，可以替代 Unity 的泛型方法
local go = CS.UnityEngine.GameObject("LuaCreate")

-- Unity 的 AddComponent<T>() 是泛型方法，Lua 里不写 <T>
-- 而是用 typeof(类型) 传给非泛型的 AddComponent(Type) 重载
go:AddComponent(typeof(CS.UnityEngine.BoxCollider))
```

> **`typeof` 和 C# 里的 `typeof` 是一一对应的**：
>
> ```csharp
> // C#
> go.AddComponent<BoxCollider>();
> go.AddComponent(typeof(BoxCollider));
> ```
>
> ```lua
> -- Lua（只支持第二种写法）
> go:AddComponent(typeof(CS.UnityEngine.BoxCollider))
> ```

### 8.11 结构体：用法和类基本一致（LuaCallStruct）

**C# 端——定义一个简单结构体：**

```csharp
public struct TestStruct
{
    public string Name;

    public string Output()
    {
        return Name;
    }
}
```

**Lua 端——和对象的调用方式保持一致：**

```lua
local obj = CS.TestStruct()       -- 实例化结构体

obj.Name = "admin"                 -- 字段赋值
print(obj.Name)                    -- 读取字段 → "admin"

print(obj:Output())                -- 冒号调用方法 → "admin"
```

> 在 Lua 里，调用 C# 的结构体和调用类几乎没有区别。但对 xLua 来说，结构体是**值类型**——这意味着结构体在传递过程中可能被复制。如果反复从同一个 C# 位置获取结构体，每次拿到的可能是不同副本，对副本的修改不会写回原处。这点在数据频繁读写时要注意。
