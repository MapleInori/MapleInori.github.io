### C# 特性和 Unity 序列化

C# 的特性（Attribute）可以理解成贴在类、字段、方法等代码元素上的“说明标签”。它本身不一定直接执行逻辑，但运行时或编辑器可以读取这些标签，然后改变显示、序列化、调用等行为。

比如 Unity 里常见的写法是这样：

```csharp
// SerializeField：让 private 字段也能被 Unity 序列化，并显示在 Inspector 里。
[SerializeField]
private int hp = 100;
```

一个字段也可以同时挂多个特性。它们从上往下贴在字段前面，Unity 会一起读取：

```csharp
// Header：在字段上方加一个分组标题。
[Header("角色基础信息")]

// SerializeField：private 字段默认不显示，加上它之后就能在 Inspector 里配置。
[SerializeField]

// Tooltip：鼠标悬停在字段名上时显示提示文字。
[Tooltip("角色初始生命值")]

// Range：给 int/float 数值加一个范围滑动条。
[Range(1, 999)]
private int hp = 100;
```

这类东西是为了让 Inspector 更好用。以后配置多了之后，一个清楚的面板能少掉很多迷惑时间。

如果一个普通类没有标记 `[System.Serializable]`，Unity 通常不会把它当成可展开的数据结构显示在 Inspector 里。比如角色配置可以这样写：

```csharp
using System;
using UnityEngine;

// System.Serializable：让普通类或结构体可以被 Unity 序列化。
// 这样它作为字段出现时，才能在 Inspector 里展开编辑。
[Serializable]
public class RoleConfig
{
    public int id;
    public string roleName;
    public Color color;
}

public class RoleConfigTest : MonoBehaviour
{
    // SerializeField：让 private 的配置对象显示到 Inspector 中。
    [SerializeField]
    private RoleConfig config;
}
```

挂到物体上之后，`config` 就能在 Inspector 里展开编辑。这个思路和 Unity 内置的 `JsonUtility` 也有点像：数据类需要能被序列化，Unity 才知道怎么存、怎么读、怎么展示。

Header：给下面一组字段加标题，Inspector 看起来更清楚。

```csharp
    [Header("基础信息")]
    public int id = 99;
```

SerializeField：private 字段也能被 Inspector 配置。

```csharp
    [SerializeField]
    private string roleName;
```

Range：限制数值范围，并在 Inspector 里显示成滑动条。

```csharp
    [Range(0, 120)]
    public int age;
```

Tooltip：鼠标悬停时显示说明，适合解释字段用途。

```csharp
    [Tooltip("用于区分阵营或状态的颜色")]
    public Color flag;
```

Space：在当前字段上方留一点空隙，避免面板太挤。

```csharp
    [Space(10)]
```

Multiline：让 string 变成多行输入框，行数相对简单。

```csharp
    [Multiline]
    public string note;
```

TextArea：也是多行文本，但可以设置最小行数和最大行数。超过最大行数后会出现滚动条。

```csharp
    [TextArea(2, 5)]
    public string description;

```

HideInInspector：字段仍然是 public，但不会显示在 Inspector 里。适合代码要公开访问、又不希望被面板手动改掉的值。

```csharp
    [HideInInspector]
    public int runtimeOnlyValue;
```

这里要注意一个小坑：`public` 字段默认会被 Unity 序列化，所以 Inspector 上改过之后，脚本里的默认值可能会被 Inspector 保存的数据覆盖。想让字段在代码里公开、但不被面板干扰，可以考虑 `[HideInInspector]`，或者更干脆地用属性、方法来控制访问。

### Unity 特殊目录

Unity 里有些目录名字不是普通文件夹，它们会被引擎特殊处理。

| 目录                | 作用                                                                     |
| ------------------- | ------------------------------------------------------------------------ |
| `Plugins`         | 放需要被 Unity 特殊识别的插件或跨语言调用代码，比如手机 SDK 接入         |
| `Resources`       | 放可以通过 `Resources.Load` 在运行时加载的资源，会跟随游戏包一起打进去 |
| `StreamingAssets` | 放原样复制进包体的资源，运行时通常按文件路径读取，不同平台路径规则不一样 |
| `Editor`          | 编辑器专用代码目录，只在 Unity 编辑器里使用，不会进入最终游戏包          |

`Resources` 的路径有个很容易糊的点：代码里加载时不写 `Resources`，也不写扩展名。

```csharp
GameObject prefab = Resources.Load<GameObject>("Prefabs/Player");
```

对应的真实路径可能是：

```text
Assets/Resources/Prefabs/Player.prefab
```

如果项目里有多个 `Resources` 文件夹，Unity 会把它们都当成运行时资源目录来看。路径上像是被合并到了一起，所以最好别在不同 `Resources` 目录里放同路径同名资源，不然维护起来会很难判断实际加载的是谁。

`Editor` 目录也类似，只要目录名叫 `Editor`，里面的脚本就属于编辑器代码。编辑器脚本可以用 `UnityEditor` 命名空间，但这类代码不能进入游戏运行逻辑，否则打包时容易失败。

### UnityEditor 要隔离

如果脚本里需要用到 `UnityEditor`，最好放在 `Editor` 文件夹里。实在要和运行时代码写在同一个文件，也要用宏定义隔开：

```csharp
using UnityEngine;

#if UNITY_EDITOR
using UnityEditor;
#endif

public class EditorOnlyExample : MonoBehaviour
{
#if UNITY_EDITOR
    [ContextMenu("Editor Log")]
    private void EditorLog()
    {
        Debug.Log("这段只会在 Unity 编辑器中编译");
    }
#endif
}
```

简单记就是：游戏运行时代码归运行时，编辑器工具归编辑器。

### ContextMenu 和 ContextMenuItem

`[ContextMenu]` 可以给脚本方法加一个右键菜单，适合写一些临时测试、数据检查、编辑器内的小工具。比如想在 Inspector 里右键脚本，直接算一下攻击和防御比例，就可以这样写：

```csharp
using UnityEngine;

public class RoleDebugExample : MonoBehaviour
{
    public float attack = 120f;
    public float defense = 80f;

    // ContextMenu：给这个方法添加组件右键菜单。
    // 在 Inspector 中右键脚本组件，或者点组件右上角菜单，就能看到这个选项。
    [ContextMenu("输出攻击防御比例")]
    private void LogAttackDefenseRatio()
    {
        Debug.Log($"攻击/防御 = {attack / defense}");
    }
}
```

`[ContextMenuItem]` 则是给某个字段添加右键菜单。它有两个参数：第一个是菜单里显示的文字，第二个是回调方法名。

```csharp
using UnityEngine;

public class CountryExample : MonoBehaviour
{
    // ContextMenuItem：给 country 这个字段添加右键菜单。
    // 第一个参数是菜单文字，第二个参数是回调方法名。
    [ContextMenuItem("输出国家", "OutCountry")]
    [Tooltip("右键字段名看看")]
    public string country = "雪国";

    private void OutCountry()
    {
        Debug.Log(country);
    }
}
```

这种写法很适合调试单个字段，比如检查配置值、快速重置字段、输出当前状态之类的。

### RequireComponent

`[RequireComponent]` 用来声明脚本依赖的组件。当这个脚本挂到 GameObject 上时，Unity 会自动把依赖组件也补上。如果依赖关系还在，Unity 也不允许直接删除那个被依赖的组件。

```csharp
using UnityEngine;

// RequireComponent：声明这个脚本依赖 Rigidbody。
// 挂上 RoleMove 之后，如果物体没有 Rigidbody，Unity 会自动添加。
[RequireComponent(typeof(Rigidbody))]
public class RoleMove : MonoBehaviour
{
    private Rigidbody rb;

    private void Awake()
    {
        rb = GetComponent<Rigidbody>();
    }
}
```

这东西的好处是减少“脚本写得好好的，结果组件忘挂了”的低级错误。像移动脚本依赖 `Rigidbody`、UI 脚本依赖 `CanvasGroup`，都可以考虑用它。

### ExecuteInEditMode

`[ExecuteInEditMode]` 可以让脚本在没有点击 Play 的编辑器状态下也执行部分生命周期函数。比如在编辑器里改一个数，场景视图就立刻刷新效果。

```csharp
using UnityEngine;

// ExecuteInEditMode：让脚本在编辑器非运行状态下也能执行。
// 新版本 Unity 里也常见 ExecuteAlways，语义更明确一点。
[ExecuteInEditMode]
public class EditModePreview : MonoBehaviour
{
    public float height = 2f;

    private void Update()
    {
        transform.position = new Vector3(
            transform.position.x,
            height,
            transform.position.z
        );
    }
}
```

它会在编辑器里执行代码，所以不要在里面随便写生成大量对象、改资源、保存数据这种操作。

这里要注意，`[ExecuteInEditMode]` 下的 `Update()` 在编辑器非运行状态里**不是稳定每帧执行**。它更像是在编辑器认为对象需要更新时执行，比如：

- 场景发生变化，比如移动物体、修改 Transform。
- Inspector 里的字段被改动。
- 脚本重新编译或组件刷新。
- Scene 视图或 Inspector 需要重绘。
- 编辑器判断这个对象需要更新。

进入 Play 模式后，它就和普通 `MonoBehaviour` 一样，`Update()` 会每帧执行。

```text
Play 模式：Update 每帧执行。
编辑器模式 + ExecuteInEditMode：Update 在编辑器刷新或对象变化时执行，不保证每帧。
```

所以它适合做一些编辑器预览、位置同步、简单刷新，不适合拿来做需要稳定逐帧推进的逻辑。如果真的想在编辑器里持续执行一段更新逻辑，一般会考虑 `EditorApplication.update` 这类编辑器回调。

### AddComponentMenu

`[AddComponentMenu]` 可以控制脚本在 **Add Component** 列表里的显示路径。第一个参数是菜单路径，第二个参数是排序值。

```csharp
using UnityEngine;

// AddComponentMenu：把脚本放到 Add Component 的指定分类里。
// "Maple/HotUpdate/Asset Bundle Loader" 会显示成多级菜单。
// 10 是排序值，数值越小越靠前。
[AddComponentMenu("Maple/HotUpdate/Asset Bundle Loader", 10)]
public class AssetBundleLoaderExample : MonoBehaviour
{
}
```

### 多选枚举和 Flags

普通枚举可以理解成“单选”：一个数字代表一个选项。

```csharp
public enum RoleType
{
    Warrior = 0,
    Mage = 1,
    Archer = 2
}
```

如果想让枚举支持“多选”，常见做法是给枚举加 `[Flags]`，并且每个选项都用 2 的幂次方。这样每个选项都会占用二进制里的一个独立位置，可以通过位运算组合起来。

```csharp
using System;
using UnityEngine;

// Flags：表示这个枚举可以按位组合。
[Flags]
public enum ElementMask
{
    None = 0,
    Fire = 1 << 0,  // 0001
    Water = 1 << 1, // 0010
    Wind = 1 << 2,  // 0100
    Earth = 1 << 3, // 1000
}

public class ElementExample : MonoBehaviour
{
    public ElementMask elements = ElementMask.Fire | ElementMask.Water;

    private void Start()
    {
        // HasFlag：判断当前组合里是否包含某个选项。
        if (elements.HasFlag(ElementMask.Fire))
        {
            Debug.Log("包含火属性");
        }
    }
}
```

简单记就是：单选枚举可以当成普通数字理解；多选枚举要按二进制位理解。`Fire | Water` 的意思不是“变成第三个选项”，而是“同时打开 Fire 和 Water 这两个开关”。

### 自定义 Inspector：外挂式编辑器开发

这里说的“外挂式开发”，不是外挂程序那个外挂，而是给原来的运行时脚本额外写一个编辑器脚本。原脚本继续负责游戏逻辑，编辑器脚本只负责 Inspector 怎么显示、怎么编辑、怎么加按钮。

比如有一个普通的 `Player` 组件，里面既有基础字段，也有对象引用、枚举和复杂数据：

```csharp
using System;
using System.Collections.Generic;
using UnityEngine;

public enum PlayerProfession
{
    Warrior,
    Mage,
    Archer
}

[Flags]
public enum PlayerLoveColor
{
    None = 0,
    Red = 1 << 0,
    Blue = 1 << 1,
    Green = 1 << 2,
    Yellow = 1 << 3
}

[Serializable]
public class ItemInfo
{
    public int id;
    public string itemName;
    public int count;
}

public class Player : MonoBehaviour
{
    public int id = 1001;
    public string playerName = "Maple";
    public int hp = 100;

    public GameObject weapon;
    public Texture cloth;

    public PlayerProfession profession;
    public PlayerLoveColor loveColor;

    public List<ItemInfo> items = new List<ItemInfo>();

    public void ResetHp()
    {
        hp = 100;
    }
}
```

`Player` 本身不需要引用 `UnityEditor`，也不需要继承什么编辑器类。因为它是运行时代码，最终打包进游戏时也可能会用到。真正的自定义 Inspector 要单独写一个编辑器脚本，通常放在 `Editor` 文件夹下：

```text
Assets/Scripts/Player.cs
Assets/Scripts/Editor/PlayerEditor.cs
```

### 自定义 Inspector 的基本骨架

`PlayerEditor.cs` 的基础结构是这样：

```csharp
using UnityEditor;
using UnityEngine;

// CustomEditor：把这个编辑器脚本绑定到 Player 组件上。
// 以后选中带有 Player 组件的物体时，Inspector 会使用这个类来绘制 Player 面板。
[CustomEditor(typeof(Player))]
public class PlayerEditor : Editor
{
    private Player player;

    private void OnEnable()
    {
        // target：Editor 基类提供的对象，表示当前正在被这个 Inspector 编辑的目标。
        // 因为 CustomEditor 绑定的是 Player，所以这里可以转成 Player。
        player = target as Player;
    }

    public override void OnInspectorGUI()
    {
        EditorGUILayout.LabelField("玩家调试面板");
    }
}
```

流程拆开看就是：

- 引入 `UnityEditor` 命名空间。
- 自定义面板类继承 `Editor`。
- 用 `[CustomEditor(typeof(Player))]` 说明这个面板是给 `Player` 用的。
- 在 `OnEnable` 里通过 `target` 拿到当前正在编辑的 `Player` 对象。
- 重写 `OnInspectorGUI`，自己决定 Inspector 里画哪些字段、按钮和提示文本。

这里的 `OnEnable`、`OnDisable` 是 `Editor` 脚本里的生命周期，不是 `MonoBehaviour` 的那套游戏运行生命周期。

- `OnEnable`：这个自定义 Inspector 开始显示时调用。
- `OnDisable`：这个自定义 Inspector 不再显示或被销毁时调用。

常见触发场景有：选中带有 `Player` 组件的物体、给物体添加 `Player` 组件、取消选中、移除组件、脚本重新编译、Inspector 刷新。所以 `OnEnable` 很适合做“拿引用、找 `SerializedProperty`、初始化编辑器状态”这类事情。

### 直接绘制基础字段

`EditorGUILayout.LabelField` 只是画文字，适合写标题、说明、状态信息。`EditorGUILayout.IntField`、`TextField`、`FloatField` 这类控件才是输入框，而且它们都有返回值：

```csharp
public override void OnInspectorGUI()
{
    EditorGUILayout.LabelField("基础数据");

    // IntField / TextField 会返回修改后的值。
    // 不把返回值赋回字段，Inspector 上的修改就不会保存。
    player.id = EditorGUILayout.IntField("玩家 ID", player.id);
    player.playerName = EditorGUILayout.TextField("玩家名", player.playerName);
    player.hp = EditorGUILayout.IntField("生命值", player.hp);

    if (GUILayout.Button("重置生命值"))
    {
        player.ResetHp();
    }
}
```

这一句：

```csharp
player.id = EditorGUILayout.IntField("玩家 ID", player.id);
```

可以拆成两步理解：

```csharp
int newId = EditorGUILayout.IntField("玩家 ID", player.id);
player.id = newId;
```

第一个参数是 Inspector 上显示的标签文字，第二个参数是当前值。Unity 把输入框画出来后，会把用户修改后的值作为返回值交回来，所以要重新赋给 `player.id`。

如果只写：

```csharp
EditorGUILayout.IntField("玩家 ID", player.id);
```

那就只是“显示了一个输入框”，没有把新值保存回字段里。看起来像能编辑，实际下一次刷新又会变回原来的值。

### 绘制对象引用：ObjectField

`EditorGUILayout.ObjectField` 用来画对象引用框，也就是 Inspector 里可以拖 GameObject、贴图、材质、Prefab 的那个槽位。

```csharp
public override void OnInspectorGUI()
{
    EditorGUILayout.LabelField("对象引用");

    // ObjectField 参数：
    // 1. 显示标题
    // 2. 原始字段值
    // 3. 允许拖入的对象类型
    // 4. 是否允许拖入场景里的对象
    player.weapon = EditorGUILayout.ObjectField(
        "持有武器",
        player.weapon,
        typeof(GameObject),
        true
    ) as GameObject;

    // Texture 通常是 Project 面板里的资源，不是场景对象，所以 allowSceneObjects 写 false。
    player.cloth = EditorGUILayout.ObjectField(
        "衣服材质贴图",
        player.cloth,
        typeof(Texture),
        false
    ) as Texture;
}
```

最后的 `as GameObject`、`as Texture` 是因为 `ObjectField` 返回的是比较泛的 `UnityEngine.Object`，需要转回字段真正需要的类型。

最后一个参数 `allowSceneObjects` 要稍微注意一下：

- `true`：允许把场景里的对象拖进去，比如 Hierarchy 里的某个 GameObject。
- `false`：只允许资源对象，更适合贴图、材质、Prefab 这类 Project 面板里的资源。

如果字段最终要保存到资源文件或 Prefab 里，通常要谨慎引用场景对象，因为场景对象离开当前场景就不一定存在了。

### 绘制枚举：EnumPopup 和 EnumFlagsField

普通枚举用 `EditorGUILayout.EnumPopup`，它是单选下拉框：

```csharp
public override void OnInspectorGUI()
{
    EditorGUILayout.LabelField("枚举数据");

    // EnumPopup：绘制普通枚举，也就是单选。
    // 返回值是 Enum，所以要强转回具体枚举类型。
    player.profession = (PlayerProfession)EditorGUILayout.EnumPopup(
        "玩家职业",
        player.profession
    );
}
```

如果枚举加了 `[Flags]`，并且每个选项按 `1 << n` 这种 2 的幂来写，就可以用 `EnumFlagsField` 画多选枚举：

```csharp
public override void OnInspectorGUI()
{
    // EnumFlagsField：绘制多选枚举。
    player.loveColor = (PlayerLoveColor)EditorGUILayout.EnumFlagsField(
        "玩家喜欢的颜色",
        player.loveColor
    );
}
```

`EnumPopup` 适合“职业只能选一个”这种情况。`EnumFlagsField` 适合“喜欢的颜色可以同时选多个”这种情况。这里也要接住返回值并赋回原字段。

### 保留默认 Inspector

如果不想完全重写 Inspector，只想在原来的默认面板上面或下面加点东西，可以调用 `DrawDefaultInspector()`：

```csharp
public override void OnInspectorGUI()
{
    EditorGUILayout.LabelField("额外调试工具");

    if (GUILayout.Button("重置生命值"))
    {
        player.ResetHp();
    }

    // DrawDefaultInspector：绘制 Unity 原本会显示的那些字段。
    DrawDefaultInspector();
}
```

这样就不用手动把 `id`、`playerName`、`hp` 每个字段都写一遍。适合刚开始学的时候用：默认字段交给 Unity，自己只补充按钮、提示、调试信息。

### 更推荐的 SerializedProperty 写法

上面直接写 `player.id = ...` 比较直观，适合理解流程。但在正式一点的自定义 Inspector 里，更推荐用 `serializedObject` 和 `SerializedProperty`。

原因是它们能更好地配合 Unity 的序列化系统，包括撤销 Undo、多对象编辑、Prefab 覆写显示等。简单说，就是更像 Unity 原生 Inspector 的工作方式。

```csharp
using UnityEditor;
using UnityEngine;

[CustomEditor(typeof(Player))]
public class PlayerEditor : Editor
{
    private SerializedProperty idProperty;
    private SerializedProperty playerNameProperty;
    private SerializedProperty hpProperty;
    private SerializedProperty itemsProperty;

    private void OnEnable()
    {
        // serializedObject：Editor 当前正在编辑的序列化对象。
        // FindProperty：按字段名找到对应的序列化字段。
        idProperty = serializedObject.FindProperty("id");
        playerNameProperty = serializedObject.FindProperty("playerName");
        hpProperty = serializedObject.FindProperty("hp");
        itemsProperty = serializedObject.FindProperty("items");
    }

    public override void OnInspectorGUI()
    {
        // Update：先从目标对象同步最新数据到 serializedObject。
        serializedObject.Update();

        // PropertyField：按 Unity 序列化系统绘制字段。
        EditorGUILayout.PropertyField(idProperty, new GUIContent("玩家 ID"));
        EditorGUILayout.PropertyField(playerNameProperty, new GUIContent("玩家名"));
        EditorGUILayout.PropertyField(hpProperty, new GUIContent("生命值"));

        // 第三个参数 true 表示 includeChildren。
        // 对数组、List、自定义类这类复杂字段，会把子字段也画出来。
        EditorGUILayout.PropertyField(itemsProperty, new GUIContent("道具信息"), true);

        // ApplyModifiedProperties：把 Inspector 上的修改写回目标对象。
        serializedObject.ApplyModifiedProperties();
    }
}
```

这里最容易忘的是开头和结尾：

```csharp
serializedObject.Update();
serializedObject.ApplyModifiedProperties();
```

可以先记成：画字段前先 `Update`，画完字段后 `Apply`。中间用 `PropertyField` 画出来的内容，Unity 会负责处理字段变化。

`FindProperty("items")` 里的字符串必须和原脚本里的字段名一致，大小写也要一致。字段改名以后，这里也要跟着改，不然就找不到数据。

`PropertyField` 的第三个参数也很重要：

```csharp
EditorGUILayout.PropertyField(itemsProperty, new GUIContent("道具信息"), true);
```

这里的 `true` 表示 `includeChildren`，也就是把子数据一起画出来。对 `List<ItemInfo>`、数组、自定义类这种复杂数据来说，写 `true` 后展开才能看到里面每个元素和元素里的字段。

### 两种写法怎么选

刚开始学可以先用直接写法，因为它最容易看懂：

```csharp
player.hp = EditorGUILayout.IntField("生命值", player.hp);
player.weapon = EditorGUILayout.ObjectField("持有武器", player.weapon, typeof(GameObject), true) as GameObject;
```

等到要做正式工具、要支持 Undo、Prefab、多对象选择，或者要绘制数组、List、自定义类时，就换成 `SerializedProperty`：

```csharp
serializedObject.Update();
EditorGUILayout.PropertyField(itemsProperty, new GUIContent("道具信息"), true);
serializedObject.ApplyModifiedProperties();
```

简单记：

- 只是学语法、做临时调试：直接拿 `target` 转成组件对象。
- 要认真做编辑器工具：优先用 `serializedObject` 和 `SerializedProperty`。
- 想保留原始字段显示：用 `DrawDefaultInspector()`。
- 想完全控制 Inspector 长什么样：重写 `OnInspectorGUI()`，自己一个控件一个控件画。
- 基础类型可以用 `IntField`、`TextField`、`FloatField` 等控件。
- 对象引用用 `ObjectField`。
- 单选枚举用 `EnumPopup`，多选枚举用 `EnumFlagsField`。
- 复杂数据、数组、List、自定义类，优先用 `PropertyField`，必要时把 `includeChildren` 设为 `true`。

### 滑动条和提示框：Slider 与 HelpBox

自定义 Inspector 里也可以做一些简单的数值校验。比如攻击力用滑动条编辑，太高或太低时在面板里直接给提示。

```csharp
public override void OnInspectorGUI()
{
    // Slider：绘制 float 滑动条。
    // 参数依次是：标题、当前值、最小值、最大值。
    player.attack = EditorGUILayout.Slider(
        new GUIContent("玩家攻击力"),
        player.attack,
        0,
        100
    );

    if (player.attack > 80)
    {
        // HelpBox：在 Inspector 中显示提示框。
        // MessageType.Error 通常显示为红色错误提示。
        EditorGUILayout.HelpBox("攻击力太高了", MessageType.Error);
    }

    if (player.attack < 20)
    {
        // MessageType.Warning 通常显示为黄色警告提示。
        EditorGUILayout.HelpBox("攻击力太低了", MessageType.Warning);
    }
}
```

`Slider` 和前面的 `IntField` 一样，也会返回修改后的值，所以要赋回原字段：

```csharp
player.attack = EditorGUILayout.Slider("玩家攻击力", player.attack, 0, 100);
```

如果只是写：

```csharp
EditorGUILayout.Slider("玩家攻击力", player.attack, 0, 100);
```

那就只是画了一个滑动条，没有把新值保存回去。

`HelpBox` 适合做编辑器内的提示和校验，比如：

- 数值太大或太小。
- 必填资源没有拖。
- 配置之间有冲突。
- 当前状态不建议继续打包或生成资源。

常用的消息类型有：

```csharp
MessageType.None
MessageType.Info
MessageType.Warning
MessageType.Error
```

### 按钮和横向布局

`GUILayout.Button` 用来画按钮。它会返回一个 `bool`，表示这一帧按钮有没有被按下。

```csharp
public override void OnInspectorGUI()
{
    // 这样只会显示按钮，但没有处理点击结果。
    GUILayout.Button("某个按钮");

    // 常见写法：用 if 接住按钮返回值。
    if (GUILayout.Button("测试"))
    {
        Debug.Log("测试");
    }
}
```

按钮的重点也是“返回值”。如果你不接住这个返回值，按钮只是显示出来，点了也不会做任何事。

如果想让多个控件排在同一行，可以用 `BeginHorizontal` 和 `EndHorizontal` 包起来：

```csharp
public override void OnInspectorGUI()
{
    // BeginHorizontal / EndHorizontal 之间的控件会横向排列。
    EditorGUILayout.BeginHorizontal();

    if (GUILayout.Button("检查"))
    {
        Debug.Log("检查");
    }

    if (GUILayout.Button("排查"))
    {
        Debug.Log("排查");
    }

    EditorGUILayout.EndHorizontal();
}
```

默认情况下，Inspector 控件通常是一行一个，纵向往下排。加了横向布局后，`检查` 和 `排查` 两个按钮就会在同一行里。

这类布局很适合放一组操作按钮，比如：

```csharp
EditorGUILayout.BeginHorizontal();

if (GUILayout.Button("刷新"))
{
    RefreshData();
}

if (GUILayout.Button("清空"))
{
    ClearData();
}

EditorGUILayout.EndHorizontal();
```

`BeginHorizontal()` 和 `EndHorizontal()` 要成对出现。少写结束语句时，Inspector 布局可能会报错，或者后面的控件排列变得很奇怪。

### 顶部菜单：MenuItem

`[MenuItem]` 可以把一个静态方法挂到 Unity 顶部菜单栏里。点击菜单项时，就会执行这个方法。

```csharp
using UnityEditor;
using UnityEngine;

public static class AssetBundleMenu
{
    // MenuItem：在 Unity 顶部菜单栏添加一个入口。
    // 菜单路径用 / 分层，这里会显示为：工具 -> 导出 AB 包。
    [MenuItem("工具/导出 AB 包")]
    private static void ExportAssetBundle()
    {
        Debug.Log("开始导出 AB 包");
    }
}
```

这个脚本要放在 `Editor` 目录下，因为 `MenuItem` 属于 `UnityEditor` 命名空间，只能在编辑器里用。

菜单路径可以继续分层：

```csharp
[MenuItem("工具/热更新/导出 AB 包")]
private static void ExportAssetBundle()
{
}
```

这样顶部菜单里就是：

```text
工具 -> 热更新 -> 导出 AB 包
```

它很适合放一些编辑器工具入口，比如导出 AB 包、生成配置表、检查资源引用、清理缓存之类的。

### 编辑器弹窗：EditorWindow

如果只是点菜单执行一个方法，用 `[MenuItem]` 就够了。如果想弹出一个自己的编辑器窗口，就要继承 `EditorWindow`。

```csharp
using UnityEditor;
using UnityEngine;

public class AssetBundleWindow : EditorWindow
{
    [MenuItem("工具/热更新/AB 包工具")]
    private static void OpenWindow()
    {
        // GetWindow<T>：打开或获取一个 EditorWindow。
        // 参数1 utility：是否是工具窗口。
        // 参数2 title：窗口标题。
        // 参数3 focus：打开后是否立刻聚焦到这个窗口。
        AssetBundleWindow window = GetWindow<AssetBundleWindow>(
            false,
            "AB 包工具",
            true
        );

        window.Show();
    }

    private void OnGUI()
    {
        EditorGUILayout.LabelField("AB 包导出工具");

        if (GUILayout.Button("导出"))
        {
            Debug.Log("导出 AB 包");
        }
    }
}
```

`GetWindow<T>()` 可以理解成“打开这个类型的编辑器窗口”。如果这个窗口已经存在，Unity 通常会把已有窗口拿出来；如果不存在，就创建一个。

三个参数是这样的：

- `utility`：是否作为工具窗口打开。`true` 更像一个浮动的小工具窗，`false` 更像普通可停靠的编辑器窗口。
- `title`：窗口标题。
- `focus`：打开后是否立刻切换焦点到这个窗口。

`OnGUI()` 就是这个窗口的绘制方法。前面自定义 Inspector 里写控件的方式，在 `EditorWindow` 里也能继续用，比如 `LabelField`、`Button`、`HelpBox`、`BeginHorizontal`。

### EditorWindow 生命周期

`EditorWindow` 也有自己的生命周期和回调。它和 `MonoBehaviour` 不一样，也和 `Editor` 自定义 Inspector 不完全一样。

常见的几个：

```csharp
using UnityEditor;
using UnityEngine;

public class WindowLifecycleExample : EditorWindow
{
    [MenuItem("工具/窗口生命周期测试")]
    private static void Open()
    {
        GetWindow<WindowLifecycleExample>("生命周期测试");
    }

    private void OnEnable()
    {
        Debug.Log("窗口启用");
    }

    private void OnDisable()
    {
        Debug.Log("窗口关闭或被销毁");
    }

    private void Update()
    {
        // 编辑器窗口的 Update。
        // 即使窗口不是当前焦点，也可能继续执行。
    }

    private void OnGUI()
    {
        EditorGUILayout.LabelField("在这里绘制窗口内容");
    }

    private void OnHierarchyChange()
    {
        Debug.Log("Hierarchy 发生变化");
    }

    private void OnProjectChange()
    {
        Debug.Log("Project 资源发生变化");
    }

    private void OnSelectionChange()
    {
        Debug.Log("当前选中对象变化");
    }
}
```

几个点要分清楚：

- `OnEnable`：窗口创建、打开、脚本重载后恢复时执行。
- `OnDisable`：窗口关闭、销毁、脚本重载时执行。
- 只是让窗口失去焦点，一般不会触发 `OnDisable`。
- 已经打开的窗口重新获得焦点，一般也不会重新触发 `OnEnable`。
- `Update` 是编辑器窗口自己的更新回调，窗口不在焦点时也可能继续执行。
- `OnGUI` 用来绘制窗口内容。

如果想监听焦点变化，通常看的是 `OnFocus` 和 `OnLostFocus`，而不是 `OnEnable` / `OnDisable`：

```csharp
private void OnFocus()
{
    Debug.Log("窗口获得焦点");
}

private void OnLostFocus()
{
    Debug.Log("窗口失去焦点");
}
```

另外几个编辑器事件也很常用：

- `OnHierarchyChange`：Hierarchy 面板里的对象发生变化时执行，比如创建、删除、重命名、调整父子关系等。
- `OnProjectChange`：Project 里的资源发生变化时执行，比如创建、删除、移动、重命名资源等。
- `OnSelectionChange`：当前选中的对象发生变化时执行。

`Selection.activeGameObject` 可以拿到当前选中的场景物体：

```csharp
private void OnSelectionChange()
{
    GameObject selected = Selection.activeGameObject;

    if (selected != null)
    {
        Debug.Log("当前选中：" + selected.name);
    }
}
```

注意 `Selection.activeGameObject` 主要是拿当前选中的 GameObject。如果你选中的是 Project 里的贴图、材质、脚本之类资源，它不一定有值。更泛一点可以用 `Selection.activeObject`。

### 逻辑运算和位运算

前面多选枚举用到了 `|` 和 `&`，这里很容易和普通逻辑运算混在一起。

先看逻辑运算：

```csharp
bool hasHp = true;
bool hasMp = false;

if (hasHp && hasMp)
{
    Debug.Log("同时有 HP 和 MP");
}

if (hasHp || hasMp)
{
    Debug.Log("至少有一个条件成立");
}
```

`&&` 是逻辑与，左右两边都为 `true`，结果才是 `true`。
`||` 是逻辑或，左右两边只要有一个为 `true`，结果就是 `true`。

它们处理的是 `bool` 条件，常用在 `if` 判断里。

再看位运算：

```csharp
[System.Flags]
public enum PlayerLoveColor
{
    None = 0,
    Red = 1 << 0,   // 0001
    Blue = 1 << 1,  // 0010
    Green = 1 << 2, // 0100
}
```

多选枚举用的不是 `||` 这种逻辑或，而是**位或** `|`：

```csharp
PlayerLoveColor color = PlayerLoveColor.Red | PlayerLoveColor.Blue;
```

它的意思是把两个二进制开关合并起来：

```text
Red   = 0001
Blue  = 0010
合并  = 0011
```

所以 `Red | Blue` 的结果不是“二选一”，而是“Red 和 Blue 都选中”。

检查某个选项是否存在，常用**位与** `&`：

```csharp
bool hasRed = (color & PlayerLoveColor.Red) != 0;

if (hasRed)
{
    Debug.Log("包含红色");
}
```

位与可以理解成“拿某一位出来检查”：

```text
color = Red | Blue = 0011
Red                 = 0001
0011 & 0001         = 0001  // 不是 0，说明包含 Red
```

也可以用 `HasFlag`，写起来更直观：

```csharp
if (color.HasFlag(PlayerLoveColor.Red))
{
    Debug.Log("包含红色");
}
```

简单记：

- `&&`：逻辑与，用于 `bool` 条件。
- `||`：逻辑或，用于 `bool` 条件。
- `&`：位与，用来检查某个二进制位。
- `|`：位或，用来合并多个二进制位。
- Flags 多选枚举用的是位运算，组合选项用 `|`，检查选项用 `&` 或 `HasFlag`。

### Resources 和 AssetBundle 的区别

`Resources` 和 `AssetBundle` 都能做运行时资源加载，但它们的定位不一样。

放在 `Resources` 下的资源，会被 Unity 打进游戏主体包里，一起发送给用户。代码里可以用 `Resources.Load` 加载：

```csharp
Sprite icon = Resources.Load<Sprite>("Icons/Icon");
```

真实路径可能是：

```text
Assets/Resources/Icons/Icon.png
```

代码里不写 `Resources`，也不写扩展名。

`Resources` 的优点是简单，适合教程、小项目、临时测试。缺点也很明显：资源会跟着主体包走。尤其在手机平台上，如果想更新 `Resources` 里的资源，通常就得重新发包，没法像热更新资源那样单独下载一份新资源覆盖掉。

`AssetBundle` 简称 AB 包，可以理解成独立于游戏主包存在的资源文件。它可以单独构建、单独下载、单独加载。使用 AB 包里的资源时，流程通常是：

```text
获得 AB 包文件
-> 通过文件路径加载 AB 包
-> 通过资源名加载 AB 包内部资源
-> 使用资源
```

对比一下：

| 项目     | Resources                  | AssetBundle                      |
| -------- | -------------------------- | -------------------------------- |
| 存储位置 | 主体包内                   | 可以独立于主体包                 |
| 加载方式 | `Resources.Load`         | 先加载 AB 包，再 `LoadAsset`   |
| 更新方式 | 通常要重新发包             | 可以下载新 AB 包替换旧 AB 包     |
| 管理成本 | 简单                       | 更复杂，需要处理构建、依赖、版本 |
| 适合场景 | 小项目、固定资源、临时测试 | 热更新、DLC、资源分包、按需下载  |

用了 AB 包管理的资源，通常就不要再放在 `Resources` 里。否则它既会被打进主包，又会被打进 AB 包，相当于多存了一份，包体也会变大。资源归谁管理要分清楚，不然后面会越来越乱。

### AssetBundle 里能放什么

AB 包存的是 Unity 资源序列化后的数据，比如：

- Prefab
- Texture / Sprite
- Material
- AudioClip
- ScriptableObject 资源
- AnimatorController
- 场景等

但 AB 包里不能直接包含 C# 脚本文件作为可执行逻辑。也就是说，AB 包可以包含“挂了某个脚本的 Prefab”，但运行时能不能正常使用这个脚本，取决于主工程里是否已经有对应的脚本类型。

简单例子：

```text
Player.prefab 挂了 PlayerController.cs
Player.prefab 被打进 AB 包
```

运行时加载这个 Prefab 时，项目里必须已经有 `PlayerController` 这个脚本类。AB 包不会把新的 C# 逻辑代码带进来执行。真正的代码热更新一般要用 ILRuntime、HybridCLR、Lua 等另外的方案，AB 包主要负责资源热更新。

### 构建 AB 包

AB 包需要先在编辑器里构建出来。常用 API 是：

```csharp
BuildPipeline.BuildAssetBundles(...)
```

这个 API 属于 `UnityEditor`，所以导出工具要放在 `Editor` 目录下。

一个很简化的导出菜单可以这样写：

```csharp
using System.IO;
using UnityEditor;
using UnityEngine;

public static class AssetBundleBuilder
{
    [MenuItem("工具/热更新/导出 AB 包")]
    private static void BuildAssetBundles()
    {
        // Application.dataPath 指向项目里的 Assets 目录。
        string outputPath = Path.Combine(Application.dataPath, "../AssetBundles");

        if (!Directory.Exists(outputPath))
        {
            Directory.CreateDirectory(outputPath);
        }

        BuildPipeline.BuildAssetBundles(
            outputPath,
            BuildAssetBundleOptions.None,
            BuildTarget.StandaloneWindows
        );

        Debug.Log("AB 包导出完成：" + outputPath);
    }
}
```

`Application.dataPath` 在编辑器里通常指向：

```text
项目根目录/Assets
```

所以：

```csharp
Path.Combine(Application.dataPath, "../AssetBundles")
```

大概就是把 AB 包输出到项目根目录下的 `AssetBundles` 文件夹。

这里的 `BuildTarget` 要和实际平台对应。比如 Windows 用 `StandaloneWindows`，Android 用 `Android`，iOS 用 `iOS`。不同平台的 AB 包一般不能混用，目标平台要打对。

AB 包配置修改后，或者 AB 包内部资源修改后，都需要重新构建 AB 包。比如你把 `Icon.png` 换成了新图，但没有重新打包，那么旧 AB 包里还是旧图。运行时加载旧 AB 包，自然也只能看到旧资源。

### 加载 AB 包里的资源

运行时加载 AB 包，一般先拿到 AB 包文件路径。这个文件可能来自：

- 安装包内的 `StreamingAssets`。
- 首次启动时从 `StreamingAssets` 拷贝到可写目录。
- 从服务器下载到 `Application.persistentDataPath`。
- 本地缓存目录。

拿到文件路径后，可以用 `AssetBundle.LoadFromFile` 加载 AB 包：

```csharp
using UnityEngine;

public class AssetBundleLoadExample : MonoBehaviour
{
    private void Start()
    {
        string path = Application.streamingAssetsPath + "/iconbundle";

        // LoadFromFile：通过文件路径加载 AB 包文件。
        AssetBundle bundle = AssetBundle.LoadFromFile(path);

        if (bundle == null)
        {
            Debug.LogError("AB 包加载失败：" + path);
            return;
        }

        // LoadAsset<T>：从 AB 包里按名称加载资源，并转成指定类型。
        Sprite icon = bundle.LoadAsset<Sprite>("Icon");

        if (icon == null)
        {
            Debug.LogError("AB 包中没有找到 Icon");
            return;
        }

        Debug.Log("Icon 加载成功：" + icon.name);
    }
}
```

这里分两层：

```csharp
AssetBundle bundle = AssetBundle.LoadFromFile(path);
```

这是加载 AB 包文件本身。

```csharp
Sprite icon = bundle.LoadAsset<Sprite>("Icon");
```

这是从 AB 包内部加载名为 `Icon` 的资源，并希望它是 `Sprite` 类型。

如果资源类型写错，或者资源名不对，就可能加载不到。比如 AB 包里实际是 `Texture2D`，你却用 `LoadAsset<Sprite>`，就容易拿不到想要的结果。

### AB 包和原资源的关系

AB 包打出来之后，运行时加载的是 AB 包文件，不是再回头找 `Assets` 目录里的原资源。

比如：

```text
Icon.png -> 打进 iconbundle -> 删除 Icon.png
```

只要已经构建出来的 `iconbundle` 文件还在，里面也确实包含了 `Icon`，运行时仍然可以从这个旧 AB 包里加载出资源。

也就是说：

```text
运行时加载旧 AB 包：不需要原资源。
重新构建 AB 包：需要原资源。
编辑器继续维护项目：最好别删原资源。
```

删除原资源后要注意几个后果：

- 旧 AB 包还能加载旧资源。
- 之后重新构建 AB 包时，源资源没了，新 AB 包里就不会再有它。
- Unity 编辑器里的场景、Prefab、Inspector 引用可能会丢，因为编辑器里引用的是源资源。
- 后续想修改、重新打包、查依赖会很麻烦。

所以正式项目里通常不会因为 AB 包已经打出来，就把源资源删掉。AB 包是发布/运行时资源，源资源是项目维护用的，两者角色不一样。

### 运行过程中替换资源

运行过程中想换资源，本质上不是“修改已经加载的 AB 包内部文件”，而是准备一份新的 AB 包，然后让程序加载新的那份。

大概流程是：

```text
服务器上放新 AB 包
-> 客户端检查版本
-> 下载新 AB 包到可写目录
-> 卸载旧资源或旧 AB 包
-> 加载新 AB 包
-> 从新 AB 包里加载资源并替换显示
```

简单示意：

```csharp
private AssetBundle currentBundle;

private void LoadIconBundle(string path)
{
    if (currentBundle != null)
    {
        // false 表示卸载 AB 包对象本身，但不强制卸载已经实例化/引用中的资源。
        currentBundle.Unload(false);
    }

    currentBundle = AssetBundle.LoadFromFile(path);
    Sprite icon = currentBundle.LoadAsset<Sprite>("Icon");

    // 后面把 icon 设置到 UI Image 或其他显示对象上。
}
```

真正项目里还要处理版本号、依赖包、下载失败、缓存清理、资源引用释放等问题。

### AB 包依赖关系

AB 包之间可能存在依赖关系。比如：

```text
A 包：Player.prefab
B 包：Player.mat、Player.png
```

如果 `Player.prefab` 引用了 `Player.mat`，而 `Player.mat` 又引用了 `Player.png`，并且这些资源被打进了 B 包，那么 A 包就依赖 B 包。

这里最容易误会的一点是：**加载 A 包本身，并不等于 Unity 会自动把 B 包也加载好。** 依赖关系会记录在构建出来的 Manifest 里，但运行时通常要自己根据 Manifest 查依赖、加载依赖。

可以这样理解：

- 只加载 A 包：A 包文件本身可以加载。
- 加载 A 包中不依赖 B 的资源：通常可以正常用。
- 加载 A 包中依赖 B 的资源：如果 B 没加载，可能出现材质、贴图、Prefab 引用丢失，或者资源表现不完整。
- 正确流程：先加载依赖包 B，再加载 A 包里的目标资源。

教程里的完整加载流程大概是：

```text
加载主 AB 包
-> 获取主 AB 包里的 AssetBundleManifest
-> 查询目标 AB 包依赖哪些 AB 包
-> 先加载依赖 AB 包
-> 再加载目标资源所在 AB 包
-> 最后加载目标资源
```

这里的“主 AB 包”一般指构建 AB 包时 Unity 额外生成的那个总 Manifest 包。它的名字通常和输出目录同名，里面有一个 `AssetBundleManifest` 对象，可以用来查询所有 AB 包和它们的依赖关系。

```csharp
AssetBundle mainBundle = AssetBundle.LoadFromFile(mainBundlePath);
AssetBundleManifest manifest = mainBundle.LoadAsset<AssetBundleManifest>("AssetBundleManifest");

string[] dependencies = manifest.GetAllDependencies("player");

foreach (string dependency in dependencies)
{
    string dependencyPath = rootPath + "/" + dependency;
    AssetBundle.LoadFromFile(dependencyPath);
}

AssetBundle playerBundle = AssetBundle.LoadFromFile(rootPath + "/player");
GameObject playerPrefab = playerBundle.LoadAsset<GameObject>("Player");
```

这个例子只是演示流程，真正项目里不能每次都无脑 `LoadFromFile`，否则很容易重复加载同一个 AB 包。一般会写一个 AB 包管理器，用字典缓存已经加载过的包。

### 同名 AB 包不能重复加载

Unity 不允许同一个 AB 包被重复加载成多个实例。如果某个 AB 包已经加载了，再加载同名或同一份 Bundle 文件，可能会报错，或者加载失败。

常见错误大概类似：

```text
The AssetBundle 'xxx' can't be loaded because another AssetBundle with the same files is already loaded.
```

所以 AB 包加载通常要做缓存：

```csharp
private readonly Dictionary<string, AssetBundle> loadedBundles = new Dictionary<string, AssetBundle>();

public AssetBundle LoadBundle(string bundleName, string path)
{
    if (loadedBundles.TryGetValue(bundleName, out AssetBundle bundle))
    {
        return bundle;
    }

    bundle = AssetBundle.LoadFromFile(path);

    if (bundle != null)
    {
        loadedBundles.Add(bundleName, bundle);
    }

    return bundle;
}
```

这样同一个包只加载一次。之后要用它时，从字典里取，不再重复 `LoadFromFile`。

### 异步加载资源

同步加载会卡住当前流程，资源大一点时可能导致画面停顿。Unity 里很多资源加载都有异步版本，常配合协程使用。

`Resources` 的异步加载是：

```csharp
private IEnumerator LoadIconFromResources()
{
    ResourceRequest request = Resources.LoadAsync<Sprite>("Icons/Icon");

    // yield return 会等到 request 加载完成后再继续往下执行。
    yield return request;

    Sprite icon = request.asset as Sprite;
    Debug.Log(icon.name);
}
```

AB 包也有类似流程。通常可以先异步加载 AB 包文件，再异步加载包里的资源：

```csharp
private IEnumerator LoadIconFromAssetBundle(string path)
{
    AssetBundleCreateRequest bundleRequest = AssetBundle.LoadFromFileAsync(path);
    yield return bundleRequest;

    AssetBundle bundle = bundleRequest.assetBundle;

    if (bundle == null)
    {
        Debug.LogError("AB 包加载失败：" + path);
        yield break;
    }

    AssetBundleRequest assetRequest = bundle.LoadAssetAsync<Sprite>("Icon");
    yield return assetRequest;

    Sprite icon = assetRequest.asset as Sprite;
    Debug.Log("Icon 加载完成：" + icon.name);
}
```

这里的节奏和 `Resources.LoadAsync` 很像：

```text
发起异步请求
-> yield return 等待完成
-> 从 request 里取结果
-> 使用资源
```

### AB 包卸载和内存

加载 AB 包和加载 AB 包里的资源，不是完全一回事。

```text
AssetBundle.LoadFromFile(path)
```

主要是把 AB 包文件加载成一个 `AssetBundle` 对象，让 Unity 能从里面继续读取资源。这个阶段可能会占用一些文件数据、压缩数据、索引信息等内存，但真正明显涨内存的，通常是后面把图片、Prefab、音频等资源加载出来的时候。

```text
bundle.LoadAsset<Sprite>("Icon")
```

这一步才会把具体资源对象加载到内存里。比如一张图片被解压成纹理后，内存占用就会上去。

卸载 AB 包时常见有两种：

```csharp
bundle.Unload(false);
bundle.Unload(true);
```

区别很关键：

| 写法              | 含义                                                                 |
| ----------------- | -------------------------------------------------------------------- |
| `Unload(false)` | 卸载 AB 包文件相关数据，但保留已经从包里加载出来、并且仍被引用的资源 |
| `Unload(true)`  | 卸载 AB 包，同时销毁从这个包里加载出来的资源对象                     |

比如：

```csharp
Sprite icon = bundle.LoadAsset<Sprite>("Icon");
bundle.Unload(false);
```

这时 `bundle` 本身被卸载了，但 `icon` 这个已经加载出来的资源还可以继续用。

如果写：

```csharp
Sprite icon = bundle.LoadAsset<Sprite>("Icon");
bundle.Unload(true);
```

那 `icon` 也会被一起卸掉。如果场景里有对象正在用它，就可能出现资源丢失、显示异常之类的问题。

所以可以先这样记：

```text
Unload(false)：卸包，留资源。
Unload(true)：卸包，也卸资源。
```

### Unload(false) 的内存坑

`Unload(false)` 很方便，因为它能让已经加载出来的资源继续使用。但如果不管理资源引用，也很容易让内存越堆越高。

比如你反复做这个流程：

```text
加载 AB 包
-> LoadAsset<Sprite>("Icon")
-> bundle.Unload(false)
-> 再次加载 AB 包
-> 再次 LoadAsset<Sprite>("Icon")
```

虽然旧 AB 包被 `Unload(false)` 了，但旧的 `Sprite` 如果还被变量、UI、场景对象引用着，就不会立刻消失。下一次重新加载 AB 包并加载 `Icon` 时，可能会得到一份新的 `Sprite` 对象。旧的还在，新的一份又来了，内存就会上涨。

这也是为什么 AB 包管理器通常不只管理“包”，还要管理资源引用、卸载时机、引用计数等东西。

### Destroy 和 Resources.UnloadUnusedAssets

`Destroy` 只是销毁场景里的对象或组件，不等于立刻释放所有资源内存。

比如一个 UI Image 正在显示 AB 包里加载出来的 Sprite：

```text
Image 对象 -> 引用 Sprite -> Sprite 引用 Texture
```

你 `Destroy(image.gameObject)` 后，这个 UI 对象没了，画面上也不显示了。但资源内存什么时候真正回收，还要看有没有别的引用，以及 Unity 什么时候做资源清理。

`Resources.UnloadUnusedAssets()` 可以让 Unity 尝试卸载当前没有被使用的资源。它不只针对 `Resources` 文件夹，也会影响运行时加载出来但已经没有引用的资源，包括 AB 包加载出来的资源。

```csharp
Destroy(iconGameObject);
yield return Resources.UnloadUnusedAssets();
```

它的大致逻辑是：Unity 会遍历场景对象、脚本字段等引用关系，找出已经没人引用的资源，然后把这些未使用资源卸掉。

注意几个点：

- 如果资源还被场景对象、静态变量、列表、字典、脚本字段引用着，它就不是“未使用”。
- `Resources.UnloadUnusedAssets()` 不是 C# 垃圾回收本身，它处理的是 Unity 资源对象。
- 它可能比较耗时，不适合每帧调用。
- 常见时机是切场景、关闭大界面、释放一批 AB 资源之后调用。

如果还想清 C# 层普通对象，通常还会看到：

```csharp
System.GC.Collect();
```

不过这个也不能乱用。正常项目里更重要的是把引用关系理清楚，别让旧资源一直被字典、列表、静态字段抓着不放。

### AB 包管理器要做什么

后面如果要写 AB 包管理器，核心目标大概是这些：

- 用字典记录已经加载过的 AB 包，避免同名包重复加载。
- 加载目标包之前，先根据主 Manifest 加载依赖包。
- 支持同步加载和异步加载资源。
- 支持按包名卸载 AB 包。
- 区分 `Unload(false)` 和 `Unload(true)`。
- 需要更完善时，加引用计数，避免某个依赖包还被别人用着就被卸载。

大概结构会像这样：

```csharp
public class AssetBundleManager
{
    private readonly Dictionary<string, AssetBundle> loadedBundles = new Dictionary<string, AssetBundle>();
    private AssetBundleManifest manifest;

    public void LoadBundle(string bundleName)
    {
        if (loadedBundles.ContainsKey(bundleName))
        {
            return;
        }

        string[] dependencies = manifest.GetAllDependencies(bundleName);

        foreach (string dependency in dependencies)
        {
            LoadBundle(dependency);
        }

        AssetBundle bundle = AssetBundle.LoadFromFile(GetBundlePath(bundleName));
        loadedBundles.Add(bundleName, bundle);
    }

    public T LoadAsset<T>(string bundleName, string assetName) where T : UnityEngine.Object
    {
        LoadBundle(bundleName);
        return loadedBundles[bundleName].LoadAsset<T>(assetName);
    }

    public void UnloadBundle(string bundleName, bool unloadAllLoadedObjects)
    {
        if (!loadedBundles.TryGetValue(bundleName, out AssetBundle bundle))
        {
            return;
        }

        bundle.Unload(unloadAllLoadedObjects);
        loadedBundles.Remove(bundleName);
    }

    private string GetBundlePath(string bundleName)
    {
        return Application.streamingAssetsPath + "/" + bundleName;
    }
}
```

这只是雏形，还没处理引用计数、异步、下载、版本、错误重试。教程阶段先把“防重复加载、处理依赖、统一卸载”这几个核心点抓住就行。
