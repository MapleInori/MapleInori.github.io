### 主要窗口

Scene、Game、Hierarchy、Project、Inspector、Console，这六个窗口基本上就是Unity的门面了。

- **Scene**：场景编辑视图，在这里摆放物体、调整位置。
- **Game**：游戏运行时的摄像机视角，点击播放键之后看到的就是这里。
- **Hierarchy**：场景层级面板，列出了当前场景里所有的游戏物体，父子关系也在这里体现。
- **Project**：项目资源面板，所有的素材、脚本、预制体文件都在这里管理。
- **Inspector**：检查器，选中物体之后在这里查看和修改它的各种属性和组件。
- **Console**：控制台，报错、警告、`Debug.Log` 的输出都打在这里。

多用几次就都熟了，没什么别的捷径。

### Inspector 的 Normal 模式和 Debug 模式

Inspector 右上角有个小菜单，默认是 **Normal** 模式，切换成 **Debug** 模式之后会显示更多隐藏字段，包括私有变量也都露出来了。调试的时候挺有用的，可以直接看到脚本里那些没有 `[SerializeField]` 但又想瞄一眼的字段。

### Transform的父子关系

每个游戏物体默认都有 Transform 组件，记录着它的位置（Position）、旋转（Rotation）、缩放（Scale）。

父子级关系这里要注意一下：Inspector 里显示的是**局部坐标（Local）**，也就是相对于父物体的坐标。如果物体没有父级，那局部坐标和世界坐标是一样的。但一旦挂了父物体，子物体的 Position 就是相对于父物体原点的偏移量了。坐标和角度是加算，缩放是乘算的，父物体 Scale 是 (2,2,2)，子物体 Scale 是 (1,1,1)，子物体实际大小其实是 2 倍。有时候搞不清楚物体为什么跑偏了或者变大了，多半是父子级 Transform 没算清楚。

### GameObject 和 Component

Unity 的核心思想：**游戏物体（GameObject）** 本身什么都不是，它只是一个容器，功能全靠挂载的**组件（Component）** 来实现。

比如一个 Cube，它之所以能显示出来是因为有 `Mesh Renderer` 和 `Mesh Filter`，之所以有物理碰撞是因为有 `Collider`，之所以受重力是因为有 `Rigidbody`。你自己写的脚本挂上去之后也是一个 Component。这种组合式的设计挺好用的，想要什么功能加什么组件，不想要了删掉就行。

### 预制体 Prefab

Prefab 就是把一个配置好的 GameObject 保存成一个资源文件，之后可以反复实例化。改一处 Prefab，所有从它生成的实例都会跟着变，批量管理很方便。

几个操作要搞清楚：

- **Overwrite（Apply）**：把实例上的修改同步回 Prefab 本体，所有实例都会受影响。
- **Revert**：把实例上的修改撤销掉，恢复成 Prefab 的样子。
- **修改 Prefab 本体**：双击 Prefab 文件进入编辑模式，改完之后所有实例都更新。
- **删除 Prefab 文件**：场景里已经实例化的物体不会消失，但它们会变成"普通 GameObject"，和 Prefab 的关联断掉，Inspector 里原来蓝色的字会变成正常颜色。

导出 Prefab 的时候，记得勾选相关依赖（纹理、材质之类的），不然拿到别的项目里是一个粉色的缺材质物体，很丑。脚本如果有依赖其他脚本也要一并带上。

### Start 和 Update

写 Unity 脚本绕不开这两个方法：

```csharp
void Start()
{
    // 游戏开始时执行一次，做初始化用
}

void Update()
{
    // 每帧执行一次，游戏逻辑写这里
}
```

`Start` 只跑一次，适合做初始化，比如获取组件引用、设置初始值。`Update` 每帧都跑，帧率多高它就跑多快，所以涉及时间相关的逻辑要乘以 `Time.deltaTime`，不然在高帧率机器上物体会跑得飞快。

### Debug 类

调试的时候最常用的几个：

```csharp
Debug.Log("普通日志，白色");
Debug.LogWarning("警告，黄色");
Debug.LogError("报错，红色");
Debug.DrawLine(Vector3.zero, Vector3.up, Color.red); // 在 Scene 视图里画线，可视化调试用
Debug.DrawRay(transform.position, transform.forward * 5, Color.green); // 画射线
```

`DrawLine` 和 `DrawRay` 只在 Scene 视图里可见，Game 视图看不到，专门用来可视化一些方向、范围之类的东西，比打 Log 直观多了。

### Rotate 方法

`transform.Rotate()` 用来让物体旋转，有两种常见用法：

```csharp
// 以自身坐标系旋转（默认）
transform.Rotate(0, 90, 0);                          // 等价于下面这行
transform.Rotate(0, 90, 0, Space.Self);

// 以世界坐标系旋转
transform.Rotate(0, 90, 0, Space.World);
```

`Space.Self` 是沿着物体自己的轴旋转，`Space.World` 是沿着世界坐标的轴旋转。物体没有倾斜的时候两者看起来差不多，一旦物体本身被旋转过，区别就很明显了。填入的Vector3参数是一个增量，不是将rotation改成目标值，而是加目标值。

### 万向节死锁（Gimbal Lock）

这个是欧拉角的经典问题。当物体绕 X 轴旋转 90 度之后，Y 轴和 Z 轴会重合，这时候无论怎么转 Y 或者转 Z 效果都一样，少了一个自由度，这就是万向节死锁。

Unity 的 Transform 里显示的旋转是欧拉角，底层其实用的是四元数（Quaternion）来避免这个问题。所以在代码里做复杂旋转运算，最好直接用 `Quaternion`，别直接改 `eulerAngles`，不然容易踩坑。

### 类名和脚本名一致

Unity 里脚本文件名和里面的类名必须保持一致，不然挂不上去，Unity 找不到对应的类。起名字的时候直接想好再建，改起来麻烦，文件和类都要改。某些版本的Unity会解决这个问题，但是最好还是类名和文件名一致。

### RotateAround

```csharp
// 让物体绕某个点、某个轴旋转指定角度
transform.RotateAround(Vector3 point, Vector3 axis, float angle);

// 例如绕世界原点、Y轴旋转，模拟公转效果
transform.RotateAround(Vector3.zero, Vector3.up, 30 * Time.deltaTime);
```

`RotateAround` 是让物体**围绕某个点**转圈，同时自身的位置也会跟着变。用来做卫星绕行星转、摄像机绕目标旋转之类的效果很方便。

### RotateAround 和 Rotate 的区别

两个方法都可以传轴和角度，但效果完全不同：

|          | `Rotate(axis, angle)` | `RotateAround(point, axis, angle)` |
| -------- | ----------------------- | ------------------------------------ |
| 旋转中心 | 物体自身原点            | 指定的外部点                         |
| 位置变化 | 不变                    | 会跟着改变                           |
| 用途     | 自转                    | 公转                                 |

`Rotate` 只是自身原地转，`RotateAround` 会同时改变 `position` 和 `rotation`，因为物体要绕着外部的点移动。

### Vector2 和 Vector3 的方向值

几个常用的快捷方向向量，不用每次都手写坐标：

**Vector3：**

| 属性                | 值         |
| ------------------- | ---------- |
| `Vector3.up`      | (0, 1, 0)  |
| `Vector3.down`    | (0, -1, 0) |
| `Vector3.left`    | (-1, 0, 0) |
| `Vector3.right`   | (1, 0, 0)  |
| `Vector3.forward` | (0, 0, 1)  |
| `Vector3.back`    | (0, 0, -1) |
| `Vector3.zero`    | (0, 0, 0)  |
| `Vector3.one`     | (1, 1, 1)  |

**Vector2：**

| 属性              | 值      |
| ----------------- | ------- |
| `Vector2.up`    | (0, 1)  |
| `Vector2.down`  | (0, -1) |
| `Vector2.left`  | (-1, 0) |
| `Vector2.right` | (1, 0)  |
| `Vector2.zero`  | (0, 0)  |
| `Vector2.one`   | (1, 1)  |

`Vector3.forward` 对应 Z 轴正方向，Unity 里的"前方"是 Z 轴。

### Translate 方法

`transform.Translate()` 用来移动物体，传进去的是一个移动量，不是目标坐标。也就是说它表达的是“往某个方向挪一点”，不是“瞬移到某个位置”。默认是Space.Self，以自身坐标轴为参考系。

```csharp
// 每帧沿自身前方移动
transform.Translate(Vector3.forward * speed * Time.deltaTime);

// 按世界坐标系的 Z 轴正方向移动
transform.Translate(Vector3.forward * speed * Time.deltaTime, Space.World);

// 按自身坐标系移动，默认就是 Space.Self
transform.Translate(Vector3.forward * speed * Time.deltaTime, Space.Self);
```

`Space.Self` 是按物体自己的坐标轴移动，物体转向之后，前后左右也会跟着变。`Space.World` 是按世界坐标轴移动，不管物体自己怎么歪，世界的 X、Y、Z 都在那里。

也可以拿别的 Transform 当参考系：

```csharp
transform.Translate(Vector3.forward * speed * Time.deltaTime, targetTransform);
```

这种写法就是“以 targetTransform 的方向为标准来移动”。

### LookAt 方法

`transform.LookAt()` 会让当前物体看向目标点或者目标物体。

```csharp
transform.LookAt(targetTransform);
transform.LookAt(targetPosition);
```

这里的“看向”更准确地说，是让物体自身的 **Z 轴正方向** 指向目标。Unity 里 `transform.forward` 就是本地 Z 轴正方向，所以模型如果导入时朝向不是 Z 轴，`LookAt` 之后可能会出现“它确实看过去了，但姿势很怪”的情况。可能是模型自己的朝向没对上。

### Transform 的父子关系操作

父子级关系可以直接通过 `parent` 或者 `SetParent` 来改：

```csharp
// 设置父物体
transform.parent = parentTransform;

// 更推荐这个，可以明确是否保持世界坐标
transform.SetParent(parentTransform);
transform.SetParent(parentTransform, true);
transform.SetParent(parentTransform, false);

// 解除父子关系
transform.SetParent(null);
```

`SetParent(parent, true)` 会尽量保持物体在世界里的位置、旋转、缩放不变；`SetParent(parent, false)` 则保留局部 Transform。

Unity 没有“给父物体设置子物体”这种独立 API，本质上还是让子物体去设置自己的父物体：

```csharp
childTransform.SetParent(parentTransform);
```

### 查找子物体

`GetChild` 是按索引找直接子物体：

```csharp
Transform child = transform.GetChild(0);
int count = transform.childCount;
```

这里要记住两个点：`GetChild` 只看第一层子物体，不会往孙子级继续找；`childCount` 也只统计直接子物体。二级、三级那些都不算。想递归找就得自己写逻辑，或者别把层级设计得太离谱。

`Transform.Find` 是按名字找子物体：

```csharp
Transform child = transform.Find("Weapon");
Transform grandChild = transform.Find("Body/Hand/Weapon");
```

它默认也是找当前 Transform 下面的子物体，不是全场景搜索。可以用路径写法往下找，但名字必须对得上。优点是比全局查找范围小，缺点是层级或者名字一改，代码就开始装死。

如果已经拿到了某个 `GameObject`，要找它下面的子物体，通常还是从 `transform` 下手：

```csharp
Transform child = player.transform.Find("Weapon");
GameObject childObject = child.gameObject;
```

### GameObject 的全局查找

`GameObject.Find()` 是按名字在整个 Hierarchy 里找物体：

```csharp
GameObject player = GameObject.Find("Player");
```

这个东西能用，但不太适合频繁用。它会在场景里查找，性能不算友好，而且强依赖名字。名字一改，引用就没了。比较适合临时测试、初始化时偷懒用一下，不适合在 `Update` 里每帧找。

按 Tag 找可以用：

```csharp
GameObject player = GameObject.FindGameObjectWithTag("Player");
GameObject[] enemies = GameObject.FindGameObjectsWithTag("Enemy");
```

`FindGameObjectWithTag` 只返回一个对象。如果场景里有多个同 Tag 的物体，不要依赖它到底返回哪一个。教程里可能会提到创建顺序之类的现象，但实际写代码时最好当成“不保证顺序”。真要多个对象，就用 `FindGameObjectsWithTag`，然后自己筛。

### Input 类

旧输入系统里常用的几个方法：

```csharp
// 键盘
Input.GetKey(KeyCode.W);      // 按住时持续为 true
Input.GetKeyDown(KeyCode.W);  // 按下那一帧为 true
Input.GetKeyUp(KeyCode.W);    // 松开那一帧为 true

// 鼠标：0 左键，1 右键，2 中键
Input.GetMouseButton(0);
Input.GetMouseButtonDown(0);
Input.GetMouseButtonUp(0);

// 触摸
int touchCount = Input.touchCount;
Touch touch = Input.GetTouch(0);
```

`GetKey` 适合持续行为，比如按住 W 往前走；`GetKeyDown` 适合触发一次的行为，比如跳跃、开菜单、发射；`GetKeyUp` 则适合松手时结算，比如蓄力攻击松开释放。别把 `GetKeyDown` 拿去写持续移动，不然角色只会抽一下。

`GetButtonDown` 用的是输入管理器（InputManager）里的虚拟按钮名：

```csharp
if (Input.GetButtonDown("Jump"))
{
    Debug.Log("Jump!");
}
```

它不直接写某个具体键，而是写一个配置好的名字，比如 `"Jump"`、`"Fire1"`。好处是以后改按键配置时不用到处改代码。

轴输入常见的是移动：

```csharp
float horizontal = Input.GetAxis("Horizontal");    // 平滑，按下时会逐渐从 0 变为 -1 或 1，松开时再慢慢回到 0
float rawHorizontal = Input.GetAxisRaw("Horizontal"); // 不平滑，立刻就是 -1、0、1
```

`GetAxis` 手感更柔一点，适合需要渐变的移动或镜头；`GetAxisRaw` 反应干脆，适合 2D 平台跳跃、格斗、菜单选择这种不想拖泥带水的输入。

### transform.forward

`transform.forward` 表示物体自身的 Z 轴正方向，也就是 Unity 认定的“前方”。

```csharp
Vector3 dir = transform.forward;
```

它会跟着物体旋转变化，所以经常拿来做“朝当前面向方向移动”“从枪口往前发射”“检测前方有没有东西”之类的逻辑。这个和 `Vector3.forward` 不一样，`Vector3.forward` 永远是世界坐标的 `(0, 0, 1)`，而 `transform.forward` 是“我自己现在面朝哪边”。一个是世界的前，一个是我的前。

### 递归查找子物体

前面说过，`transform.Find()` 和 `GetChild()` 默认都只处理当前层级下面的子物体。如果要找很深的子物体，就可以自己写一个递归查找。

```csharp
Transform FindChildRecursive(Transform parent, string childName)
{
    foreach (Transform child in parent)
    {
        if (child.name == childName)
        {
            return child;
        }

        Transform result = FindChildRecursive(child, childName);
        if (result != null)
        {
            return result;
        }
    }

    return null;
}
```

用的时候：

```csharp
Transform weapon = FindChildRecursive(transform, "Weapon");
```

这里的 `foreach (Transform child in parent)` 第一眼看着有点怪，毕竟 `parent` 是一个 `Transform`，不是数组，也不是 `List<Transform>`。它能这么写，是因为 Unity 的 `Transform` 支持被 `foreach` 遍历，遍历出来的是它下面的**直接子物体的 Transform**。

所以这段：

```csharp
foreach (Transform child in parent)
{
    // ...
}
```

大概等价于：

```csharp
for (int i = 0; i < parent.childCount; i++)
{
    Transform child = parent.GetChild(i);
}
```

也就是说，`Transform` 本身不是一个子物体数组，只是 Unity 给它提供了“遍历直接子物体”的写法。它同样不会自动遍历孙子级，所以递归函数里才要对每个 `child` 再调用一次 `FindChildRecursive`，一层一层往下翻。

这东西适合初始化时找一次，然后缓存起来。别每帧递归查，层级一复杂就开始到处翻箱倒柜。

### Unity 生命周期函数

脚本挂到物体上之后，Unity 会在不同阶段自动调用一些方法。常见顺序大概是这样：

```csharp
void Awake();       // 脚本实例被加载时执行一次，适合初始化自身数据
void OnEnable();    // 每次脚本或物体启用时执行
void Start();       // 第一次 Update 前执行一次，适合做依赖其他对象的初始化
void FixedUpdate(); // 固定时间间隔执行，常用于物理相关逻辑
void Update();      // 每帧执行，常用于输入、普通游戏逻辑
void LateUpdate();  // Update 之后执行，常用于相机跟随
void OnDisable();   // 每次脚本或物体禁用时执行
void OnDestroy();   // 对象销毁时执行
```

`Awake` 比 `Start` 更早。一般我会把“这个脚本自己就能准备好的东西”放在 `Awake`，把“需要等别的对象也准备好”的东西放在 `Start`。当然项目小的时候怎么写都能跑，项目大了之后尽量避免在初始化生命周期中依赖其他同样需要初始化的。

`FixedUpdate` 不跟帧率走，而是按固定步长走，所以物理移动、刚体受力之类的东西更适合放这里。`Update` 则是每帧一次，输入检测通常放这里。`LateUpdate` 很适合相机跟随：角色在 `Update` 里动完，相机再跟上。

物体失活时，可以理解为它身上的组件也一起被停掉了。`SetActive(false)` 会让这个 GameObject 以及它的子物体从活动状态里退场，脚本也不会继续跑 `Update`。

### Light 组件

`Light` 组件就是 Unity 里的灯。几个常见属性先记个大概就行，真做画面的时候再回来细调。

- **Type**：灯光类型。常见有 Directional Light（平行光/方向光）、Point Light（点光源）、Spot Light（聚光灯）、Area Light（区域光）。
- **Mode**：光照模式。Realtime 是实时计算，Baked 是烘焙进贴图，Mixed 则是两边都沾一点。
- **Intensity**：灯光强度，越大越亮。
- **Indirect Multiplier**：间接光强度倍率，主要影响反弹光/漫反射那部分的贡献。
- **Shadow Type**：阴影类型。Hard Shadows 边缘硬，Soft Shadows 边缘柔一点。
- **Bias**：阴影偏移，用来缓解阴影痤疮那种一片脏脏的自阴影问题。调太大又会让影子和物体分开，看起来像影子迟到了半步。

`Cookie` 可以理解成给灯光加一张投影遮罩。灯光照出去时会带上这张图案，比如窗格影子、树叶缝隙、奇怪仪式现场，都是它能干的活。

`Draw Halo` 是绘制光晕效果。`Flare` 是镜头光斑，更像镜头被强光照到时出现的那种炫光，不是 Cookie 那种把图案投到物体表面上。

`Render Mode` 有 Auto、Important、Not Important。它大致是在告诉 Unity 这盏灯重不重要，影响渲染时怎么处理它。平时 Auto 就够用，除非真的在做光照优化或者画面调校。

### Culling Mask 和 Layer

`Culling Mask` 决定这盏灯会影响哪些 Layer。选了某个 Layer，灯就照那个 Layer 上的物体；取消掉，灯就不管它。`Nothing` 是谁都不照，`Everything` 是全都照。

如果场景里有很多灯，合理去掉无关 Layer 是有机会减轻渲染压力的。比如角色用的特写灯不需要照到整张地图，那就只让它影响角色 Layer。这样逻辑也清楚，性能也不至于被无意义的灯光交互慢慢吃掉。

底层的 LayerMask 本质上是位运算。Unity 默认最多 32 个 Layer，所以可以理解成有 32 个开关位：

```csharp
int layer0 = 1 << 0;       // 第 0 层
int layer1 = 1 << 1;       // 第 1 层
int layer0And1 = layer0 | layer1; // 同时包含第 0 层和第 1 层，也就是二进制 11，十进制 3
```

所以 3 不是“第三层”，而是“第 0 层和第 1 层都打开”。

代码里如果要设置灯光的 `Culling Mask`，也是直接改这个位掩码：

```csharp
Light light = GetComponent<Light>();

light.cullingMask = 0;   // Nothing，不照任何 Layer
light.cullingMask = -1;  // Everything，照所有 Layer
```

这里不要写成 `32`。`32` 的意思不是“32 个 Layer 全选”，而是二进制里第 5 位为 1，也就是只选中第 5 层：

```csharp
light.cullingMask = 1 << 5; // 只照第 5 层，值就是 32
```

多个 Layer 可以用按位或拼起来：

```csharp
light.cullingMask = (1 << 0) | (1 << 5); // 同时照第 0 层和第 5 层
```

如果不想自己数 Layer 下标，更推荐直接按名字取：

```csharp
light.cullingMask = LayerMask.GetMask("Default", "Player");
```

### 静态阴影烘焙

不会动的静态物体可以做光照烘焙，把光照和阴影预先算成贴图。这样运行时就不用每帧重新计算这些阴影，性能会好很多。

代价是它适合“不动”的东西。静态物体如果运行时突然移动，烘焙出来的阴影不会跟着走，原来的地方可能还残留着影子。

### 获取和控制组件

Unity 的功能大多都挂在组件上，所以脚本里经常要获取组件、添加组件、启用组件、禁用组件、销毁组件。

```csharp
// 获取组件
Rigidbody rb = GetComponent<Rigidbody>();

// 添加组件
BoxCollider box = gameObject.AddComponent<BoxCollider>();

// 启用或禁用组件
rb.enabled = false;
rb.enabled = true;

// 激活或失活整个物体
gameObject.SetActive(false);
gameObject.SetActive(true);

// 销毁组件或物体
Destroy(box);
Destroy(gameObject);
```

这里要分清楚：`SetActive` 是控制整个 `GameObject` 的激活状态；组件自己通常是用 `enabled` 来开关。比如关掉一个 `Collider`，可以让它不参与碰撞；关掉整个物体，那它身上一串组件基本都一起退场。

### 子弹跟随枪口方向发射

想让子弹发射方向随着枪口调整，核心就是用枪口的 `position` 和 `forward`。枪口在哪里，子弹就从哪里生成；枪口朝哪边，子弹就往哪边飞。

```csharp
public Transform muzzle;
public GameObject bulletPrefab;
public float bulletSpeed = 20f;

void Fire()
{
    GameObject bullet = Instantiate(bulletPrefab, muzzle.position, muzzle.rotation);
    Rigidbody rb = bullet.GetComponent<Rigidbody>();
    rb.velocity = muzzle.forward * bulletSpeed;
}
```

`Instantiate` 里直接传 `muzzle.rotation`，子弹生成时朝向就和枪口一致。真正决定飞行方向的是 `muzzle.forward`。所以如果子弹横着飞、倒着飞、斜着飞，先看枪口物体的 Z 轴是不是指向真正的枪口前方。很多时候不是代码错，是枪口节点自己站歪了。

### OnGUI 和 GUI 类

`OnGUI()` 是 Unity 旧的即时模式 GUI（IMGUI）入口。它适合拿来快速测试、临时显示调试信息、写一点工具界面。正式游戏 UI 现在一般还是用 uGUI 或 UI Toolkit，`OnGUI` 更像一把随手拿来用的小刀，锋利，但不适合拿去盖房子。

```csharp
void OnGUI()
{
    GUI.Label(new Rect(10, 10, 200, 30), "Hello GUI");

    if (GUI.Button(new Rect(10, 50, 100, 30), "Click"))
    {
        Debug.Log("Clicked");
    }
}
```

`GUI.Label` 是显示文本，`GUI.Button` 是按钮。按钮会在被点击的那一次返回 `true`，所以通常直接放在 `if` 里判断。

`GUI.Toggle`、`GUI.HorizontalSlider`、`GUI.VerticalSlider` 这种控件要注意：它们会把新的状态作为返回值丢回来，必须接住并保存。否则界面看起来被点了，下一次 `OnGUI` 重绘时又回到原来的值，仿佛刚才的操作只是我脑内发生的。

```csharp
bool isOn = false;
float volume = 0.5f;

void OnGUI()
{
    isOn = GUI.Toggle(new Rect(10, 10, 120, 30), isOn, "Enable");
    volume = GUI.HorizontalSlider(new Rect(10, 50, 120, 30), volume, 0f, 1f);
}
```

竖向滑动条也是一样，参数里传进去的值应该是变量，而不是常数：

```csharp
volume = GUI.VerticalSlider(new Rect(10, 90, 30, 120), volume, 1f, 0f);
```

如果写成固定值：

```csharp
GUI.VerticalSlider(new Rect(10, 90, 30, 120), 0.5f, 1f, 0f);
```

那它每次重绘都会被固定回 `0.5f`，自然就拖不动。不是鼠标不努力，是代码不给它活路。

`GUI.Toolbar` 可以做一组选项：

```csharp
int selected = 0;
string[] names = { "Move", "Rotate", "Scale" };

void OnGUI()
{
    selected = GUI.Toolbar(new Rect(10, 10, 240, 30), selected, names);
}
```

`GUI.VerticalScrollbar` 是滚动条，通常用来控制某个滚动位置。它和滑动条很像，也要保存返回值：

```csharp
float scrollValue = 0f;

void OnGUI()
{
    scrollValue = GUI.VerticalScrollbar(
        new Rect(10, 10, 20, 100),
        scrollValue,
        20f,
        0f,
        100f
    );
}
```

### OnGUI 坐标系

Unity 里有好几套坐标系，容易把人绕成麻花。屏幕坐标和 Sprite Editor 里常见的坐标原点在左下角，X 往右，Y 往上。

但 `OnGUI` 的坐标原点在**左上角**，X 往右，Y 往下：

```csharp
GUI.Label(new Rect(0, 0, 200, 30), "左上角");
```

所以 `Rect(10, 10, 100, 30)` 表示从屏幕左上角往右 10、往下 10 的位置。这个方向和很多 Unity 世界/屏幕坐标直觉不一样，写的时候要在脑子里切一下频道。

### Time.time 和 Time.deltaTime

`Time.time` 是从游戏开始运行到现在经过的总时间，单位是秒。

```csharp
Debug.Log(Time.time);
```

`Time.deltaTime` 是上一帧到这一帧之间经过的时间，也可以理解成上一帧耗了多久。写移动、计时、冷却时很常用：

```csharp
timer += Time.deltaTime;
transform.Translate(Vector3.forward * speed * Time.deltaTime);
```

这样写的好处是不同帧率下速度比较稳定。60 FPS 时每帧加得少，30 FPS 时每帧加得多，最后一秒钟加起来差不多还是一秒。

### 不要在 OnGUI 里累计 deltaTime

如果在 `OnGUI` 里写：

```csharp
void OnGUI()
{
    timer += Time.deltaTime;
}
```

结果可能和 `Update` 里不一样。因为 `OnGUI` 不是严格每帧只执行一次，它可能因为不同的 GUI 事件在一帧里被调用多次，比如 Layout、Repaint、鼠标输入等等。于是 `Time.deltaTime` 可能在同一帧里被加了不止一次，计时就飘了。

对比一下更明显：

```csharp
float updateTimer = 0f;
float guiTimer = 0f;

void Update()
{
    updateTimer += Time.deltaTime;
}

void OnGUI()
{
    guiTimer += Time.deltaTime;

    GUI.Label(new Rect(10, 10, 250, 30), "Update: " + updateTimer);
    GUI.Label(new Rect(10, 40, 250, 30), "OnGUI: " + guiTimer);
}
```

真正的计时逻辑放 `Update` 里更稳，`OnGUI` 只负责显示结果就好：

```csharp
float timer = 0f;

void Update()
{
    timer += Time.deltaTime;
}

void OnGUI()
{
    GUI.Label(new Rect(10, 10, 250, 30), "Timer: " + timer);
}
```

简单记就是：`Update` 管游戏逻辑，`OnGUI` 管临时显示。不要把锅铲和温度计混在一起用，厨房会很热闹。
